## Goal

The system cannot be tricked or broken by failure or repetition. Fix structural
drift while the codebase is small enough to restructure cheaply, then layer
robustness (auth, shutdown, crash recovery, idempotency).

## Guiding Principle

Design-heavy / structural work first. Now is the cheapest it will ever be to
correct drift, fix type erasure, and reshape interfaces. Mechanical fixes and
feature work build on the corrected foundation.

## Key Distinction: Intent vs Workflow (Ch6/Ch7)

Intent is a strictly offline / test-time construct. It defines the blueprint:
which nodes exist, what types flow between them, what the submission schema is.
Intent schemas are validated at build time (TypeBox + tsc) and at submission
time (Value.Parse). Intent has no runtime presence — it is consumed once to
produce a workflow.

Workflow is the runtime counterpart. The engine executes nodes, manages state
transitions, handles retries and compensation. Workflow state lives in the DB.
Node executors are runtime code that reads/writes DB rows.

Step 1 operates entirely in intent-space: defining schemas that constrain what
the runtime can receive. The engine (workflow-space) gains validation hooks
that call into intent schemas, but the schemas themselves are offline artifacts.

## Scope

8 steps across 3 phases. Steps 1-3 are structural (design/type/interface
changes that reshape code). Steps 4-6 are immunity (auth, idempotency,
shutdown). Steps 7-8 are functional gaps + hardening.

SCOPE RATIONALE (original analysis)
================================================================================

### Why design-first sequencing?

The codebase is ~12,800 source lines across ~45 files. Every week it grows.
Structural changes (type flow, interfaces, registry shape) have O(n) cost in
the number of files that depend on them. Doing these first means:

  - Intent typing (Step 1): Every subsequent step that adds/modifies node
    executors gets type safety for free. Retrofitting schemas after warm pool
    and crash recovery land means touching those files twice.

  - Provider interfaces (Step 3): Defining StorageProvider/TunnelProvider now
    means Slice 3 provider implementations slot in cleanly. Doing it after
    means potential interface mismatches.

  - Foundation cleanup (Step 2): Removing dead code and fixing imports before
    adding auth middleware and shutdown logic means less confusion about which
    function is canonical.

### What was cut and why?

  - SSE backpressure (EX3): Requires artificial load testing infrastructure to
    validate. Unbounded buffering hasn't caused issues in single-user E2E.
    Cost/benefit doesn't justify a dedicated step. Revisit at multi-tenant.

  - Field name drift (S2.D4): Batch-fixing cosmetic mismatches is busywork
    that creates noisy diffs. Fix each mismatch when touching the file for
    real work. Some will be fixed by intent typing (Step 1).

  - Chaos testing: The PE8 plan proposed "kill control plane mid-workflow,
    verify recovery" as an exercise step. Instead, Step 6 includes targeted
    crash recovery tests that validate the same invariants with less
    infrastructure overhead.

### Dependencies between steps

  Step 1 (Intent) ──→ Step 8 (Warm pool) [typed node I/O]
  Step 2 (BT6)    ──→ Step 8 (Warm pool) [no legacy functions]
  Step 6 (Crash)  ──→ Step 8 (Warm pool) [interrupted warm claims handled]
  Step 4 (Auth)   ──→ Step 6 (Shutdown)  [auth context for cancel propagation]

  Steps 1-3 are independent of each other (can parallelize).
  Steps 4-5 are independent of each other.
  Step 7 (Logs) is independent of everything except Step 2 (needs CAS).
  Step 8 depends on Steps 1, 2, 6.

### New findings from exploration (not in PE8)

  - db.ts legacy functions (BT6) are fully dead: imported but never called.
    Cleanup is even simpler than expected.
  - getLogUpdates() in storage.ts:255 is dead code (never called). The
    catch-up path isn't just broken — it was never wired.
  - logs.py _send_log_batch sends entries individually despite batching them
    (loop over batch, POST each). Should POST the batch.
  - sse-protocol.ts:503 has a dangling unsubscribe bug: `{} as WebSocket`
    will never match the subscribed fakeWs.
  - recoverWorkflows() uses raw updateWorkflowNode() from db.ts, not
    CAS-based state-transitions. This is itself a BT6-class bug.

    compensate() methods that call provider.terminate(), but the engine never

    Fix: Phase B Step 6 (crash recovery + compensation wiring).

    main.ts:149-151 orphanScanTask() is "Stub: no-op for Slice 1". The background
    job that should catch leaked VMs doesn't exist yet.
    Fix: Phase B Step 6 (shutdown + recovery).

    main.ts:161-172 shutdown() clears intervals and calls process.exit(0). No VM
    enumeration, no terminate calls, no WAL checkpoint.
    Fix: Phase B Step 6.

  GAP 5 (test interruption): Test afterAll hooks track VMs and delete them, BUT
    if bun test is killed before afterAll runs (Ctrl-C, pkill, timeout), VMs leak.
    Both orbstack-e2e.test.ts and orbstack-exercise.test.ts have proper
