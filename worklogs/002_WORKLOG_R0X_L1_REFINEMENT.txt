VALIDATION APPROACH EVOLUTION
================================================================================

Tried multiple validation methods. Effectiveness ranked:

1. BEHAVIOR TRACING (most effective)
   - 65+ traces across 7 domains
   - Critical bugs found: warm pool race, cancel flow, orphan lifecycle
   - Cross-module interactions revealed issues static analysis missed

2. E2E FLOW ANALYSIS
   - Launch-run, terminate, warm pool, orphan cleanup traces
   - Caught missing SSE messages, undefined DB methods

3. SPEC-PARITY AUDITS
   - Useful but less novel - mostly caught obvious inconsistencies

4. DEEP FILE VALIDATION
   - Interface extraction, state machines, dependency direction
   - Good for individual files, missed integration issues

Lesson: Behavior tracing should be weighted more heavily than static analysis.

================================================================================
CRITICAL FIXES (Spec Changes Required)
================================================================================

C5: HELD STATE ELIMINATION
    State machine: 6 states → 5 states
    AVAILABLE → CLAIMED → ACTIVE → COMPLETE (+ FAILED)
    HELD replaced with debug_hold_until: timestamp | null
    Benefit: Simpler, earlier warm pool replenishment

C6: BLOB ARCHITECTURE
    Separate blobs table from objects table
    Schema: blobs (id, checksum, payload, size_bytes, ...)
            objects (id, type, blob_id FK, ...)
    checksum NOT unique (logs use append)
    CAS dedup opt-in per object type, not schema constraint

A1: COMMAND ACKNOWLEDGMENT PROTOCOL
    command_id: autoincrement on StartRun/Cancel/PrepareSnapshot messages
    pending_command_acks: number[] in heartbeat
    30s timeout, 3 retries from control plane
    Reason: SSE is lossy, command loss causes silent failures
