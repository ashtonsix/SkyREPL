PHASE 1 SUMMARY: Research Findings
================================================================================

Five parallel research streams converged on a clear picture of the Slice 1
critical path. Key findings by stream:

**R1 (Rough Notes Digest)**
Validated the kill-safe execution philosophy (two-phase spawn, intent-first),
byte-by-byte SSE reader, atomic claiming, and workflow_state format. Found
three stale claims: (a) manifest sealing incorrectly deferred to Slice 3 —
it is part of launch-run and must be in Slice 1; (b) allocation.ts missing
from the "First 10 Files" priority list despite being central to the state
machine; (c) the behavioral coverage audit offered at the end was already
completed in worklog 011. Key validated insight: Zod is NOT spec-mandated,
it is an implementation choice (spec is library-agnostic for validation).

**R2 (Spec Exploration)**
Chapter-by-chapter MVK extraction for Ch05-Ch11 yielded the end-to-end data
path: CLI hashes files -> POST /v1/blobs/check -> upload missing -> POST
/v1/workflows/launch-run -> workflow engine executes 7-node DAG -> agent
receives start_run via SSE -> downloads files -> executes command -> streams
stdout/stderr -> sends run_complete -> finalize. Minimum DAG: resolve-instance
-> [CB: warm/cold] -> wait-for-boot -> create-allocation -> sync-files ->
await-completion -> finalize-run. The Conditional-Branch pattern is required
even for MVK (warm pool decision).

**R3 (Impl Audit)**
136 total files. 53 at L2 (~22k lines). 83 at L1. Critical finding: 7
workflow node files (510 lines total) are L1-only and ALL sit on the Slice 1
critical path. These are the actual steps the DAG executes. The agent
subsystem is the best-covered: all 5 Slice 1 agent files have L2 pseudocode
(1,981 lines). Resource CRUD for instance.ts (33 lines) and run.ts (35 lines)
are L1-only gaps. ~6,400 lines of existing L2 are NOT on Slice 1 path
(AWS provider, orphan scanner, cleanup-manifest, snapshots, etc.).

**R4 (Backlog Blockers)**
Three AC items MUST be fixed before L3 entry: AC1 (workflow_state strings —
impl uses wrong names like 'run:timeout' vs spec's 'launch-run:timeout'),
AC2 (AVAILABLE->FAILED transition missing from spec but needed for cleanup),
AC3 (DAG dependency persistence — workflow_nodes has no depends_on column,
crash after pattern application loses topology). Three BT items block Slice 1
testing: BT2 (completed_at on FAILED), BT8 (expires_at on seal), BT9
(WorkflowDefinition format mismatch between launch-run and engine). Three BT
items affect quality: BT5 (scoring), BT6 (transition duplication), BT7
(ID type mismatch string vs number).

**R5 (Legacy Patterns)**
OrbStack uses orbctl CLI, not an SDK. Cloud-init has a timing bug on OrbStack
(runcmd fires before FS ready) — solution: skip cloud-init entirely, use
`orbctl run` post-boot to inject bootstrap script. Multi-fallback IP discovery
(orbctl info -> DNS {name}.orb.local -> hostname -I inside VM). Chaos injection
wrapper wraps any provider. Byte-by-byte SSE confirmed essential. Workdir
cleanup must chdir to home first (avoid getcwd on deleted directory). Bootstrap
script is composed from helper functions (header, disable-apt-daily, init,
download-agent, start-agent).

**Convergence across streams:**
All five streams independently confirmed: (a) the 7 workflow nodes are the
critical gap; (b) AC1-AC3 are genuine blockers; (c) the agent is well-covered;
(d) OrbStack provider is simpler than expected (60 L2 lines, CLI-based).

**Surprises:**
- Manifest sealing is part of launch-run, not a separate slice (R1 corrected)
- orbstack.ts.l2.txt is only 60 lines — will need significant expansion at L3
- The legacy codebase skips cloud-init entirely for OrbStack (R5)
- Two parallel API directory structures exist (flat L2 files + nested L1 files)

### 2.2 Conflicts Between Sources (and Resolution)

1. **Manifest sealing: Slice 1 or Slice 3?**
   R1 (rough notes) says Slice 3. R2 (spec) says launch-run includes sealing.
   STATUS.txt MVK definition lists "Manifests: DRAFT -> SEALED."
   RESOLUTION: Slice 1. The launch-run finalize node seals the manifest.
   BT8 (expires_at) must be fixed at the same time.

2. **Warm pool in MVK: yes or no?**
   STATUS.txt MVK says "CLAIMED -> ACTIVE -> COMPLETE" (no AVAILABLE).
   R2 shows the CB pattern (warm/cold) is part of the launch-run DAG.
   R1 notes warm pool claiming is "partially" actionable.
   RESOLUTION: Include the CB pattern structure but always take the cold path
   for initial testing. The warm pool query and claiming logic should exist
   (it is simple) but warm pool replenishment is deferred. This keeps the
   DAG shape honest while removing the hard integration problem.

3. **allocation.ts priority: high or low?**
   R1 (rough notes) omits it from top 10. R3 (audit) shows it at L2 (565
   lines). R4 (backlog) has 5 items touching it (BT2, BT5, BT6, BT7, AC2).
   RESOLUTION: allocation.ts is already at L2 and does not need promotion,
   but it needs cleanup during L3 (consolidate transitions per BT6, fix
   types per BT7, add completed_at per BT2). It is high priority for
   correctness, not for promotion.

4. **File sync: S3 or local?**
   R2 (spec) describes presigned URLs to S3. For OrbStack (local dev), S3
   means MinIO or a local blob store.
   RESOLUTION: Use local filesystem blob store for Slice 1. The control plane
   serves blobs directly via HTTP (no S3/MinIO dependency). The presigned URL
   abstraction still works — the URL just points to localhost. This removes
   an entire infrastructure dependency from the first slice.

5. **Log streaming: WebSocket or SSE?**
   R2 (spec) says "WS /v1/runs/:id/logs" for CLI log streaming.
   RESOLUTION: WebSocket for CLI streaming (Hono supports it via Bun's
   WebSocket). SSE only for agent command delivery (different direction).

6. **L1->L2 promotion vs direct L3?**
   R3 identifies 10 files / 621 lines as critical-path L1 gaps.
   RESOLUTION: The 7 workflow node files (510 lines) should get L2 treatment
   because they encode business logic (node contracts, error handling,
   resource emission). The 3 remaining files (instance.ts, run.ts,
   hold-expiry.ts totaling 111 lines) are thin CRUD/timers with clear
   patterns from existing L2 neighbors — go direct to L3.

### 2.3 Technology Decisions

| Decision             | Choice                  | Rationale                                        |
|----------------------|-------------------------|--------------------------------------------------|
| TypeScript runtime   | Bun                     | Used in legacy, built-in SQLite, fast startup    |
| SQLite driver        | bun:sqlite              | Zero-dep, WAL mode, built-in to Bun              |
| HTTP framework       | ~Hono~ ElysiaJS         | ~Lightweight, Bun-native, WebSocket support~     |
| Validation           | ~Zod~ TypeBox           | ~De facto TS standard, good error messages~      |
| Python agent         | stdlib only             | Spec-mandated (Ch11), no pip install on VMs      |
| Monorepo structure   | Bun workspaces          | control/, cli/, shared/ as packages              |
| Package manager      | Bun                     | Single tool for runtime + deps + scripts         |
| Test framework       | bun:test                | Built-in, fast, compatible with Hono testing     |
| Process management   | Single Bun process      | Workflow engine runs in-process (no worker pool) |
| Blob storage (Slice 1) | Local filesystem      | Avoid MinIO/S3 dep for first slice               |

Not choosing yet (defer to Slice 2+):
- S3 client library (when moving to real blob storage)
- Tailscale integration approach
- CLI framework (use raw arg parsing for `repl run` only)

### Step 5: Workflow Engine (Core Loop + CB Pattern)
Complexity: L | Dependencies: Steps 2, 3 | Gate: DAG executes, crash recovers

  Source: impl-pseudo/control/src/workflow/engine.ts.l2.txt (724 lines),
          patterns.ts.l2.txt (462 lines), state-transitions.ts.l2.txt (690)
  Output: impl-pseudo/control/src/workflow/

  5a. Engine core (engine.ts)
      WorkflowEngine class:
      - submit(request): Create workflow + nodes in DB, start execution.
      - executeLoop(workflowId): Load nodes, find ready (pending + deps
        met), execute concurrently, loop until complete/failed.
      - Node execution: Set node 'running' BEFORE execute(), then
        'completed'/'failed' AFTER. Two-phase for crash safety.
      - Ready detection: node.status === 'pending' AND all nodes in
        node.depends_on have status 'completed' or 'skipped'.

  5b. Crash recovery (engine.ts)
      recoverWorkflows(): On startup, find workflows with status='running'.
      For each: find nodes with status='running'. If idempotent, reset to
      'pending'. If not, mark 'failed'. Resume executeLoop.

  5c. Conditional-Branch pattern (patterns.ts)
      CB: evaluate condition -> mark one branch's nodes as 'pending',
      other as 'skipped' -> set join node depends_on to chosen branch.
      Used for warm-pool decision in launch-run.

  5d. State transitions (state-transitions.ts)
      Single canonical source for allocation state transitions (BT6 fix).
      validateTransition, transitionAllocation (with CAS), completed_at
      on ALL terminal states (BT2 fix).

  5e. Workflow + node types (engine.types.ts)
      WorkflowSubmission, NodeExecutor interface, NodeContext.

  Gate: Can submit a workflow with a CB pattern, execute it, verify nodes
        run in dependency order. Kill process mid-execution, restart,
        verify recovery completes the workflow.

### Step 6: launch-run Intent (7-Node DAG)
Complexity: L | Dependencies: Steps 4, 5 | Gate: end-to-end with mock agent

  Source: impl-pseudo/control/src/intent/launch-run.ts.l2.txt (740 lines),
          Step 0d L2 node files
  Output: impl-pseudo/control/src/intent/launch-run.ts,
          impl/control/src/workflow/nodes/*.ts

  The DAG:

    resolve-instance
        |
       [CB: warm available?]
       / \
    claim  spawn-instance
       \ /
    wait-for-boot
        |
    create-allocation
        |
    sync-files
        |
    await-completion
        |
    finalize-run

  6a. Blueprint registration (launch-run.ts)
      Register launch-run workflow type with engine. Define node graph
      with depends_on edges. Input schema: {command, spec, provider,
      files?, env?}. Fix BT9 (align WorkflowDefinition format with
      engine's WorkflowBlueprint).

  6b. Node implementations (nodes/*.ts)
      resolve-instance: Query warm pool. For Slice 1, always returns
        warmAvailable=false (cold path). Apply CB pattern.
      spawn-instance: Two-phase spawn. Phase 1: insert instance record
        with workflow_state='spawn:pending', emit to manifest. Phase 2:
        call provider.spawn(), update record on success/failure.
      claim-allocation: Atomic CAS claim (not used in cold path for MVK
        but structurally present for CB correctness).
      wait-for-boot: Poll instance status via provider.get() until
        status='running'. For OrbStack, near-instant.
      create-allocation: Insert allocation with status=CLAIMED, run_id
        set, workdir assigned. Emit to manifest (priority 90).
      sync-files: Generate blob download URLs. Send start_run message
        via SSE to agent. Wait for sync_complete log message.
        Transition CLAIMED -> ACTIVE on success.
      await-completion: Block until run_complete message received from
        agent. Extract exit_code.
      finalize-run: Transition ACTIVE -> COMPLETE. Set completed_at.

### Step 8: Agent (Python)
Complexity: M | Dependencies: none (parallel with Steps 2-7)
Gate: agent boots, connects SSE, executes command, streams logs

  Source: impl-pseudo/agent/*.l2.txt (agent 319, executor 630, sse 520,
          heartbeat 204, logs 308 = 1,981 lines)
  Output: impl-pseudo/agent/

  8a. SSE reader (sse.py)
      Byte-by-byte read(1) reader. Connect to /v1/agent/commands.
      Parse SSE events (data: lines). Exponential backoff reconnect
      (2s -> 64s, max 10 retries). Deduplication via processed_command_ids.

  8b. Heartbeat (heartbeat.py)
      Background thread. Send immediately on startup (before first sleep).
      Then every HEARTBEAT_INTERVAL_MS (10s). Payload: instance_id,
      active_allocations, workflow_state. HTTP POST to /v1/agent/heartbeat.

  8c. Log flusher (logs.py)
      Background thread. Buffer log messages locally. Flush every 100ms.
      sync_complete bypasses buffer (flush_and_send immediately).
      HTTP POST to /v1/agent/logs with newline-delimited JSON.
      batch_sequence_id for ordering.

  8d. Executor (executor.py)
      On start_run: clean workdir (chdir home first! R5 sharp edge),
      download files from presigned URLs in parallel (urllib, no deps),
      report sync_complete. Execute command via subprocess, capture
      stdout/stderr, stream to log buffer. On exit, send run_complete
      with exit_code.

  8e. Main (agent.py)
      Acquire lock (/tmp/repl-agent.lock). Read env vars (REPL_CONTROL_
      PLANE_URL, REPL_INSTANCE_ID, REPL_JOIN_TOKEN). Start threads:
      heartbeat, log_flusher, sse_command_handler. Dispatch commands
      to executor. Graceful shutdown on SIGTERM.

  Note: stdlib only. No pip, no venv, no requirements.txt.
  Note: Can develop/test locally against the control plane.

  Gate: Agent boots, connects SSE, receives start_run, downloads a test
        file, executes "echo hello", streams stdout, sends run_complete.

================================================================================
OPERATIONAL GATE: Definition of Done
================================================================================

Slice 1 is "done" when ALL of the following are true:

1. END-TO-END EXECUTION
   `repl run "echo hello"` streams "hello" back to the terminal via an
   OrbStack VM. The complete data path works:
   CLI -> control plane -> workflow engine -> OrbStack provider (spawns VM)
   -> bootstrap installs agent -> agent connects SSE -> receives start_run
   -> executes command -> streams stdout via log channel -> CLI displays
   output via WebSocket -> CLI exits with correct exit code.

2. STATE MACHINE INTEGRITY
   Allocation transitions through CLAIMED -> ACTIVE -> COMPLETE.
   Workflow transitions through pending -> running -> completed.
   Manifest transitions through DRAFT -> SEALED (with expires_at set).
   All terminal states have completed_at timestamps.

3. AGENT HEARTBEAT
   Agent sends heartbeat every 10s. Control plane tracks last_heartbeat.
   Missing heartbeat for >5 minutes triggers allocation FAILED transition.
   (Note: full heartbeat timeout enforcement is Slice 2 scope, but the
   heartbeat channel must be operational.)

4. CRASH RECOVERY
   Kill control plane mid-run. Restart. Workflow engine discovers
   interrupted workflows via recoverWorkflows(). Running nodes with
   idempotent operations reset to pending. Workflow resumes and completes.
   No orphan instances remain after recovery.

5. FILE SYNC (basic)
   CLI can specify local files. Files are uploaded to control plane blob
   store. Agent downloads them before executing command. sync_complete
   triggers CLAIMED -> ACTIVE transition.

NOT required for Slice 1 done:
- Warm pool reuse (always cold-spawn)
- SSH access to instances
- Artifact collection
- Snapshot creation
- Multi-provider support
- Budget enforcement
- Idempotency keys
- Orphan scanner
- Tailscale networking
- Cancellation/pause/resume


================================================================================
KNOWN RISKS
================================================================================

1. ORBSTACK AVAILABILITY ON CI
   Risk: OrbStack requires macOS. CI runners are typically Linux.
   Impact: Integration tests can only run locally or on macOS CI.
   Mitigation: Accept local-only integration tests for Slice 1. The
   provider interface allows a mock provider for unit tests. Full CI
   integration testing deferred to when AWS provider exists.

2. WORKFLOW ENGINE COMPLEXITY
   Risk: The engine (executeLoop + CB pattern + crash recovery) is the most
   complex component. Under-specifying crash recovery could leave subtle
   bugs.
   Impact: Workflows that appear to complete may leave inconsistent state.
   Mitigation: Step 5 has the largest complexity rating (L) and should be
   implemented first among the application-logic steps. Invest in unit
   tests for the engine before building nodes on top of it. The L2
   pseudocode (724 + 462 + 690 = 1,876 lines) provides strong guidance.

3. AGENT-CONTROL PLANE TIMING
   Risk: The SSE connection, heartbeat, and log channels create a
   distributed systems timing problem. The sync-files node blocks until
   sync_complete arrives — but what if the message is lost or delayed?
   Impact: Stuck workflows, false FAILED transitions.
   Mitigation: Timeouts on sync-files and await-completion nodes. Agent
   command retry protocol (pending_command_acks in heartbeat). For Slice 1,
   use generous timeouts (5 minutes) and address edge cases in Slice 2.

4. BOOTSTRAP FAILURE MODES
   Risk: The bootstrap script runs via `orbctl run` post-boot. If it fails
   (Python not installed, network issue, disk full), the agent never
   connects and the workflow hangs until timeout.
   Impact: Poor failure UX — user waits for timeout instead of immediate
   error.
   Mitigation: Legacy pattern (R5): bootstrap failures are tolerated;
   missing heartbeat after timeout is the failure signal. For Slice 1,
   use a 60-second boot timeout in wait-for-boot node. Log bootstrap
   output to /tmp/bootstrap.log on the VM for debugging.

5. SCOPE CREEP FROM BACKLOG ITEMS
   Risk: BT2, BT5, BT6, BT7, BT8, BT9 are all "fix during L3" items that
   touch Slice 1 code. Each is small individually but together they add
   friction to every file touched.
   Impact: Steps take longer than estimated due to accumulated fixes.
   Mitigation: Fix BT2/BT7/BT8/BT9 in Step 3 (shared types) and Step 5
   (state transitions) where the types are first defined — this front-loads
   the fixes. BT5/BT6 are addressed naturally by consolidation in Step 5.

6. L1->L2 PROMOTION QUALITY
   Risk: The 7 workflow node files need L2 pseudocode before L3. If the
   L2 content is vague or inconsistent with the engine interface, L3
   implementation will hit design questions.
   Impact: Blocked L3 progress, rework.
   Mitigation: Step 0d uses the engine's NodeExecutor interface and R2's
   node contract table (inputs/outputs/resource emission per node) as the
   template. Each node L2 file must specify: execute() signature, input
   validation, DB operations, resource emission, error paths, output type.

7. SINGLE-PROCESS ARCHITECTURE
   Risk: Workflow engine, API server, SSE connections, and background tasks
   all run in one Bun process. A blocking operation stalls everything.
   Impact: Heartbeat ACKs delayed, SSE connections dropped.
   Mitigation: Bun's event loop handles async I/O well. The only blocking
   operation is provider.spawn() (orbctl create), which is already async.
   SQLite WAL mode allows concurrent reads. Monitor for issues and
   consider worker threads in Slice 2 if needed.


================================================================================
FUTURE SLICES (tentative, post-Slice 1)
================================================================================

These are rough directions, not commitments. Scope and ordering will be refined
after Slice 1 lands and we see what actually hurts.

**Slice 2: Immunity & Robustness**
Goal: The system cannot be tricked or broken by failure or repetition.
- Idempotency-key middleware (POST mutations are safely retryable)
- Heartbeat timeout enforcement (missing heartbeat -> FAILED transition)
- Warm pool: replenishment + claiming (CB pattern already wired in Slice 1)
- Two-phase spawn hardening: kill control plane between DB write and orbctl
  call, verify reconciliation on restart
- Chaos harness: artificial latency + partial failure injection around provider
- Cancellation / force-stop of in-progress runs

**Slice 3: Cleanup & Cost Protection**
Goal: Zero cloud litter, bounded resource lifetime.
- Orphan scanner: discover provider resources with no matching DB record
- Manifest lifecycle: expiry sweeper, cleanup-manifest workflow
- Scheduled termination of idle instances
- Hold-expiry background job (allocation timeout for stuck claims)
- Budget / cost guardrails (provider-level spend limits)

**Slice 4: Multi-Provider & Networking**
Goal: Run on real cloud infrastructure, not just local VMs.
- AWS provider (EC2 Spot via provider interface)
- Tailscale networking (control plane ↔ instance connectivity)
- S3 blob storage (replace local filesystem store)
- SSH access to running instances

**Slice 5: Artifacts & Reproducibility**
Goal: Capture outputs, enable re-runs.
- Artifact collection (files copied back from instance after run)
- Snapshot creation (checkpoint instance state)
- Run replay / re-run from identical spec
- Multi-run workflows

Each slice has its own operational gate (like Slice 1's "echo hello" test).
Slices are roughly sequential but may overlap where independent.
