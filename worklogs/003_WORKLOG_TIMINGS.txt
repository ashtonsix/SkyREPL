# WORKLOG: Timing Consolidation
Status: COMPLETE
Started: 2026-02-04

## Context

This worklog represents two milestones:
1. **Spec cleanup**: Made normative vs non-normative distinctions explicit
   (constraint relationships are normative; specific values are examples)
2. **First L2 work**: shared/src/config/timing.ts.l2.txt is the first file
   promoted from L1 to L2 fidelity level

## Problem Statement

Timing values are scattered across 15 spec chapters (~100 values) and 45+ v2/ files (~50 values). Issues:
1. Duplication - same value defined in multiple places
2. Normative/non-normative confusion - spec mixes hard constraints with example values
3. No single source of truth - changes require updating multiple locations
4. Constraint relationships implicit - easy to break invariants during tuning

## Findings Summary

### Values Found

| Location | Count | Notes |
|----------|-------|-------|
| Spec chapters | ~100 | Many marked as examples |
| v2/ pseudo-impl | ~50+ | Hard-coded constants |
| v2/docs/timeouts.md.l1.txt | 0 | Structure only, no values |

### Consistency Check

Most values are consistent between spec and v2/, but both sources have problems:
- **Spec**: Values inline with prose, unclear which are normative
- **v2/**: Values scattered across files as magic numbers

### Categories Identified

| Category | Example Values | Constraint Type |
|----------|----------------|-----------------|
| Heartbeat chain | 10s interval, 30s ack, 30s stale | Ordering constraint |
| Hold duration | 5m default, 30m extend, 24h max | Bounded range |
| Background tasks | 30s-1h intervals | Tunable per deployment |
| Provider operations | 5m spawn, 30m snapshot | Provider-dependent |
| Cleanup | 10m orphan age, 24h blob grace | Safety margins |
| Retries | 2s-60s backoff | Tunable |
| Cache TTLs | 30s-1h per resource type | Performance tuning |

## Resolution Plan

### Phase A: Spec Cleanup

Goal: Separate normative constraints from operational guidance

**A1. Add "Timing Constraints" section to Ch02 (MATERIAL)**
- Define constraint vocabulary: "MUST", "SHOULD", "MAY", "typically"
- Document constraint chains (heartbeat, cleanup, hold)
- Reference from other chapters

**A2. Update each chapter's timing references**
Replace inline values with:
- Constraint references: "heartbeat_interval < ack_timeout (see §2.X)"
- Non-normative markers: "typical deployment uses 30s [NON-NORMATIVE]"
- Config references: "see TIMEOUTS table for defaults"

**A3. Create normative timing tables in spec**
One table per category with columns:
| Parameter | Constraint | Default | Range | Rationale |

### Phase B: v2/ Consolidation

Goal: Single source of truth for timing constants

**B1. Create shared/src/config/timing.ts.l2.txt**
```
// Centralized timing constants with documentation
export const TIMING = {
  // Heartbeat chain (constraint: interval < ack < stale)
  HEARTBEAT_INTERVAL_MS: 10_000,
  HEARTBEAT_ACK_TIMEOUT_MS: 30_000,
  STALE_DETECTION_MS: 30_000,

  // Hold chain (constraint: default < max_ssh < max_explicit < absolute)
  DEFAULT_HOLD_MS: 300_000,  // 5 min
  MAX_SSH_EXTENSION_MS: 7_200_000,  // 2h
  MAX_EXPLICIT_EXTENSION_MS: 28_800_000,  // 8h
  ABSOLUTE_MAX_HOLD_MS: 86_400_000,  // 24h

  // ... etc
} as const;

// Constraint validators
export function validateTimingConstraints(): void {
  assert(TIMING.HEARTBEAT_INTERVAL_MS < TIMING.HEARTBEAT_ACK_TIMEOUT_MS);
  // ... etc
}
```

**B2. Update v2/ files to import from timing.ts**
Replace magic numbers with `TIMING.X` references

**B3. Expand v2/docs/timeouts.md.l1.txt → .l2.txt**
Add:
- Actual default values (reference timing.ts)
- Tuning guidance per deployment type
- Provider-specific adjustments
- Example configurations (development, production, high-latency)

### Phase C: Documentation

**C1. Operational tuning guide**
Create v2/docs/tuning-guide.md.l1.txt:
- When to adjust which values
- Trade-offs (faster detection vs more false positives)
- Deployment profiles

**C2. Config file format**
Define how users override defaults:
- Environment variables
- Config file (skyrepl.config.ts)
- CLI flags for one-off changes
