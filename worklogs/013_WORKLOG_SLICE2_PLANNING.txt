PE2: PROVIDER SCOPE EXPANSION (S2.P1) — DESIGN SKETCH
================================================================================

### Current State

Provider interface is purely compute-centric:
  - Provider<TInstance, TSnapshot>: spawn, terminate, list, get, generateBootstrap
  - FeatureProvider<TConfig, TState>: attach(instanceId), detach(instanceId)
  - Both assume compute instances exist

### The Problem

Non-compute infrastructure has different lifecycles:
  - Storage: create bucket/database, configure, use, delete
  - Tunnel: expose service, configure routes, revoke
  - Messaging: create channel, subscribe, publish, delete

None of these have "instances" to spawn/terminate.

### Recommended Architecture: Separate Category Interfaces

Three category-specific interfaces + unified registry helper:

```
PROVIDERS
  COMPUTE (Provider<TInstance, TSnapshot>) .... existing
    OrbStack, AWS, Lambda, RunPod
  STORAGE (StorageProvider) ................... NEW
    DATABASE: SQLite (current), PostgreSQL
    OBJECT:   S3, GCS, Backblaze, MinIO (local FS)
  TUNNEL (TunnelProvider) ..................... NEW
    Tailscale Funnel, Cloudflared, ngrok
  FEATURE (FeatureProvider<TConfig, TState>) .. existing
    Tailscale SSH (instance augmentation)
```

Key interfaces:
  - StorageProvider: create(), delete(), get(), list(), capabilities
  - TunnelProvider: expose(service, options), revoke(), status()
  - Each category has its own registry map
  - Unified getAnyProvider(category, name) for cross-cutting queries
  - Lifecycle hooks generalized (heartbeat task types become pluggable)

### Slice 2 Scope for S2.P1

DESIGN ONLY in Slice 2. No implementation of new provider categories.
  - Define StorageProvider and TunnelProvider interfaces
  - Refactor registry.ts to support multiple categories
  - Update spec Ch08 with one-line pointers to docs/
  - Implementation of S3/Cloudflared providers is Slice 3+

This is architectural ground-laying, not feature delivery.

PE4: LOG STREAMING ARCHITECTURE (S2.C2 + S2.D2 + S2.D3)
================================================================================

### Current Data Flow

Agent (executor.py _stream_output, line-by-line)
  -> LogBuffer (logs.py, 100ms local accumulation)
  -> Send Queue (5s or 64KB threshold, POST /v1/agent/logs)
  -> Control Plane (agent.ts:138, creates log object, appends to blob)
  -> WebSocket broadcast (sseManager.broadcastLog)
  -> CLI (client.ts:streamLogs, writes to stdout/stderr)

### Issues Confirmed

S2.D2: getLogUpdates polls blobs.last_referenced_at (updated on EVERY append).
  Cannot distinguish new vs old logs. Should use objects.accessed_at or add
  objects.updated_at.

S2.D3: No id: field in SSE/WebSocket messages. No reconnection resume.
  Need monotonic sequence per stream + Last-Event-ID support.

S2.C3: Log objects created with no ownership wiring (current_manifest_id=null,
  no object_tags, no manifest_resources). Fix by wiring existing spec mechanisms.

### Design for Slice 2

1. Fix S2.D2: Add updated_at to objects table. Poll that column.
2. Fix S2.D3: Add monotonic sequence_id to each log message. Store in
   objects.metadata_json. CLI sends last seen ID on reconnect.
3. Fix S2.C3: Wire log objects into manifest ownership on creation —
   create object_tags, register in manifest_resources, set current_manifest_id.
4. S2.C2 streaming: Keep current pass-through for Slice 2. S3 chunking
   and progressive enhancement deferred to Slice 3 (needs S3 provider).

No schema migration needed — Slice 1 uses in-memory DB. Schema changes
go into 001_initial.sql directly until first production deployment.

PE5: AUTHENTICATION DESIGN (EX5)
================================================================================

### Current State

Token generated: spawn-instance.ts:64 (crypto.randomBytes(16).toString("hex"))
Token delivered: via SKYREPL_REGISTRATION_TOKEN env var + /etc/skyrepl/config.json
Token verified: NEVER. All agent endpoints accept any caller.

### MVP Design for Slice 2

1. Store token hash in instances table (new column: registration_token_hash)
2. Bearer token middleware on agent routes:
   - Extract from Authorization: Bearer <token>
   - For SSE: fallback to ?token= query param
   - SHA256 hash, compare against instances.registration_token_hash
   - 401 on mismatch
3. Agent code: add Authorization header to all HTTP requests
4. Agent SSE: add token to query params on EventSource connection

PE6: WARM POOL DESIGN (EX6 + BT5)
================================================================================

### Current State

resolve-instance.ts always returns { warmAvailable: false }, skips claim node.
findWarmAllocation in db.ts missing 100/50/0 scoring algorithm.
No replenishment after run completion. No expiry of stale AVAILABLE allocations.

### Full Lifecycle Design

1. SCORING (BT5 fix):
   findWarmAllocation adds CASE statement:
     100 = init_checksum matches (exact snapshot)
     50  = init_checksum IS NULL (vanilla instance)
     0   = mismatch (excluded)
   Tiebreaker: created_at ASC (FIFO)

2. CLAIM PATH:
   resolve-instance calls findWarmAllocation with scoring.
   If found: return { warmAvailable: true, allocationId }.
   claim-warm-allocation uses claimAllocation CAS (already implemented).
   If race lost: retry 3x with 50ms backoff, then fall through to cold path.

3. RECYCLE (in finalize):
   After completeAllocation(), check replenishment eligibility:
     - Instance healthy (heartbeat fresh, not terminating)
     - Below MAX_ALLOCATIONS_PER_INSTANCE
   If eligible: createAvailableAllocation(instanceId) — new AVAILABLE allocation.

4. EXPIRY (background task):
   Periodic scan for stale CLAIMED allocations (CLAIMED_TIMEOUT_MS).
   Periodic scan for old AVAILABLE allocations beyond age limit.
   Transition to FAILED (triggers instance cleanup cascade).

### Implementation Order

  Step 1: Fix BT5 scoring in findWarmAllocation
  Step 2: Enable warm path in resolve-instance
  Step 3: Add replenishment logic to finalize
  Step 4: Create background reconciliation task
  Step 5: Integration test (spawn -> complete -> recycle -> claim)

Schema is already complete for warm pool. No DDL changes needed.

PE10: GRACEFUL SHUTDOWN + CANCELLATION (EX4)
================================================================================

### Current State

main.ts signal handlers (SIGINT/SIGTERM) clear interval handles then
call process.exit(0). No draining, no VM cleanup, no WAL checkpoint.

### Shutdown Sequence Design

1. ACCEPT BARRIER: Set flag, reject new workflow submissions (503)
2. DRAIN IN-FLIGHT: Query workflows WHERE status='running'. Wait up to
   30s (configurable). If timeout, force-fail remaining workflows.
3. TERMINATE VMs: Query allocations WHERE status IN (AVAILABLE, CLAIMED,
   ACTIVE). Call provider.terminate() for each. Mark FAILED.
4. WAL CHECKPOINT: PRAGMA wal_checkpoint(TRUNCATE)
5. EXIT: process.exit(0)

### Cancel API

Missing: POST /v1/workflows/:id/cancel endpoint.
Design: Try running->cancelled transition. If already terminal, return
success (idempotent). Fire-and-forget cancel message to agent via SSE.
Agent receives cancel -> SIGTERM -> wait 10s -> SIGKILL.

### Crash Recovery

On startup, recoverWorkflows() already identifies running workflows.
Need: for each running node, check idempotent flag. If idempotent, reset
to pending. If not idempotent, mark as failed with CRASH_RECOVERY error.
