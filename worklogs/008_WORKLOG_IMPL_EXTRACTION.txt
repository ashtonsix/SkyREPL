CONTEXT
================================================================================

WS2 redistributes content from the monolithic spec (~17,400 lines mixing normative
requirements, pseudo-code, UX, and commercial content) into appropriate homes.

The guiding principle: spec and impl evolve as FORKS, not as source and destination.
  - impl/ pseudo-code is free to iterate toward working code (increasing fidelity)
  - spec/ is free to crystallise around core requirements (decreasing fidelity)
Content is COPIED to impl/, then TRANSFORMED in-place within spec — never deleted.

--- Pass 1 Post-mortem ---

The impl/ L2 files are in great shape. However, pass 1 also made aggressive spec
changes that were meant to be deferred: the "Spec stubs" directive
(`> Implementation: impl-pseudo/path/file.l2.txt §Section`) was applied as a REPLACEMENT
rather than an ADDITION, eviscerating spec content. Three failure modes:

  1. Over-stubbing: Substantial definitions replaced with 2-line stubs. E.g. the
     workflow_nodes table was reduced to a one-line description and a pointer to
     impl — that's not a spec, that's a declaration that workflow nodes are
     implementation-defined.

  2. Destructive translation: SQL schemas were converted to TypeScript interfaces
     in db.types.ts.l2.txt, then the SQL originals deleted from spec. Creating a
     DB driver interface made sense, but the underlying schema was lost. Migrations
     directory in impl/control/ never created.

  3. Over-eager deletion: Anything "pseudo-codey" was removed — including example
     queries, annotated SQL, and illustrative code that served specification
     purposes. These needed creative home-finding (tests, reference docs) or should
     have been left in-place for a later refinement pass.

Remediation: `git checkout -- spec/` restored all spec chapters to pre-pass-1 state.
All impl/ L2 files preserved. Both the original spec pseudo-code and the new impl/
L2 files now coexist — the intended fork.

State at pass 1 close:

  L2 files: 52 files, ~21,790 lines

--- Phase B Conventions (applied, complete) ---

1. CROSS-REFERENCE STUBS (additive, never replacing)

   Format: `> See also: impl-pseudo/path/file.l2.txt §Section Name`
   Placed after the spec content they reference, not before.
   One stub per logical block (not per line).

2. CONTENT CATEGORY HANDLING

   SQL DDL (Ch02 ~55%):
     KEEP AS-IS. The DDL IS the data model specification.
     Add cross-ref to migration files: `> Migration: impl-pseudo/control/migrations/001.sql.l2.txt`
     Add normative annotations where missing: "The instances table MUST include..."

   SQL example queries (Ch02, scattered):
     KEEP AS-IS. They illustrate the data model and serve as acceptance criteria.
     Mark as: `-- Example: <purpose>` if not already labeled.
     These are also future test fixtures (impl/tests/).

   JSON request/response examples (Ch09 ~65%):
     KEEP AS-IS. The examples define the API contract.
     Add cross-ref to handler L2 files.
     Add normative annotations: "The response MUST include..."

   CLI command definitions (Ch10 ~44%):
     KEEP AS-IS. These are UX specifications.
     Add cross-ref to L2 command handler files where they exist.

   ASCII diagrams + blueprints (Ch05, Ch07, Ch11):
     KEEP AS-IS. Already specification-grade.
     These are the authoritative source; L2 copies should cross-ref back.

   TypeScript type/interface definitions:
     KEEP AS-IS for now. Types are already semi-formal.
     Add cross-ref to corresponding L2 file.
     Annotate key fields with MUST/SHOULD where behavior is implied but not stated.

   TypeScript/Python function implementations:
     Phase B: add cross-ref stubs only.
     Phase C: these are the primary transformation target — see Phase C treatment
     rules for REPLACE, CONDENSE, and REWRITE categories.

   Configuration tables (Ch15 ~43%):
     KEEP AS-IS. Already specification-grade (timing constraints, deployment profiles).
     Add cross-ref to timing.ts.l2.txt where applicable.

3. NORMATIVE LANGUAGE

   Use RFC 2119 keywords: MUST, MUST NOT, SHOULD, SHOULD NOT, MAY.
   Add as annotations wrapping existing pseudo-code blocks, not replacing them.
   Focus on behavioral contracts, invariants, and safety properties.
