# hold-expiry.ts - Debug Hold Expiry Task

# Note: Timing constants import from ../../../../shared/src/config/timing.ts.l2.txt
# - HOLD_EXPIRY_CHECK_INTERVAL_MS (60s)
# - ABSOLUTE_MAX_HOLD_MS (24h)

## Purpose
Clear expired debug_hold_until timestamps on COMPLETE allocations.
Does NOT transition state - only clears the timestamp field.

## Scan
- Query: allocations WHERE status='COMPLETE' AND debug_hold_until IS NOT NULL AND debug_hold_until < NOW

## Action

For each allocation with expired debug hold:

1. B20: Atomic update with idempotency check:
   - UPDATE allocations SET debug_hold_until=NULL WHERE id=? AND debug_hold_until IS NOT NULL AND debug_hold_until < ? RETURNING *
   - If 0 rows: already processed by another thread, skip to next
   - This prevents duplicate processing if multiple workers race
2. Log: "Cleared expired debug hold for allocation ${allocation_id}"
3. No state transition occurs - allocation remains COMPLETE
4. No warm pool replenishment here (happens at ACTIVE→COMPLETE transition)

## Max Hold Enforcement
- Absolute limit: ABSOLUTE_MAX_HOLD_MS (24h) from completed_at timestamp
- Hard check: debug_hold_until > completed_at + ABSOLUTE_MAX_HOLD_MS → force clear
- Query: WHERE status='COMPLETE' AND debug_hold_until IS NOT NULL AND (debug_hold_until < NOW OR debug_hold_until > completed_at + ABSOLUTE_MAX_HOLD_MS)

## Cycle
- Every HOLD_EXPIRY_CHECK_INTERVAL_MS (60s)
- Aligns with other background task intervals

## Batch Processing
- Process in batches of 50 to prevent lock contention
- Log metrics: expired_count per cycle
- Alert if error rate > 5%

## Notes
- SSH activity DOES auto-extend debug_hold_until (keeping instances alive while users are connected)
- Clearing debug_hold_until makes instance eligible for termination
- Instance termination checks for ANY allocation with debug_hold_until > NOW
