# cleanup-manifest.ts - Cleanup Manifest Intent

Clean up all resources owned by a manifest in priority order.
See L2 for per-provider fan-out, atomic allocation transitions with IN clause, B21 cleanup_processed_at protection, feature cleanup orphaning.

## Exports

cleanupManifest(input), cleanupManifestWorkflow (WorkflowDefinition)

## Input/Output

CleanupManifestInput: manifestId, force?, idempotencyKey?
CleanupManifestOutput: workflowId, manifestId, resourcesCleaned, errors, completedAt

## Workflow Pattern

Parallel-Fan-Out by Priority Group (highest to lowest)

## Priority Order

90: Allocations (transition to COMPLETE, atomic with IN clause)
80: Runs (mark cleaned)
70: Artifacts (delete with B21 protection)
65: Features (detach, best-effort)
60: Objects (delete with B21 protection, explicit S3 cleanup)
50: Instances (terminate via provider)
10: Snapshots (delete from provider and DB)

## Nodes

validate-manifest, load-resources, cleanup-allocations, cleanup-runs, cleanup-artifacts, cleanup-features, cleanup-objects, cleanup-instances, cleanup-snapshots, delete-manifest, finalize

## Error Handling Policies

skip_and_continue: allocations, runs, artifacts, features, objects, snapshots
retry_then_skip: instances (expensive)
fail_workflow: manifest deletion

## B21 Cleanup Processing

Uses cleanup_processed_at flag to prevent double-processing during crash recovery
Check flag before processing, set flag after completion in transaction

## Feature Cleanup Orphaning

Feature detach failures logged but workflow continues (skip_and_continue)
Orphans handled by background reconciliation (FeatureProvider.reconcile())
Common failures: network timeout, instance terminated, partial cleanup

## PerProviderOrchestration

Resources grouped by provider, cleanup proceeds independently per provider
Cross-provider failures don't block unrelated providers
