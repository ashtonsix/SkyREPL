# launch-run.ts - Launch Run Intent (Ch7)

Acquire instance (warm pool or spawn), sync files, execute command, collect artifacts.
See L2 for workflow node handlers, warm pool fallback logic, and pattern applications.

## Inputs
runId, command, env, files, spec, preferWarmPool, allowSpotFallback, artifactPatterns, holdDuration, createSnapshot, idempotencyKey

## Outputs
workflowId, runId, instanceId, allocationId, fromWarmPool, artifacts, snapshotId

## Workflow Nodes
check-budget, resolve-instance, claim-warm-allocation, spawn-instance, wait-for-boot, create-allocation, sync-files, await-completion, finalize-run

## PatternApplications
- CB at resolve-instance: warm pool vs spawn decision
- CB at claim-warm-allocation: try-fallback on race loss
- IAR at wait-for-boot: feature installation (Tailscale, GPU drivers)
- CB at finalize-run: conditional snapshot creation

## Key Behaviors
- Warm pool prefers matching init_checksum (snapshot-based restart)
- File sync bypasses batching via flush_and_send (critical for timeout)
- Sync failure transitions allocation CLAIMED→FAILED
- Successful sync transitions CLAIMED→ACTIVE
- Snapshot-only mode: skip command execution if create_snapshot=true and no command
- Finalize: ACTIVE→COMPLETE, set debug_hold_until, warm pool replenishment, seal manifest

## ResourceEmissions
- instance (priority 50, at spawn-instance)
- allocation (priority 90, at create-allocation)
- run (priority 80, at create-allocation)
- object (priority 10, at finalize-run if snapshot created)

## Materialization
Workflow record in DB. Triggers instance spawn, allocation creation, and manifest sealing.
