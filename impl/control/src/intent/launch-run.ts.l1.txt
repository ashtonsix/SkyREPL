## Purpose
Acquire instance (warm pool or spawn), sync files, execute command, collect artifacts

## Workflow Graph

## Nodes

1. **resolve-instance**: Check warm pool via db.findWarmAllocation(spec)
   - If found: route to claim-allocation branch
   - If not found: route to spawn-instance branch
   - Uses conditional-branch pattern

2. **spawn-instance**: Two-phase spawn (DB then cloud)
   - Phase 1: Create instance record in DB
   - Phase 2: Call provider.spawn()
   - Compensation: Terminate instance if provider_id exists

3. **claim-allocation**: Atomic claim of AVAILABLE allocation
   - Uses optimistic locking (db.claimAllocation)
   - Returns null if race lost → falls back to spawn-instance

4. **create-allocation**: Create allocation record (after spawn only)
   - Creates allocation with status='CLAIMED'

5. **sync-files**: Download files to instance
   - Sends file URLs to agent via SSE
   - No wait (start-run handles sync confirmation)

6. **start-run**: Send command via SSE, wait for sync_complete
   - Waits for agent sync_complete event
   - Control plane transitions allocation: CLAIMED → ACTIVE
   - Sets run.started_at timestamp

7. **wait-completion**: Wait for run_complete or timeout
   - Polls for run completion (24hr default)
   - Captures exit_code

8. **collect-artifacts**: Upload matching files
   - Sends artifact collection request to agent
   - Agent uploads to presigned URLs

9. **finalize**: ACTIVE → COMPLETE, seal manifest, create snapshot if requested
   - Transition allocation: ACTIVE → COMPLETE
   - Set debug_hold_until if configured (for debugging)
   - Warm pool replenishment: create new AVAILABLE allocation if eligible
   - Calls db.sealManifest()
   - Optional provider.createSnapshot()

## Warm Pool Fallback Flow

When preferWarmPool=true (default):
1. resolve-instance checks db.findWarmAllocation(spec, initChecksum)
2. If AVAILABLE allocation found matching spec:
   - Route to claim-allocation
   - Atomic claim via optimistic locking
   - If claim fails (race): fall back to spawn-instance
3. If no matching allocation:
   - Route directly to spawn-instance

When preferWarmPool=false:
- Skip warm pool check, always spawn new instance

## Error Handling

See engine.ts for retry strategy. Key behaviors:
- CAPACITY_UNAVAILABLE: retry-with-alternative (different region)
- PROVIDER_API_ERROR: retry with exponential backoff
- Spawn failure: compensate (terminate partial instance)
