# allocation.ts - Allocation State Machine (Ch5)

Run-to-instance binding. AVAILABLE allocations ARE the warm pool.
See L2 for state transition logic, warm pool replenishment, debug hold semantics, single-tenant enforcement.

## Status Enum
AVAILABLE | CLAIMED | ACTIVE | COMPLETE | FAILED

## State Transitions
- AVAILABLE → CLAIMED (claim-allocation node)
- CLAIMED → ACTIVE (sync_complete log) | FAILED (sync timeout 10min)
- ACTIVE → COMPLETE (finalize node) | FAILED (heartbeat loss, spot interrupt)

## Key Invariants
- At most ONE ACTIVE allocation per instance (single-tenant)
- COMPLETE can have debug_hold_until (blocks termination, max 24h)
- Warm pool replenishment: ACTIVE→COMPLETE creates NEW AVAILABLE allocation
- Each run gets its own allocation record (never reused)
