# engine.ts - Workflow Execution Engine

Core workflow execution engine with crash recovery, retry logic, and timeout handling.
See L2 for executeLoop, retry strategy, error mapping, subworkflow support, topology signatures.

## Exports

WorkflowEngine interface: submit, get, cancel, pause, resume, retry, waitForCompletion, spawnSubworkflow

## Constants

MAX_SUBWORKFLOW_DEPTH = 3, MAX_PARALLEL_BRANCHES = 10
Timing from timing.ts: DEFAULT_WORKFLOW_TIMEOUT_MS (60m), DEFAULT_NODE_TIMEOUT_MS (5m), PENDING_WORKFLOW_TIMEOUT_MS (30m)

## submit(type, input, idempotencyKey?, timeoutMs?)

Check idempotency, create DRAFT manifest, create workflow (status='pending')
Set timeout_at = now + (timeoutMs ?? DEFAULT_WORKFLOW_TIMEOUT_MS)
Create nodes from blueprint, start executeLoop (async), return immediately

## cancel(workflowId, reason)

Fire-and-forget CancelRunMessage via SSE
Transition allocation: ACTIVE -> COMPLETE (not FAILED, cancellation is intentional)
No explicit verification - heartbeat timeout handles unresponsive agents
Seal manifest

## retry(workflowId, fromNodeId?)

Creates NEW workflow with same inputs (new ID, new idempotency key: ${original}:retry:${timestamp})
NOT a state transition (failed->running is invalid)
Original workflow preserved for audit trail

## Execution Loop

Find ready nodes (dependencies satisfied), execute concurrently
Check workflow timeout before each node
On completion: seal manifest, set status='completed'
On failure: call determineRetryStrategy, handle result

## Timeout Hierarchy

Workflow-level: DEFAULT_WORKFLOW_TIMEOUT_MS, checked before each node
Wait operations: explicit timeout per-operation, exempt from node timeout
Node-level: DEFAULT_NODE_TIMEOUT_MS, applies to execute() overall

## Crash Recovery

On startup: find workflows with status='running' or 'pending'
Pending >30min without progress: fail with PENDING_TIMEOUT
Running: reset interrupted nodes (idempotent=retry, non-idempotent=fail), resume executeLoop

## Retry Decision Logic

determineRetryStrategy(error, node, attempt) returns RetryStrategy
ERROR_RETRY_MAPPING defines retryable errors and strategies
CAPACITY_UNAVAILABLE: alternative (different region, 3 retries)
PROVIDER_API_ERROR: same_params with exponential backoff (3 retries)
RATE_LIMITED: same_params respecting retry_after_ms (5 retries)
SPOT_INTERRUPTED: fallback (spot once, then on-demand, 1 retry)
OPERATION_TIMEOUT: same_params (2 retries)
Non-retryable: VALIDATION_ERROR, INVALID_STATE_TRANSITION, DATABASE_ERROR, INSUFFICIENT_PERMISSIONS

## Subworkflow Support

MAX_SUBWORKFLOW_DEPTH = 3, child increments depth
Child inherits parent's manifestId, tracks parentWorkflowId
Manifest sealing waits for all child workflows to complete
