S2.P1: Provider scope expansion beyond compute
  Status: DESIGN ONLY (no new provider implementations yet)
  Severity: HIGH — architectural scope expansion
  Context: Provider scope currently covers compute providers (OrbStack, AWS
    EC2) and FeatureProviders for features installed on a compute instance
    (e.g. Tailscale SSH). Both assume compute instances exist. Need to expand
    scope to include non-compute infrastructure:
    - Infrastructure provisioned for control plane and CLI: record storage
      (SQLite vs PostgreSQL), blob storage (Backblaze vs S3 vs GCS vs Azure
      Blob Storage vs MinIO), local mode public tunnel (Tailscale Funnel vs
      Cloudflared vs ngrok)
    - Non-compute infrastructure (e.g. message brokering — example only,
      no concrete plans)
    Note: Don't conflate Tailscale core (SSH debugging on instances) with
    Tailscale Funnel (exposing local control plane + S3 to public internet).
    These are distinct provider concerns.
  Impact: Provider contract (contracts.ts), registry (registry.ts), types
    (types.ts), extensions (extensions.ts) all need rethinking. Spec Ch08
    needs updating. Non-compute providers have different lifecycles (no
    spawn/terminate).
  Deliverable: Interface definitions (StorageProvider, TunnelProvider),
    registry refactor for multiple categories, spec Ch08 pointers.
  Blocks: S3/Backblaze/Cloudflared/ngrok implementations
  Ref: PE2 design sketch, user direction 2026-02-09

S2.C1: Intent as offline typing module
  Status: DESIGN + IMPLEMENTATION (Phase 1-2 only)
  Severity: HIGH — type erasure causes silent bugs (instanceId/instance_id)
  Context: Workflow currently uses type erasure — ctx.workflowInput cast to
    unknown, ctx.getNodeOutput uses manual type assertions. Confirmed bug:
    create-allocation reads instance_id but wait-for-boot outputs instanceId.
    Intent becomes an offline module guaranteeing type flow correctness across
    all rewrite variations. TypeBox schemas for intent types, validated at
    build time. Core intents must not blur with provider/workflow-specific
    enhancements (extra nodes, topology, input/output attributes).
    Builder/helper pattern TBD — several alternatives need consideration.
  Deliverable: launch-run.schema.ts, launch-run.types.ts, build-time
    validation wiring.
  Later: Provider extension schemas, factory pattern.
  Ref: PE3 design sketch, post-Step-7 audit C1

S2.C2: Log streaming — catch-up path + volume fallback
  Status: DESIGN + PARTIAL FIX
  Severity: HIGH — catch-up path broken, no volume fallback
  Context: Live streaming WORKS (SSEManager WebSocket pass-through, spec-aligned
    with Ch9). What's broken: reconnection/catch-up — getLogUpdates queries
    object_tags that are never created (see S2.C3), so late-connecting clients
    get no history. What's missing (well-motivated spec drift, not in Ch2/Ch9):
    (a) Progressive enhancement: when log volume is extreme, CLI shows byte
    count only and logs go straight to S3 — streaming is opt-out, not the only
    path. (b) Non-line-based framing: multi-GB single lines exist in ML
    workloads, per-line framing is not robust. (c) Buffer-then-persist in chunks
    rather than per-append (reduces SQLite write amplification). S3 fallback
    needs S3 provider from S2.P1 for full solution.
  Bundles: S2.D2 (wrong poll column), S2.D3 (missing SSE id field),
    S2.C3 (log ownership wiring) — all log-related, fix together.
  Ref: PE4 design sketch, post-Step-7 audit C2

EX5: Authentication — agent endpoints wide open
  Status: DESIGN + IMPLEMENTATION
  Severity: HIGH — blocks any non-local deployment
  Context: Token generated in spawn-instance (crypto.randomBytes) but never
    verified. Design sketch in PE5: store token hash in DB, Bearer middleware
    on agent routes, agent code adds Authorization header.
  ~6 files changed. CLI auth out of scope for now.
  Ref: PE5 design sketch, exercise phase audit

EX6: Warm pool lifecycle (includes BT5 scoring)
  Status: DESIGN + IMPLEMENTATION
  Severity: HIGH — 20s+ VM spawn overhead on every run currently
  Context: resolve-instance always returns warmAvailable=false. Full lifecycle
    needed: scoring (100/50/0), claim path with retry, recycle in finalize,
    background expiry. Design sketch in PE6. No DDL changes needed.
  Subsumes: BT5 (scoring algorithm)
  Blocked by: foundation fixes (BT6 legacy functions must go first)
  Ref: PE6 design sketch, exercise phase audit

EX4: Graceful shutdown + crash recovery
  Status: DESIGN + IMPLEMENTATION
  Severity: MEDIUM-HIGH — SIGTERM kills without draining; crash recovery stub
  Context: main.ts signal handlers just call process.exit(). Need: accept
    barrier, drain in-flight workflows (30s timeout), terminate orphaned VMs,
    WAL checkpoint. Also: cancel API (POST /v1/workflows/:id/cancel), crash
    recovery hardening (idempotent flag per node). Design sketch in PE10.
  PE1 finding: crash recovery is highest-impact gap in current impl.
  Ref: PE10 design sketch, exercise phase audit

S2.C3: Wire log objects into manifest ownership
  Status: OPEN (part of log streaming bundle)
  Severity: MEDIUM — correctness of cleanup + catch-up path
  Context: Log objects created with current_manifest_id=null, no object_tags,
    no manifest_resources entry — invisible to cleanup and to getLogUpdates
    (which queries object_tags). No DDL changes needed. Fix: on log creation,
    (1) create object_tags (run_id:<id>, stream:<stream>), (2) register in
    manifest_resources via emitResource, (3) set current_manifest_id. Spec's
    existing ownership mechanisms (Ch2 object_tags + manifest_resources +
    current_manifest_id) already cover this — the previous description
    proposing resource_id/resource_type columns on objects was accidental
    spec drift.
  Also: getLogUpdates queries object_tags but log creation only writes
    metadata_json — align creation to match query path.
  Ref: PE4/PE9, post-Step-7 audit C3
