================================================================================
11. AGENT
================================================================================

## Purpose

The Agent Protocol defines communication between the control plane and Python
agents running on remote instances. The agent is intentionally simple: stdlib-only
Python that receives commands, executes runs, streams logs, and reports status.
Complexity lives in the control plane, not the agent.


## 11.1 Communication Architecture

Three distinct channels, all agent-initiated:

```
    ┌─────────────────────────────────────────────────────────────────────────┐
    │                          CONTROL PLANE                                  │
    └───────────────────────────────┬─────────────────────────────────────────┘
                                    │
          ┌─────────────────────────┼─────────────────────────┐
          │                         │                         │
          │ SSE Stream              │ HTTP POST               │ Presigned URLs
          │ (commands)              │ (logs/heartbeat)        │ (artifacts)
          │                         │                         │
          ▼                         ▲                         ↕
    ┌─────────────────────────────────────────────────────────────────────────┐
    │                              AGENT                                      │
    ├─────────────────────────────────────────────────────────────────────────┤
    │  Threads:                                                               │
    │    • Command handler  ─ SSE consumer, receives commands                 │
    │    • Heartbeat        ─ 10s interval, liveness signal                   │
    │    • Log flusher      ─ 100ms local buffer, 5s network batch           │
    │    • Spot monitor     ─ 5s interval, AWS only                           │
    │    • Run execution    ─ spawned per run                                 │
    └─────────────────────────────────────────────────────────────────────────┘

    ╭─────────────────────────────────────────────────────────────────────────╮
    │  LEGEND:  ▼ control→agent    ▲ agent→control    ↕ bidirectional        │
    ╰─────────────────────────────────────────────────────────────────────────╯
```

### Why Agent-Initiated

All connections originate from agent: instances may be behind NAT/VPCs; simpler
firewall rules (outbound only); control plane doesn't need instance IPs; load
balancer compatible without sticky sessions.

### Channel Purposes

**SSE Stream (control -> agent)**: Command delivery. Single long-lived connection;
agent reads byte-by-byte to avoid buffer deadlock (see 11.6).

**HTTP POST (agent -> control)**: Status reporting. Two-level log batching: local
buffer (stdout/stderr collection), network batch on interval OR size threshold.
Control plane reorders batches via batch_sequence_id. Heartbeats at configured
interval. Fire-and-forget with retry. Intervals configurable (see Ch15 §15.9).

**Presigned URLs**: Artifact upload/download bypasses control plane via direct
S3 PUT/GET (or control plane endpoint for small objects <=64KB - see 11.11).

## 11.2 Schema Notes

**allocation_id** in run messages: Tracks which allocation executes which run,
enabling resource management and SSH session tracking. See Ch5 for allocation
lifecycle.

**workflow_state** in RunCompleteMessage: Uses workflow state format
(e.g., 'launch-run:complete') instead of boolean, enabling richer state
information. See Ch6 for state format.

**workflow_state** in HeartbeatMessage: Control plane knows instance lifecycle
phase without polling.

**active_ssh_sessions**: Reports total sessions across all allocations; control
plane uses debug_hold_until (Ch5) for per-allocation tracking.

## 11.3 Message Types

### Control -> Agent (SSE Stream)

| Type              | Purpose                                    |
|-------------------|-------------------------------------------|
| start_run         | Begin run with file sync and execution    |
| cancel_run        | Terminate running run                     |
| capture_artifacts | Upload artifacts (mid-run or emergency)   |
| prepare_snapshot  | Prepare instance for snapshot creation    |
| trigger_tailscale | JIT Tailscale installation                |
| heartbeat_ack     | Acknowledge agent heartbeat               |

### Agent -> Control (HTTP POST)

| Type                   | Purpose                                    |
|------------------------|-------------------------------------------|
| log                    | Stdout/stderr chunk or sync_complete      |
| artifact               | Artifact upload confirmation              |
| init_complete          | Init script finished successfully         |
| run_complete           | Run finished with exit code               |
| prepare_snapshot_ack   | Acknowledge snapshot preparation complete |
| spot_interrupt_start   | AWS spot interruption detected, starting checkpoint |
| spot_interrupt_complete| Checkpoint done, artifacts uploaded       |

## 11.4.5 Command Acknowledgment Protocol

SSE is lossy. Without acknowledgment, command loss causes silent failures (runs never
start, snapshots never captured). This protocol ensures reliable delivery.

**Protocol:**
1. Control plane sends command with unique `command_id`
2. Agent receives command, queues ack, processes command
3. Agent includes `pending_command_acks` in next heartbeat
4. Control plane marks acknowledged, removes from send queue
5. Control plane retries unacknowledged commands after COMMAND_ACK_TIMEOUT (max 3 retries)

**Deduplication**: Agent tracks last 100 `processed_command_ids` to ignore retries.

**Type signatures:**
```typescript
interface AgentCommandState {
  pending_command_acks: Set<number>;    // Commands awaiting ack transmission
  processed_command_ids: Set<number>;   // Last 100 command IDs (dedup)
}

interface CommandDeliveryState {
  command_id: number;
  sent_at: number;
  retry_count: number;
  acknowledged: boolean;
}
```

**Configuration:**
- `COMMAND_ACK_TIMEOUT_MS` = 30000 (30s, configurable)
- `MAX_COMMAND_RETRIES` = 3
- `PROCESSED_COMMAND_HISTORY_SIZE` = 100 (fixed)

**Edge Cases:**
- Agent restart: may re-execute recent commands (idempotent handlers)
- Control plane restart: pending commands reloaded from `command_delivery` table
- Network partition: commands queue on control side, delivered when reconnected

### Immediate Initial Heartbeat

Heartbeat thread sends a heartbeat BEFORE its first sleep to close the timing
gap where control plane might terminate for "no heartbeat" before agent reports
ready. Control plane should also allow grace period during Tailscale setup.

```python
def heartbeat_thread():
    if not send_heartbeat():  # Immediate on startup
        log("WARN", "Initial heartbeat failed")

    while not shutting_down:
        time.sleep(HEARTBEAT_INTERVAL_MS / 1000.0)  # Default: 10s
        # ... rest of heartbeat logic
```

### Self-Termination is BACKUP Safety

Agent's `shutdown -h now` is LAST RESORT. Normal path: control plane terminates
via provider API. Backup: agent self-terminates if control plane unreachable
(crash, network partition, bug). Agent doesn't rely on control plane for
termination - shuts down after heartbeat timeout regardless.

### Degraded State and Graceful Panic

When control plane becomes unreachable, agent progresses through degraded and
panic states before self-terminating:

```
Timeline (from last successful heartbeat ack):
t=0        Last ack received
t=2m       Enter DEGRADED state (workflow_state = 'degraded:control_plane_unreachable')
           - Continue heartbeating (control plane may recover)
           - Provides early warning visibility to CLI users
t=15m      Enter PANIC state, initiate graceful shutdown:
           1. Send HeartbeatPanicStartMessage (best-effort, may fail if CP unreachable)
           2. Execute checkpoint script (PANIC_CHECKPOINT_BUDGET_MS = 5m)
           3. Upload artifacts with failover URLs
           4. Send HeartbeatPanicCompleteMessage (best-effort)
           5. Initiate shutdown sequence
t=20m      Expected self-termination complete
t=25m      Control plane force-terminates (if agent failed to self-term)
```

If control plane recovers during degraded window (2m-15m), agent receives ack,
clears degraded state, and resumes normal operation. This prevents unnecessary
panic checkpoints for transient network issues.

The panic checkpoint path reuses spot interruption logic (see §11.5 Spot Monitor)
with different timeout budget: 5m for panic vs 90s for spot. Environment variable
`SKYREPL_CHECKPOINT_REASON` is set to `heartbeat_timeout` (vs `spot_termination`).

## 11.6 SSE Implementation Details

### Byte-by-Byte Reading

Python's HTTP libraries use 8KB internal buffers. `read(n)` and `readline()`
block until buffer fills or connection closes. SSE messages are ~150 bytes and
infrequent, so buffered reads block forever.

Solution: byte-by-byte reading bypasses buffers at cost of syscall overhead -
negligible for low-frequency control signaling (agent spends 99.9% of time
blocked on read(1)).

```python
def read_lines(response):
    """Reads newline-delimited strings without deadlocking."""
    while not shutting_down:
        line = b""
        while True:
            byte = response.read(1)
            if not byte:
                return  # Stream closed
            if byte == b'\n':
                break
            line += byte

        text = line.decode('utf-8', errors='replace').strip()
        if text:
            yield text
```

### Connection Management

```python
def command_stream():
    try:
        conn = get_connection(CONTROL_PLANE)
        conn.request("GET", build_agent_url("/v1/agent/commands"),
                    headers={"Accept": "text/event-stream"})
        resp = conn.getresponse()

        if resp.status != 200:
            log("ERROR", f"Command stream failed: {resp.status}")
            return

        for line in read_lines(resp):
            msg = json.loads(line)
            yield msg

    except Exception as e:
        log("ERROR", f"Command stream error: {e}")
```

Reconnects with exponential backoff: 2s initial (REPL_SSE_RECONNECT_BASE_S),
64s max (REPL_SSE_RECONNECT_MAX_S), 10 retries (REPL_SSE_MAX_RETRIES).

## 11.7 Agent Configuration

### Timing Constants

Configurable via environment variables (cloud-init user data). Constraints:
HEARTBEAT_INTERVAL < HEARTBEAT_DEGRADED < HEARTBEAT_TIMEOUT < STALE_DETECTION
(control plane, see Ch15). RECONNECT_BASE < RECONNECT_MAX.

| Constant                | Default | Env Variable                  | Purpose                       |
|-------------------------|---------|-------------------------------|-------------------------------|
| HEARTBEAT_INTERVAL_MS   | 10000   | REPL_HEARTBEAT_INTERVAL_MS    | Agent -> control heartbeat    |
| HEARTBEAT_DEGRADED_MS   | 120000  | REPL_HEARTBEAT_DEGRADED_MS    | Degraded state threshold (2m) |
| HEARTBEAT_TIMEOUT_MS    | 900000  | REPL_HEARTBEAT_TIMEOUT_MS     | Panic/shutdown trigger (15m)  |
| PANIC_CHECKPOINT_MS     | 300000  | REPL_PANIC_CHECKPOINT_MS      | Checkpoint budget during panic|
| LOG_FLUSH_INTERVAL_MS   | 100     | REPL_LOG_FLUSH_INTERVAL_MS    | Batched log upload            |
| SPOT_CHECK_INTERVAL_MS  | 5000    | REPL_SPOT_CHECK_INTERVAL_MS   | AWS metadata polling          |
| SHUTDOWN_GRACE_PERIOD_S | 5       | REPL_SHUTDOWN_GRACE_PERIOD_S  | Delay before self-termination |
| SSE_RECONNECT_BASE_S    | 2       | REPL_SSE_RECONNECT_BASE_S     | Initial reconnect delay       |
| SSE_RECONNECT_MAX_S     | 64      | REPL_SSE_RECONNECT_MAX_S      | Maximum reconnect delay       |
| SSE_MAX_RETRIES         | 10      | REPL_SSE_MAX_RETRIES          | Reconnects before shutdown    |

**Heartbeat Timeout Constraint:** Agent self-terminates after 15m without heartbeat
acknowledgment (HEARTBEAT_TIMEOUT_MS = 900000). Control plane force-terminates
instances after 25m without heartbeat (see Ch15 §15.9 for INSTANCE_STALE_THRESHOLD_MS).
This 10-minute gap ensures agent-initiated cleanup completes before provider-initiated
termination, reducing false positives for long-running jobs while maintaining safety.

### When to Adjust

| Adjustment                 | Use Case                 | Trade-off                      |
|----------------------------|--------------------------|--------------------------------|
| Longer heartbeat (20-30s)  | Stable, long-running     | Slower dead instance detection |
| Shorter heartbeat (5s)     | High-churn ephemeral     | Higher control plane load      |
| Longer log flush (500ms-1s)| Heavy logging            | Delayed UI visibility          |
| Shorter spot check (2-3s)  | Critical spot workloads  | Higher metadata API load       |
| Shorter heartbeat timeout  | Dev/test, fast failure   | More false positives, less     |
| (30s-60s)                  | detection needed         | time for graceful checkpoint   |
| Shorter degraded threshold | Earlier UX warning       | More noise for brief hiccups   |
| (30s-60s)                  |                          |                                |

## 11.8 Lock Acquisition

Prevents duplicate agents on warm pool reuse. Lock file `/tmp/repl-agent.lock`
persists across restarts within same boot; cleared on termination.

```python
def acquire_lock():
    lock_file = "/tmp/repl-agent.lock"
    try:
        lock_fd = open(lock_file, 'w')
        fcntl.flock(lock_fd, fcntl.LOCK_EX | fcntl.LOCK_NB)
        return lock_fd
    except IOError:
        return None

lock_fd = acquire_lock()
if not lock_fd:
    log("FATAL", "Another agent already running")
    sys.exit(1)
```

## 11.9 Spot Interruption Handling

AWS spot interruption gives 2-minute warning via instance metadata:

```python
def check_spot_interruption():
    if PROVIDER != "aws":
        return None

    try:
        # Get IMDSv2 token
        token_req = urllib.request.Request(
            "http://169.254.169.254/latest/api/token",
            method="PUT",
            headers={"X-aws-ec2-metadata-token-ttl-seconds": "21600"}
        )
        with urllib.request.urlopen(token_req, timeout=2) as resp:
            token = resp.read().decode()

        # Check for interruption notice
        interrupt_req = urllib.request.Request(
            "http://169.254.169.254/latest/meta-data/spot/instance-action",
            headers={"X-aws-ec2-metadata-token": token}
        )

        try:
            with urllib.request.urlopen(interrupt_req, timeout=2) as resp:
                return json.loads(resp.read().decode())
        except urllib.error.HTTPError as e:
            if e.code == 404:
                return None  # No interruption pending
            raise
    except:
        return None
```

**Interruption Response (Two-Phase Protocol):**
1. Set `spot_interrupted` flag
2. Send `spot_interrupt_start` message (interrupt detected, starting checkpoint)
3. Execute checkpoint script if configured (see below)
4. Upload partial artifacts (best effort)
5. Send `spot_interrupt_complete` message (checkpoint done, artifacts uploaded)
6. Send `run_complete` with `interrupted: true`
7. Terminate run process

Polling interval balances responsiveness with metadata API rate limits.
Configurable via REPL_SPOT_CHECK_INTERVAL_MS (see Ch15 §15.9).

**User Checkpoint Script:**

If repl.toml specifies `checkpoint_script`, agent executes it before artifacts,
allowing user code to save state. Script path relative to repl.toml; timeout
via `checkpoint_timeout` (see Ch10 for configuration).

**Checkpoint Environment Variables:**
- `SKYREPL_CHECKPOINT_REASON=spot_termination`
- `SKYREPL_RUN_ID=run-xyz789`
- `SKYREPL_WORKDIR=/home/ubuntu/work`

Example checkpoint script:

```python
#!/usr/bin/env python3
# checkpoint.py - Save model state on spot interruption
import torch
import os

model = load_current_model()  # User's model loading logic
checkpoint_dir = "checkpoints"
os.makedirs(checkpoint_dir, exist_ok=True)

# Save checkpoint with timestamp
checkpoint_path = f"{checkpoint_dir}/spot_interrupt_{int(time.time())}.pt"
torch.save({
    'epoch': current_epoch,
    'model_state_dict': model.state_dict(),
    'optimizer_state_dict': optimizer.state_dict(),
    'loss': current_loss,
}, checkpoint_path)

print(f"Saved checkpoint to {checkpoint_path}")
```

Checkpoint output streams as regular logs. Script terminated if exceeds timeout,
then proceeds with artifact upload.

## 11.10 Workdir Cleanup

Called at start of `start_run` before downloading files. Critical for warm pool:
previous run's secrets visible (security), stale files affect behavior
(correctness), accumulated files fill disk.

```python
def cleanup_workdir(workdir):
    if os.path.exists(workdir):
        os.chdir(os.path.expanduser("~"))  # Avoid error if cwd was workdir
        try:
            shutil.rmtree(workdir)
        except Exception as e:
            log("WARN", f"Error removing workdir: {e}")
```

## 11.11 Presigned URL Abstraction

Control plane generates presigned URLs for all file operations. Agent code
unchanged - same PUT/GET regardless of backing store.

| Size    | URL Points To          | Benefit                                |
|---------|------------------------|----------------------------------------|
| >64KB   | S3 directly            | Control plane never sees payload       |
| <=64KB  | Control plane endpoint | Avoids S3 round-trip, lower latency    |

Agent just does `urllib.request.Request(url, data=file_data, method="PUT")`.
See Ch2 for 64KB threshold rationale.

## 11.12 JIT Tailscale Installation

Tailscale installed on-demand when SSH requested or instance enters warm pool.
Uses static binary (faster than apt, cross-distro, no lock contention).

Benefits: faster startup (no bootstrap install), only pay overhead when SSH
needed, auth key injected at install time (no baked secrets).

```python
def install_tailscale_jit():
    if tailscale_status in ("ready", "installing"):
        return tailscale_status == "ready"

    existing_ip = detect_tailscale_ip()
    if existing_ip:
        tailscale_ip = existing_ip
        tailscale_status = "ready"
        return True

    tailscale_status = "installing"
    # ... download static binary, start daemon, authenticate
```

## 11.13 Error Handling Philosophy

**Transient (retry with backoff):** Network timeout, SSE dropped, presigned URL
expired.

**Fatal (clean shutdown):** Missing env vars, lock failed, heartbeat timeout,
max retries exhausted.

No complex recovery. If fundamentally wrong, terminate and let control plane
spawn fresh instance rather than limp along degraded.

## 11.14 Failsafe Self-Termination Diagnostics

Best-effort diagnostic transmission to `/v1/instances/:id/panic` before shutdown
(heartbeat timeout, SSE loss, fatal errors, signals, spot interruption,
self-termination).

```python
def send_panic_diagnostics(reason: str):
    """
    Best-effort transmission of diagnostics before shutdown.
    Does not block shutdown - returns immediately on failure.
    """
    try:
        payload = {
            "instance_id": INSTANCE_ID,
            "reason": reason,
            "last_state": {
                "workflow_state": current_workflow_state,
                "run_id": current_run_id,
                "allocation_id": current_allocation_id,
                "uptime_seconds": time.time() - agent_start_time,
                "last_heartbeat_ack": last_heartbeat_ack_time,
                "sse_reconnect_count": sse_reconnect_count,
            },
            "diagnostics": {
                "cpu_percent": psutil.cpu_percent(),
                "memory_percent": psutil.virtual_memory().percent,
                "disk_percent": psutil.disk_usage('/').percent,
                "active_threads": threading.active_count(),
                "log_buffer_size": len(log_buffer),
            },
            "error_logs": get_recent_error_logs(max_lines=50),
            "timestamp": int(time.time() * 1000),
        }

        # 2-second timeout - don't block shutdown
        response = http_post(
            f"{CONTROL_PLANE}/v1/instances/{INSTANCE_ID}/panic",
            json=payload,
            timeout=2.0
        )

        if response.status == 200:
            log("INFO", f"Panic diagnostics sent: {reason}")
        else:
            log("WARN", f"Panic endpoint returned {response.status}")

    except Exception as e:
        # Failures are expected and acceptable
        log("WARN", f"Failed to send panic diagnostics: {e}")
```

Shutdown sequence with panic: set flag → send diagnostics (2s timeout) →
Tailscale logout → terminate run → flush logs → grace period → self-terminate.

Best-effort: proceeds regardless of transmission success. Useful for debugging
mysterious disappearances (post-mortem context). 70-90% success rate sufficient
since it supplements normal logging.
