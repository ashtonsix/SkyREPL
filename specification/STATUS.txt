1. SPEC OVERVIEW
----------------

SkyREPL v2 is a cloud orchestration platform for running computational workloads
on ephemeral compute instances. Part of the Sky Computing ecosystem (SkyPilot as
prominent touchstone), SkyREPL focuses specifically on lifecycle management.

The spec (~15,000 lines, 15 chapters) is complete and ready for implementation.

Key design goals:
- Explicit ALLOCATIONS replace implicit grace periods (6-state machine)
- Explicit workflow DAGs replace scattered state machines
- Warm pool optimization (multiple runs per instance)
- Multi-user/team support (development-focused but collaboration-ready)

2. ARCHITECTURE
---------------

Three-tier ownership model: MANIFEST → RESOURCE → MATERIAL

MATERIAL (Ch 2):
  - RECORDS: SQL tables (instances, runs, allocations, workflows)
  - OBJECTS: Blobs with tags (INTEGER PK, checksum as tag for dedup)
  - EXTERNAL: Provider state via API (cached in-memory)

RESOURCE (Ch 3):
  - Logical entities: Instance, Run, Allocation, Workflow, Snapshot, Artifact
  - Materialized from MATERIAL sources

MANIFEST (Ch 4):
  - Ownership boundary between workflows
  - States: DRAFT → SEALED (expired derived from resource count)

ALLOCATIONS (Ch 5):
  - Run-to-instance bindings
  - AVAILABLE → CLAIMED → ACTIVE → HELD → COMPLETE (+ FAILED)
  - Warm pool = allocations WHERE status='AVAILABLE'
  - Tailscale JIT install via ProxyCommand on first SSH

WORKFLOWS (Ch 6):
  - DAG execution engine
  - 4 patterns: Insert-and-Reconverge, Conditional-Branch, Parallel-Fan-Out,
    Retry-with-Alternative
  - Single-step helpers hide complexity for simple operations

INTENTS (Ch 7):
  - Offline types/interfaces for provider implementation
  - Testing/verification framework
  - Providers compose helpers with provider-specific API calls

Minimum Viable Kernel:
  - Single intent (launch-run) + OrbStack provider
  - Workflow engine with Conditional-Branch pattern
  - Allocations: CLAIMED → ACTIVE → COMPLETE (skip HELD initially)
  - Manifests: DRAFT → SEALED
  - Basic material layer (instances, runs, allocations)

The Core Insight:
  Cloud compute orchestration isn't fundamentally a compute problem - it's a
  lifecycle chaos problem. Everyone can spin up instances. The hard part is
  tracking what's running, ensuring cleanup on failure, and avoiding cost
  surprises. SkyREPL treats resource lifecycle as a first-class problem.
