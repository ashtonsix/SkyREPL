================================================================================
3. RESOURCES
================================================================================

## Purpose

Resources are the logical entities we reason about in SkyREPL. This section
specifies how resources are materialized from underlying materials, how they
model state and lifecycle, and how reconciliation handles external state drift.

## ID System

All resource IDs are integers (INTEGER PRIMARY KEY) in the database. They are
displayed to users as slugs for human readability.

**Slug Definition:**
A slug is a base-36 alphanumeric representation that directly corresponds to
the integer ID after base conversion. Example: ID 12345 → slug "9ix"

CONVERSION:
- Display format: `id.toString(36)` (e.g., 12345 → "9ix")
- Parse from slug: `parseInt(slug, 36)` (e.g., "9ix" → 12345)

EXCEPTION - Content-Addressable Material:
The `objects` table now uses INTEGER PRIMARY KEY AUTOINCREMENT (changed from
TEXT checksum for multi-tenant support). Checksums are stored as tags to enable
deduplication via query-before-insert pattern. This allows bucket-based ACLs
and cost attribution while preserving content-addressable semantics.

## Architecture Recap

- RESOURCE = logical item materialized by pulling from various material sources
- Some resources are COMPLEX: backed by records + objects + external state
- Some resources are SIMPLE: backed by single object
- Status model: `workflow_state` uses "<workflow>:<phase>" format throughout lifecycle
- Availability is INFERRED, not stored: `isAvailable(r) = r.workflow_state.endsWith(':complete') && r.current_manifest_id === null`
- Ownership: Resources must always have clear owner (manifest or detached)

## 3.1 Resource Type Catalog

This catalog defines all resource types in the system, their material backing,
key fields, and relationships.

### 3.1.1 Complex Resources

These resources combine multiple material sources and have rich state machines.

#### Instance

The fundamental compute resource - a VM or container running workloads.

MATERIAL BACKING:
- Primary: instances table (records)
- Secondary: objects table (feature installations via feature_installation type)
- External: Provider API (IP address, actual status, provider-specific metadata)

KEY FIELDS:
- id: integer, primary key
- provider: string (aws, lambda, runpod, orbstack)
- provider_id: string (ID in provider's system, for external API calls)
- spec: string (instance type/spec, e.g., "g5.xlarge", "ubuntu:noble:arm64")
- region: string (provider region/zone, use 'local' for local instances)
- ip: string | null (cached from external, TTL-based)
- workflow_state: string (format: "<workflow>:<phase>", e.g., 'launch-run:provisioning', 'spawn:complete')
- workflow_error: string | null (error details if workflow_state ends in ':error')
- is_spot: boolean
- spot_request_id: string | null (for AWS spot tracking)
- init_checksum: string | null (identifies init script for warm pool matching)
- created_at: timestamp
- last_heartbeat: timestamp
- current_manifest_id: number | null (ownership tracking)

RELATIONSHIPS:
- Has many: Allocations (instance can host multiple allocations over time)
- Has many: FeatureInstallations (stored as objects, linked by instance_id tag)
- Has one: TailscaleMachine (stored as object, linked by instance_id tag)
- Belongs to: Manifest (optional, via current_manifest_id)
- Managed by: Provider (external system)

MATERIALIZATION SOURCES:
1. Base record from instances table
2. Feature installations from objects WHERE type='feature_installation' AND tags.instance_id=id
3. Tailscale machine from objects WHERE type='tailscale_machine' AND tags.instance_id=id
4. External state from provider.get(provider_id) - cached with TTL

LIFECYCLE STATES (workflow_state examples):
- "spawn:pending" - DB record created, awaiting provider.spawn() (two-phase spawn)
- "launch-run:provisioning" - Being created by launch-run workflow
- "launch-run:bootstrapping" - Cloud-init running
- "launch-run:complete" - Workflow complete, instance ready
- "terminate:draining" - Terminate workflow started
- "terminate:complete" - Terminated

AVAILABILITY INFERENCE:
Availability is NOT a stored state. Use the helper function:
```typescript
function isAvailable(instance: Instance): boolean {
  return instance.workflow_state?.endsWith(':complete') &&
         instance.current_manifest_id === null;
}
```

NOTE: Tailscale state (IP, status) is NOT in instances table. It's stored in
TailscaleMachine object only, since it's an external entity requiring cleanup.

#### Run

A job execution - the command, workdir, files, and execution metadata.

MATERIAL BACKING:
- Primary: runs table (records)
- Secondary: objects with type='run_file' (file manifests, many-to-one)
- Secondary: objects with type='log' (streaming logs, many-to-one)
- Secondary: objects table (artifacts via artifact type)
- External: None (pure control plane state)

KEY FIELDS:
- id: integer, primary key
- command: string (shell command to execute)
- workdir: string (working directory path)
- max_duration_ms: number (timeout)
- workflow_state: string (format: "<workflow>:<phase>", e.g., 'launch-run:running')
- workflow_error: string | null (error details if workflow_state ends in ':error')
- exit_code: number | null
- create_snapshot: boolean (snapshot after completion)
- init_checksum: string | null (links to instance init)
- spot_interrupted: boolean
- created_at: timestamp
- started_at: timestamp | null
- finished_at: timestamp | null
- current_manifest_id: number | null

RELATIONSHIPS:
- Has many: RunFiles (file manifest entries)
- Has many: RunLogs (log stream entries)
- Has many: Artifacts (stored as objects, linked by run_id tag)
- Has one: Allocation (run-to-instance binding)
- Belongs to: Manifest (optional, via current_manifest_id)

MATERIALIZATION SOURCES:
1. Base record from runs table
2. File objects from objects JOIN object_tags WHERE type='run_file' AND tag='run_id:'||id
3. Log objects from objects WHERE type='log' AND tag='run_id:'||id ORDER BY timestamp DESC LIMIT N
4. Artifacts from objects WHERE type='artifact' AND tags.run_id=id
5. Allocation via allocations WHERE run_id=id (should be 1:1 during execution)

LIFECYCLE STATES (workflow_state examples):
- "launch-run:pending" - Created, waiting for allocation
- "launch-run:preparing" - Workdir being set up
- "launch-run:running" - Command executing
- "launch-run:complete" - Completed successfully
- "launch-run:error" - Execution failed (details in workflow_error)
- "launch-run:cancelled" - User-cancelled
- "launch-run:timeout" - Max duration exceeded

AVAILABILITY INFERENCE:
```typescript
function isAvailable(run: Run): boolean {
  return run.workflow_state?.endsWith(':complete') &&
         run.current_manifest_id === null;
}
```

#### Allocation

The run-to-instance binding with its own lifecycle. Replaces grace_period.

MATERIAL BACKING:
- Primary: allocations table (records)
- External: None (pure control plane state)

KEY FIELDS:
- id: integer, primary key
- run_id: integer | null (null for AVAILABLE state - this IS the warm pool)
- instance_id: integer
- user: string (Unix username on instance, platform-dependent)
- workdir: string (allocated directory path)
- status: string (state machine: AVAILABLE → CLAIMED → ACTIVE → HELD → COMPLETE → FAILED)
- hold_until: timestamp | null (for HELD state, supports SSH access)
- hold_reason: string | null (why held: "ssh_active", "manual_extend", etc.)
- created_at: timestamp
- updated_at: timestamp
- current_manifest_id: number | null

RELATIONSHIPS:
- Belongs to: Instance (required, via instance_id)
- Belongs to: Run (optional, via run_id - null in AVAILABLE state)
- Belongs to: Manifest (optional, via current_manifest_id)

MATERIALIZATION SOURCES:
1. Record from allocations table
2. Related instance via instances WHERE id=instance_id
3. Related run (if run_id not null) via runs WHERE id=run_id

LIFECYCLE STATES (status field):
- "AVAILABLE" - Instance advertises empty slot (THIS IS THE WARM POOL)
- "CLAIMED" - Run assigned, workdir being prepared
- "ACTIVE" - Run executing
- "HELD" - Run finished but allocation preserved (SSH access, debugging)
- "COMPLETE" - Hold expired, allocation finalized
- "FAILED" - Lifecycle error (e.g., file sync failure)

KEY INSIGHT: Warm pool query is just:
```sql
SELECT * FROM allocations
WHERE status = 'AVAILABLE'
AND instance_id IN (SELECT id FROM instances WHERE spec = ? AND init_checksum = ?)
```

#### Workflow

A multi-step operation (DAG execution). All orchestration goes through workflows.

MATERIAL BACKING:
- Primary: workflows table (records)
- Secondary: workflow_nodes table (node execution state, many-to-one)
- Secondary: objects table (workflow-specific state via blob type)
- External: None (pure control plane state)

KEY FIELDS:
- id: integer, primary key
- type: string (launch-run, terminate-instance, create-snapshot, cleanup, etc.)
- parent_workflow_id: integer | null (for subworkflows)
- status: string (pending, running, completed, failed, cancelled)
- current_node: string | null (currently executing node ID)
- input_json: string (JSON-encoded input parameters)
- output_json: string | null (JSON-encoded output/results)
- error_message: string | null
- created_at: timestamp
- started_at: timestamp | null
- finished_at: timestamp | null
- current_manifest_id: number | null (workflow's own manifest for resources)

RELATIONSHIPS:
- Has many: WorkflowNodes (execution nodes)
- Has one: Manifest (workflow creates manifest for its resources)
- Has one: Parent Workflow (optional, for subworkflows)
- Has many: Child Workflows (subworkflows spawned by nodes)

MATERIALIZATION SOURCES:
1. Base record from workflows table
2. Nodes from workflow_nodes WHERE workflow_id=id
3. Manifest via manifests WHERE workflow_id=id


LIFECYCLE STATES (status field):
- "pending" - Created, not yet started
- "running" - Executing nodes
- "completed" - Successfully finished
- "failed" - Error during execution
- "cancelled" - User-cancelled or aborted
- "rolling_back" - Compensation in progress

### 3.1.2 Simple Resources

These resources are backed by a single object in the objects table.

#### Snapshot

A point-in-time disk image for fast instance spawning.

MATERIAL BACKING:
- Primary: objects table (type='snapshot')
- External: Provider API (provider's snapshot/image system)

KEY FIELDS (stored in objects.tags_json):
- id: integer (snapshot ID for references)
- provider: string
- provider_id: string (ID in provider's snapshot system)
- spec: string (instance type this snapshot is for)
- init_checksum: string (init script this snapshot includes)
- name: string (user-friendly name)
- created_at: timestamp
- last_used_at: timestamp
- use_count: number

PAYLOAD (objects.payload):
- Binary JSON encoding provider-specific metadata

RELATIONSHIPS:
- Referenced by: SpawnOptions (instances can spawn from snapshots)
- Managed by: Provider (external system)

MATERIALIZATION:
1. Get object from objects WHERE id=snapshot_id
2. Parse tags_json for metadata
3. Optionally fetch external state from provider.getSnapshotByName(name)

LIFECYCLE:
- Created by create-snapshot workflow
- Deleted by cleanup workflow or manual deletion
- No status field - existence is the state

#### Artifact

Output file from a run (logs, model checkpoints, results).

MATERIAL BACKING:
- Primary: objects table (type='artifact')
- External: S3/MinIO (for artifacts >4KB) OR inline payload (for artifacts ≤4KB)

KEY FIELDS (stored in objects.tags_json):
- id: integer (artifact ID for references)
- run_id: integer (which run produced this)
- path: string (relative path in workdir)
- size_bytes: number
- keep: boolean (exempt from auto-cleanup)
- created_at: timestamp

PAYLOAD (objects.payload):
- For artifacts ≤4KB: stored inline in payload column
- For artifacts >4KB: payload is null, s3_key/s3_bucket point to S3

RELATIONSHIPS:
- Belongs to: Run (via run_id tag)
- Stored in: objects table (inline) OR S3 (external blob storage)

MATERIALIZATION:
1. Get object from objects WHERE id=artifact_id
2. Parse tags_json for metadata
3. Generate presigned URL (S3 URL for large artifacts, control plane endpoint URL for small artifacts)
4. Agent/CLI downloads via same HTTP GET regardless of storage location

LIFECYCLE:
- Created when agent uploads artifact
- Marked keep=true if user flags for retention
- Deleted by cleanup workflow based on retention policy

#### TailscaleMachine

Tailscale network configuration for an instance. This is an external entity
that requires cleanup, so it's tracked as a separate object.

MATERIAL BACKING:
- Primary: objects table (type='tailscale_machine')
- External: Tailscale API (device registration, status, IP)

KEY FIELDS (stored in objects.tags_json):
- instance_id: integer
- tailscale_ip: string
- hostname: string
- status: string (not_installed, installing, ready, failed)
- auth_key_used: boolean (whether auth key was consumed)
- created_at: timestamp

PAYLOAD (objects.payload):
- Binary JSON encoding Tailscale device metadata from API

RELATIONSHIPS:
- Belongs to: Instance (one-to-one, via instance_id tag)

MATERIALIZATION:
1. Get object from objects WHERE type='tailscale_machine' AND tags.instance_id=id
2. Parse tags_json and payload for metadata
3. Optionally fetch external state from Tailscale API

LIFECYCLE:
- Created during instance bootstrap (JIT installation)
- Status updated as installation progresses
- Deleted when instance terminates (deregister device from Tailscale API)

NOTE: Tailscale state is NOT stored in instances table. The instance just
references the TailscaleMachine object when needed.

#### FeatureInstallation

Represents a feature installed on an instance (Jupyter, code-server, etc.).

MATERIAL BACKING:
- Primary: objects table (type='feature_installation')
- External: None (installation state tracked in control plane)

KEY FIELDS (stored in objects.tags_json):
- instance_id: integer
- feature_name: string (jupyter, code-server, etc.)
- version: string
- status: string (installing, ready, failed)
- port: number | null (exposed port for feature)
- url: string | null (access URL if applicable)
- installed_at: timestamp

PAYLOAD (objects.payload):
- Binary JSON encoding feature-specific config and state

RELATIONSHIPS:
- Belongs to: Instance (via instance_id tag)

MATERIALIZATION:
1. Get all objects WHERE type='feature_installation' AND tags.instance_id=id
2. Parse tags_json for each feature
3. Return array of installed features

LIFECYCLE:
- Created by feature installation workflow/subworkflow
- Status updated as installation progresses
- Deleted when instance terminates

#### Blob

Generic binary blob with checksum-based deduplication (used for file sync).

MATERIAL BACKING:
- Primary: objects table with type='blob' (INTEGER ID, checksum stored as column)
- External: S3/MinIO (for large payloads >4KB, via s3_key/s3_bucket fields)

KEY FIELDS (stored as object record):
- id: integer (AUTOINCREMENT, changed from checksum for multi-tenant support)
- type: 'blob'
- checksum: sha256 hash (column, used for deduplication via query-before-insert)
- checksum_bytes: bytes covered by checksum (== size_bytes when current, < when stale)
- size_bytes: number
- payload: binary (inline for blobs ≤4KB) OR null with s3_key for blobs >4KB

RELATIONSHIPS:
- Referenced by: run_file objects via checksum column match

MATERIALIZATION:
1. Get object WHERE type='blob' AND checksum = $checksum
2. If s3_key present, generate presigned S3 URL for download (>4KB blobs)
3. Otherwise generate presigned control plane endpoint URL for inline payload (≤4KB blobs)
4. Agent/CLI code identical for both - same HTTP GET to presigned URL

LIFECYCLE:
- Created when agent uploads file (query by checksum before insert for dedup)
- Garbage collected when no run_file objects reference this checksum

## 3.2 Materialization

Materialization is the process of constructing a resource by fetching from
material sources and combining them. Keep it simple - it's just "fetch and
combine", not elaborate dependency resolution.

### 3.2.1 Materialization Interface

All resource materializers implement this simple interface:

```typescript
interface ResourceMaterializer<T extends Resource> {
  /**
   * Materialize resource from materials.
   * @param id - Resource identifier
   * @param options - Materialization options (TTL-based caching)
   * @returns Materialized resource or null if not found
   */
  materialize(
    id: string,
    options?: MaterializationOptions
  ): Promise<T | null>;

  /**
   * Materialize multiple resources efficiently (batch operation).
   */
  materializeBatch(
    ids: string[],
    options?: MaterializationOptions
  ): Promise<Map<string, T>>;
}

interface MaterializationOptions {
  /**
   * TTL duration in milliseconds for external state caching.
   * Default: 30000 (30 seconds).
   * Set to 0 to force fresh fetch.
   */
  ttl?: number;

  /**
   * When was the last refresh? If (last_refresh + ttl) > now, use cached.
   * If not provided, always fetches fresh.
   */
  last_refresh?: number;
}
```

PRINCIPLE: Materialization is just fetch-and-combine. No elaborate depth
tracking, no complex includeExternal flags. If you need external state, check
the TTL and fetch if stale. That's it.

### 3.2.2 Instance Materialization

Instances are the most complex resource to materialize.

```typescript
async function materializeInstance(
  id: string,
  options: MaterializationOptions = {}
): Promise<Instance | null> {
  const ttl = options.ttl ?? 30_000; // 30 seconds default
  const now = Date.now();

  // 1. Get base record
  const record = db.getInstanceRecord(id);
  if (!record) return null;

  // 2. Get feature installations
  const features = await db.getObjects({
    type: 'feature_installation',
    tags: { instance_id: id }
  });

  // 3. Get Tailscale machine (if exists)
  const tailscaleMachine = await db.getObject({
    type: 'tailscale_machine',
    tags: { instance_id: id }
  });

  // 4. Get external state if stale or missing
  let externalState = null;
  const cacheKey = `instance:${id}:external`;
  const cached = cache.get(cacheKey); // Simple in-memory cache

  const isStale = !options.last_refresh ||
                  (now - options.last_refresh) > ttl;

  if (isStale || !cached) {
    const provider = getProvider(record.provider);
    externalState = await provider.get(record.provider_id);

    if (externalState) {
      cache.set(cacheKey, externalState, now);
    }
  } else {
    externalState = cached;
  }

  // 5. Combine into materialized Instance
  const instance: Instance = {
    id: record.id,
    provider: record.provider,
    providerId: record.provider_id,
    spec: record.spec,
    region: record.region,
    ip: externalState?.ip ?? record.ip, // Prefer external (fresh)
    workflowState: record.workflow_state,
    workflowError: record.workflow_error,
    isSpot: record.is_spot === 1,
    spotRequestId: record.spot_request_id,
    initChecksum: record.init_checksum,
    createdAt: record.created_at,
    lastHeartbeat: record.last_heartbeat,
    currentManifestId: record.current_manifest_id,

    // Relationships
    features: features.map(f => parseFeatureInstallation(f)),
    tailscaleMachine: tailscaleMachine ? parseTailscaleMachine(tailscaleMachine) : null,

    // External state
    externalStatus: externalState?.status,
    externalIp: externalState?.ip,
  };

  return instance;
}
```

### 3.2.3 Run Materialization

Runs combine record data with related files, logs, and artifacts.

```typescript
async function materializeRun(
  id: string,
  options: MaterializationOptions = {}
): Promise<Run | null> {
  // 1. Get base record
  const record = db.getRunRecord(id);
  if (!record) return null;

  // 2. Get file manifest
  const files = db.getRunFiles(id);

  // 3. Get recent logs (limit to avoid huge payloads)
  const logs = db.getRunLogs(id, { limit: 1000, order: 'DESC' });

  // 4. Get artifacts
  const artifactObjects = await db.getObjects({
    type: 'artifact',
    tags: { run_id: id }
  });

  // 5. Get allocation (1:1 relationship during execution)
  const allocation = db.getAllocationByRunId(id);

  // 6. Combine into materialized Run
  const run: Run = {
    id: record.id,
    command: record.command,
    workdir: record.workdir,
    maxDurationMs: record.max_duration_ms,
    workflowState: record.workflow_state,
    workflowError: record.workflow_error,
    exitCode: record.exit_code,
    createSnapshot: record.create_snapshot === 1,
    initChecksum: record.init_checksum,
    spotInterrupted: record.spot_interrupted === 1,
    createdAt: record.created_at,
    startedAt: record.started_at,
    finishedAt: record.finished_at,
    currentManifestId: record.current_manifest_id,

    // Relationships
    files: files.map(f => ({ path: f.path, checksum: f.checksum, size: f.size_bytes })),
    logs: logs.map(l => ({ stream: l.stream, data: l.data, timestamp: l.timestamp })),
    artifacts: artifactObjects.map(a => parseArtifact(a)),
    allocation: allocation,
  };

  return run;
}
```

### 3.2.4 Simple Object Materialization

Simple objects (Snapshot, Artifact, etc.) just unwrap from objects table.

```typescript
async function materializeSnapshot(
  id: string,
  options: MaterializationOptions = {}
): Promise<Snapshot | null> {
  // 1. Get object
  const obj = await db.getObject(id);
  if (!obj || obj.type !== 'snapshot') return null;

  // 2. Parse tags
  const tags = JSON.parse(obj.tags_json);

  // 3. Optionally get external state (check if snapshot still exists)
  const ttl = options.ttl ?? 300_000; // 5 minutes for snapshots
  const now = Date.now();
  const isStale = !options.last_refresh ||
                  (now - options.last_refresh) > ttl;

  let externalExists = null;
  if (isStale) {
    const provider = getProvider(tags.provider);
    const external = await provider.getSnapshotByName(tags.name);
    externalExists = external !== null;
  }

  // 4. Combine into materialized Snapshot
  const snapshot: Snapshot = {
    id: obj.id,
    provider: tags.provider,
    providerId: tags.provider_id,
    spec: tags.spec,
    initChecksum: tags.init_checksum,
    name: tags.name,
    createdAt: tags.created_at,
    lastUsedAt: tags.last_used_at,
    useCount: tags.use_count,

    // External state
    externalExists: externalExists,
  };

  return snapshot;
}
```

### 3.2.5 Caching Strategy

Caching is an implementation detail. Use a simple in-memory cache with TTL.

CACHE STRUCTURE:
```typescript
class ResourceCache {
  private cache = new Map<string, { value: any; timestamp: number }>();

  get(key: string): any | null {
    const entry = this.cache.get(key);
    if (!entry) return null;
    return entry.value;
  }

  set(key: string, value: any, timestamp: number): void {
    this.cache.set(key, { value, timestamp });
  }

  invalidate(key: string): void {
    this.cache.delete(key);
  }

  clear(): void {
    this.cache.clear();
  }
}
```

CACHE KEYS:
- `instance:{id}:external` - External provider state for instance
- `snapshot:{id}:external` - External provider state for snapshot

CACHE INVALIDATION:
- On resource mutation (create/update/delete): Invalidate cache entry
- On workflow state change: Invalidate all resources in workflow manifest
- On reconciliation: Clear entire cache for provider

DEFAULT TTLs:
- Instance external state: 30 seconds
- Snapshot external state: 5 minutes
- Other resources: No external caching (pure control plane state)

## 3.3 Reconciliation

Reconciliation is the process of ensuring our control plane state matches
external provider state. Keep it simple - fetch both states, compare, and
decide what action to take.

### 3.3.1 Reconciliation Function

NOT a method on every materializer. Just a standalone function.

```typescript
async function reconcileInstance(id: string): Promise<ReconciliationResult> {
  // 1. Get our record
  const record = db.getInstanceRecord(id);
  if (!record) {
    return { action: 'none', reason: 'instance not found in DB' };
  }

  // 2. Get fresh external state
  const provider = getProvider(record.provider);
  const external = await provider.get(record.provider_id);

  // 3. Compare and decide action
  if (!external) {
    // Instance deleted externally
    return {
      action: 'mark_terminated',
      reason: 'instance deleted externally',
      updates: { workflow_state: 'terminate:finished' }
    };
  }

  const updates: any = {};
  let needsUpdate = false;

  // Check IP
  if (external.ip && external.ip !== record.ip) {
    updates.ip = external.ip;
    needsUpdate = true;
  }

  // Check status (only if significantly different)
  const externalMapped = mapProviderStatus(external.status);
  const recordMapped = mapProviderStatus(record.workflow_state);
  if (externalMapped !== recordMapped) {
    // Don't overwrite workflow status unless critical
    if (externalMapped === 'terminated') {
      updates.workflow_state = 'terminate:finished';
      needsUpdate = true;
    }
  }

  if (needsUpdate) {
    return { action: 'update', reason: 'state drift detected', updates };
  }

  return { action: 'none', reason: 'state matches' };
}

interface ReconciliationResult {
  action: 'none' | 'update' | 'mark_terminated' | 'abort';
  reason: string;
  updates?: Record<string, any>;
}
```

PRINCIPLE: Reconciliation sees everything in one place (our records + fresh
external state) and decides what to do. No elaborate drift reports with
severity classifications. Just: "Here's what's different, here's what I'll do."

### 3.3.2 Reconciliation Triggers

When should reconciliation run?

1. MANUAL - CLI command: `repl reconcile instances`
2. PERIODIC - Background job every 5 minutes
3. ON-DEMAND - When workflow detects unexpected state (e.g., instance not
   responding)

### 3.3.3 Reconciliation Actions

What can reconciliation do?

- `none` - State matches, nothing to do
- `update` - Update our DB to match external (IP changed, minor differences)
- `mark_terminated` - Instance deleted externally, mark as terminated in DB
- `abort` - Critical mismatch, halt active workflows, require manual intervention

WHEN TO ABORT vs UPDATE:
- If instance has active workflow and is terminated externally → ABORT
- If instance is "available" (detached) and terminated externally → UPDATE
- If IP changed but instance still running → UPDATE
- If status changed but not critical → UPDATE

## 3.4 Resource Lifecycle

Resources have status, ownership, and state transitions managed by workflows.

### 3.4.1 Status Model

All resources use workflow_state with format `<workflow>:<phase>`:

```
workflow_state = <workflow>:<phase>
```

EXAMPLES:
- "spawn:pending" - Instance DB record created, awaiting provider API call
- "launch-run:provisioning" - Instance being created by launch-run workflow
- "launch-run:complete" - Launch-run workflow completed successfully
- "terminate:complete" - Termination workflow completed

KEY DESIGN: No explicit "available" state. Availability is INFERRED:
```typescript
function isAvailable(resource: Instance | Run): boolean {
  return resource.workflow_state?.endsWith(':complete') &&
         resource.current_manifest_id === null;
}
```

This documents availability logic in one place and eliminates state synchronization bugs.

ERROR HANDLING:
When a workflow fails, set `workflow_state = '<workflow>:error'` and populate
`workflow_error` with structured error details as JSON matching the API ErrorResponse
structure.

**workflow_error format:**
```json
{
  "code": "PROVIDER_API_ERROR",
  "category": "provider",
  "message": "Spot capacity unavailable in us-east-1",
  "details": {
    "region": "us-east-1",
    "spec": "g6.xlarge",
    "alternative_regions": ["us-west-2"]
  }
}
```

The workflow_error field contains a JSON-serialized error object that matches the
structure defined in 09_API.txt (Section 9.6.1 Error Response Structure).
This enables consistent error reporting across workflow failures and API responses.

BENEFITS:
- Clear attribution: "What workflow owns this resource?"
- Phase tracking: "What phase is the workflow in?"
- Debugging: "Why is this instance in this state?"
- No separate "available" state to synchronize
- Error details preserved without overloading status

### 3.4.2 Valid States by Resource Type

#### Instance States

All states use format `<workflow>:<phase>`:

SPAWN WORKFLOW:
- "spawn:pending" - DB record created, awaiting provider.spawn() call

LAUNCH-RUN WORKFLOW:
- "launch-run:provisioning" - Being created by provider
- "launch-run:bootstrapping" - Cloud-init running
- "launch-run:installing_features" - Features being installed
- "launch-run:complete" - Workflow complete (available if unowned)
- "launch-run:error" - Error during launch (see workflow_error)

TERMINATE WORKFLOW:
- "terminate:draining" - Termination initiated
- "terminate:complete" - Terminated

CREATE-SNAPSHOT WORKFLOW:
- "create-snapshot:preparing" - Preparing for snapshot
- "create-snapshot:snapshotting" - Snapshot in progress
- "create-snapshot:complete" - Snapshot complete

Note: "available" is NOT a stored state. Check `isAvailable(instance)` instead.

#### Run States

All states use format `<workflow>:<phase>`:

LAUNCH-RUN WORKFLOW:
- "launch-run:pending" - Created, waiting for allocation
- "launch-run:preparing" - Workdir being set up, files downloading
- "launch-run:running" - Command executing
- "launch-run:complete" - Completed (check exit_code for success/failure)
- "launch-run:error" - Lifecycle error (see workflow_error)
- "launch-run:cancelled" - User-cancelled
- "launch-run:timeout" - Max duration exceeded

Note: Run success vs failure is determined by exit_code, not workflow_state.
Both exit_code=0 and exit_code!=0 result in "launch-run:complete".

#### Allocation States

Simple state machine (NO workflow prefix - allocations ARE the primitive):
- "AVAILABLE" - Instance advertises empty slot (warm pool)
- "CLAIMED" - Run assigned, workdir being prepared
- "ACTIVE" - Run executing
- "HELD" - Run finished but allocation preserved
- "COMPLETE" - Hold expired, allocation finalized
- "FAILED" - Lifecycle error (e.g., file sync failure)

#### Workflow States

Simple state machine:
- "pending" - Created, not yet started
- "running" - Executing nodes
- "completed" - Successfully finished
- "failed" - Error during execution
- "cancelled" - User-cancelled or aborted
- "rolling_back" - Compensation in progress

### 3.4.3 State Transition Rules

State transitions are ONLY performed by workflows (or workflow engine).
Direct status updates via API are forbidden (except by workflow engine).

VALID TRANSITIONS:
- Instance: spawn:pending → launch-run:provisioning → bootstrapping → installing_features → complete
- Instance: *:complete → terminate:draining → terminate:complete
- Run: launch-run:pending → preparing → running → complete
- Allocation: AVAILABLE → CLAIMED → ACTIVE → HELD → COMPLETE

INVALID TRANSITIONS (will fail validation):
- Instance: launch-run:complete → launch-run:provisioning (can't go backwards)
- Run: launch-run:running → launch-run:pending (can't reset)
- Allocation: ACTIVE → AVAILABLE (can't reclaim)

ENFORCEMENT:
```typescript
function validateStateTransition(
  resourceType: string,
  currentStatus: string,
  newStatus: string
): { valid: boolean; error?: string } {
  const transitions = STATE_TRANSITION_RULES[resourceType];
  const allowed = transitions[currentStatus] ?? [];

  if (!allowed.includes(newStatus)) {
    return {
      valid: false,
      error: `Invalid transition for ${resourceType}: ${currentStatus} → ${newStatus}`
    };
  }

  return { valid: true };
}
```

### 3.4.4 Ownership Rules

Resources must ALWAYS have clear ownership. Ownership is tracked via manifests.

OWNERSHIP STATES:
1. OWNED BY MANIFEST - current_manifest_id is set
   - Manifest has active workflow
   - Resource lifecycle managed by workflow
   - Cannot be taken by other workflows (ownership conflict)

2. DETACHED - current_manifest_id is null
   - Manifest sealed (SEALED state, EXPIRED is derived from resource count)
   - Resource eligible for adoption by new workflow
   - Examples: available instances in warm pool

3. OWNERSHIP TRANSFER - via manifest release/take protocol
   - Workflow A releases manifest (sets current_manifest_id = null on resources)
   - Workflow B takes manifest or individual resources
   - Atomic transaction to prevent conflicts

OWNERSHIP CONFLICT RESOLUTION:
```typescript
async function claimResource(
  resourceId: string,
  newManifestId: string
): Promise<{ success: boolean; error?: string }> {
  return db.transaction(() => {
    const resource = db.getResourceById(resourceId);

    if (resource.current_manifest_id && resource.current_manifest_id !== newManifestId) {
      // Already owned by another manifest
      return {
        success: false,
        error: `Resource ${resourceId} owned by manifest ${resource.current_manifest_id}`
      };
    }

    // Claim resource
    db.updateResource(resourceId, { current_manifest_id: newManifestId });

    return { success: true };
  });
}
```

OWNERSHIP GUARANTEES:
- A resource can only be in one manifest at a time
- Manifest deletion releases all owned resources (sets current_manifest_id = null)
- Workflow compensation respects ownership (won't destroy transferred resources)
- Dangling resources (no manifest, no workflow) are detected by reconciliation
