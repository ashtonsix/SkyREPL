1. INTRODUCTION
================================================================================

1.1 WHAT IS SKYREPL?
--------------------

SkyREPL makes cloud compute access feel as natural as local development, by
treating resource lifecycle - not just compute - as a first-class problem.

Sky Computing is a new compute paradigm proposed by academia that aims to enable
seamless multi-cloud orchestration. SkyPilot is the most prominent project in
this ecosystem, focusing on cluster management and job scheduling. SkyREPL takes
a complementary approach, emphasizing lifecycle management and reproducibility
for development workflows.

The system consists of three components:
- A control plane deployed on a VPS
- A CLI client for user interaction
- A Python agent that runs on remote instances

Target users: ML researchers and engineers who value reproducibility and
cost-efficiency. The kind of person who has a "forgot to terminate an instance"
horror story and never wants another one.

Supported providers (v2 vision): The v2 architecture is designed for broad
multi-cloud support, aiming to abstract provider-specific details behind a
unified interface. Target provider ecosystem includes:

- **Major clouds**: AWS, GCP, Azure
- **GPU-focused clouds**: Lambda Labs, RunPod, CoreWeave, Vast.ai
- **HPC/clusters**: SLURM (for traditional cluster environments)
- **Local development**: OrbStack (for testing workflows locally)

This is the v2 ambition. The legacy system currently supports AWS, Lambda Labs,
RunPod, and OrbStack, but the redesigned provider system (Chapter 8) is built to
accommodate this broader set of backends with minimal per-provider code.


1.2 THE CORE INSIGHT
--------------------

Cloud compute orchestration isn't fundamentally a compute problem - it's a
lifecycle chaos problem. Everyone can spin up instances. The hard part is
tracking what's running, ensuring cleanup on failure, and avoiding cost
surprises.

This spec crystallizes around three concepts:

1. ALLOCATIONS AS LIFECYCLE PRIMITIVE
   The warm pool isn't a separate thing - it's a query: `WHERE status='AVAILABLE'`.
   HELD state replaces vague "grace periods" with explicit semantics. Every
   run-to-instance binding has a clear state machine.

2. MANIFESTS AS OWNERSHIP BOUNDARIES
   Every resource has a clear owner. When a workflow fails, its manifest tells
   you exactly what needs cleanup. No more orphaned resources from crashed
   operations.

3. WORKFLOWS AS EXPLICIT STATE MACHINES
   Replace implicit state scattered through 3000 lines of control.ts with
   visible DAGs. Fixed pattern library (4 patterns). Each step has compensation
   actions for rollback.

1.4 ARCHITECTURE OVERVIEW
-------------------------

Three-tier ownership model:

    MANIFEST -> RESOURCE -> MATERIAL

1.4.1 Foundation
    MATERIAL (Chapter 2)
        The underlying data sources. Three types:
        - RECORDS: Dedicated SQL tables (instances, runs, allocations, workflows)
        - OBJECTS: Binary blobs in shared table with tags (flexible schema)
        - EXTERNAL: Provider state accessed via API (cached in-memory only)

    RESOURCE (Chapter 3)
        Logical entities materialized from materials. Types: Instance, Run,
        Allocation, Workflow, Snapshot, Artifact. Resources are the domain objects
        the system reasons about.

    MANIFEST (Chapter 4)
        Ownership boundary between workflows. Tracks all resources created by a
        workflow. States: DRAFT -> SEALED (EXPIRED derived from resource count).
        Enables clean handoff between operations without coupling them.

1.4.2 Execution
    ALLOCATIONS (Chapter 5)
        Run-to-instance bindings with explicit state machine. Warm pool is a query,
        not a separate system. HELD state for explicit lifecycle control.

    WORKFLOWS (Chapter 6)
        DAG-based execution engine with fixed pattern library (4 patterns). Each
        step has compensation actions for rollback on failure.

    INTENTS (Chapter 7)
        High-level operations (launch, terminate, snapshot, cleanup) that combine
        allocations and workflows to achieve user goals.

1.4.3 Interface
    PROVIDERS (Chapter 8)
        Cloud provider abstraction layer supporting AWS, Lambda Labs, RunPod, and
        OrbStack. Normalizes instance lifecycle and provider-specific quirks.

    API (Chapter 9)
        Two-layer HTTP API design. Core operations and workflow orchestration
        exposed through RESTful endpoints.

    CLI (Chapter 10)
        Command-line interface with consistent output formatting. User-facing
        commands for instance management and workflow control.

    AGENT (Chapter 11)
        Python agent running on remote instances. Handles communication between
        control plane and remote execution environment.

1.4.4 Operations
    ORPHANS (Chapter 12)
        Detection and cleanup of untracked cloud resources. Reconciliation between
        control plane state and actual provider state.

    TYPES (Chapter 13)
        TypeScript type organization and conventions. Shared types across layers.

    IMPLEMENTATION (Chapter 14)
        Rollout phases and migration approach from current codebase to redesigned
        architecture.

    OPERATIONS (Chapter 15)
        Observability, billing tracking, deployment strategies, and operational
        concerns.


1.5 READING GUIDE
-----------------

Recommended reading order:

1. Material (2) -> Resource (3) -> Manifest (4)
   Foundation. Read these first to understand the data model.

2. Allocation (5) + Workflows (6) in parallel
   Execution primitives. Both depend on Manifest (4), independent of each other,
   and combine in Intents (7).

3. Intents (7)
   Where allocations and workflows come together. The core operations.

4. Provider (8) -> API (9) -> CLI (10), Agent (11) in parallel
   Interface layer. How users and instances interact with the system.

5. Orphans (12) -> Types (13) -> Implementation (14) -> Operations (15)
   Synthesis and operational concerns. Read last.
