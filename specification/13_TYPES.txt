================================================================================
13. TYPES
================================================================================

## Purpose

Complete TypeScript type definitions for all SkyREPL entities, organized by layer
to avoid circular dependencies. Three distinct type categories:

- **Record types** (db.types.ts): Raw database rows, snake_case, primitive types
- **Materialized types** (types.ts): Rich domain objects, camelCase, with relationships
- **API types** (api/*.types.ts): HTTP request/response shapes

Goals: Proper use of generics, no `any` types, clear separation between layers.

## 13.1 File Organization

### 13.1.1 Directory Structure

Types are organized by component (control/cli/shared/agent) with a paired
file pattern. The control plane uses layered directories internally.

**Dependency Rules:**
- control/ imports from shared/ only (never cli/)
- cli/ imports from shared/ only (never control/)
- shared/ has zero runtime dependencies
- Within control/: material/ ← resource/ ← provider/ ← workflow/ ← intent/

### 13.1.2 Paired Files Pattern

Each implementation file (foo.ts) has a corresponding type file (foo.types.ts).

WHAT GOES IN foo.types.ts:
- Interfaces and types used ONLY by foo.ts
- Request/response types for APIs defined in foo.ts
- Internal state types for foo.ts
- Type guards and validation schemas for foo.ts

WHAT DOESN'T GO IN foo.types.ts:
- Types shared across multiple files (use types.ts)
- Database record types (use control/src/material/db.types.ts)
- API contract types (use shared/src/api/)

### 13.1.3 Control/Shared Type Split

Types split between control/ and shared/ based on consumer:

**shared/src/api/*.ts** - HTTP contract types (request/response shapes)
- Used by CLI to construct requests and parse responses
- Used by control plane to validate and respond
- Example: LaunchRunRequest, LaunchRunResponse

**control/src/intent/*.types.ts** - Internal workflow types
- Used only within control plane workflow execution
- Not exposed to CLI
- Example: LaunchRunInput (workflow engine input), LaunchRunOutput

This separation ensures CLI never imports from control/ - it communicates
via shared API contract types over HTTP.

### 13.1.4 Cross-Cutting Types (types.ts)

The types.ts file contains ONLY types that are:
1. Used by 3+ files across codebase
2. Fundamental primitives (IDs, timestamps, enums)
3. Core domain objects (materialized resources)

/**
 * Workflow-aware state format: "<workflow>:<phase>"
 * Examples: 'spawn:pending', 'launch-run:provisioning', 'terminate:complete'

 * Availability is INFERRED, not stored:
 *   isAvailable(r) = r.workflowState.endsWith(':complete') && r.currentManifestId === null

export type WorkflowState = string;

/** Parse workflow-aware status */

  workflow: string | null;
  state: string;
} {
  const colonIndex = status.indexOf(':');
  if (colonIndex === -1) {
    return { workflow: null, state: status };

  return {
    workflow: status.substring(0, colonIndex),
    state: status.substring(colonIndex + 1),
  };
}

### 13.1.5 Type Placement Decision Tree

Follow this decision tree when adding a new type:

```
Is this type a database row representation?
  YES --> db.types.ts
  NO  |
      v
Is this type an API request/response shape?
  YES --> api/{resource}.types.ts or api/workflows.types.ts
  NO  |
      v
Is this type provider-specific?
  YES --> providers/provider.types.ts
  NO  |
      v
Is this type workflow engine internals?
  YES --> workflows/engine.types.ts
  NO  |
      v
Is this type specific to a single module (foo.ts)?
  YES --> foo.types.ts (create if doesn't exist)
  NO  |
      v
Is this type used by 3+ files across src/?
  YES --> types.ts
  NO  --> types.ts (cross-cutting by elimination)
```

### 13.2.3 Status Enums (Workflow-Aware)

Status uses "<workflow>:<state>" format for active resources, or terminal states
for completed/detached resources.

```typescript
// types.ts

/** Workflow-aware status format */
export type ResourceStatus = string;

/** Status parsing result */
export interface ParsedStatus {
  workflow: string | null;
  state: string;
}

/** Parse workflow-aware status string */
export function parseStatus(status: ResourceStatus): ParsedStatus {
  const colonIndex = status.indexOf(':');
  if (colonIndex === -1) {
    return { workflow: null, state: status };
  }
  return {
    workflow: status.substring(0, colonIndex),
    state: status.substring(colonIndex + 1),
  };
}

/** Build workflow_state string */
export function buildWorkflowState(workflow: string, phase: string): WorkflowState {
  return `${workflow}:${phase}`;
}

/** @deprecated Use buildWorkflowState instead */
export const buildStatus = buildWorkflowState;

/** Check if status indicates active workflow */
export function isActiveStatus(status: ResourceStatus): boolean {
  return status.includes(':');
}

/** Check if status is terminal (no active workflow) */
export function isTerminalStatus(status: ResourceStatus): boolean {
  return !status.includes(':');
}

// Pre-defined workflow_state builders for instances
export const InstanceWorkflowState = {

  launchComplete: () => buildWorkflowState('launch-run', 'complete'),

  running: () => buildWorkflowState('launch-run', 'running'),

### 13.2.4 API Envelopes

Standard request/response wrappers for API consistency.

```typescript
// api/resources.types.ts

import type { ID, TimestampMs, ErrorCategory } from '../types';

// =============================================================================
// REQUEST ENVELOPES
// =============================================================================

/** Standard API request with optional idempotency */
export interface ApiRequest<T> {
  data: T;
  idempotencyKey?: string;
}

/** Pagination parameters */
export interface PaginationParams {
  limit?: number;     // Default 50, max 100
  offset?: number;    // Default 0
  cursor?: string;    // Alternative to offset
}

/** Sort parameters */
export interface SortParams {
  sortBy?: string;
  sortOrder?: 'asc' | 'desc';
}

/** Filter parameters (resource-specific) */
export interface FilterParams {
  status?: string | string[];
  provider?: string | string[];
  createdAfter?: TimestampMs;
  createdBefore?: TimestampMs;
}

/** List request combining filters, sort, and pagination */
export interface ListRequest extends PaginationParams, SortParams {
  filters?: FilterParams;
}

// =============================================================================
// RESPONSE ENVELOPES
// =============================================================================

/** Successful single-resource response */
export interface ApiResponse<T> {
  data: T;
  meta?: ResponseMeta;
}

/** Successful list response with pagination */
export interface ListResponse<T> {
  data: T[];
  pagination: PaginationMeta;
  meta?: ResponseMeta;
}

/** Response metadata */
export interface ResponseMeta {
  requestId: string;
  timestamp: TimestampMs;
  durationMs: number;
}

/** Pagination metadata */

  limit: number;
  offset: number;
  hasMore: boolean;
  nextCursor?: string;

  code: ErrorCodeType,
  message: string,
  category: ErrorCategory,
  details?: Record<string, unknown>,
  retryAfterMs?: number
): ApiError {
  return {
    error: { code, message, category, details, retryAfterMs },
  };
}

+-----------------------------------------------------------------------------+

|  |                         API TYPES                                     |  |

  readonly parentWorkflowId: ID | null;

  // Resource Management
  emitResource(type: ResourceType, id: ID, priority?: number): Promise<void>;
  claimResource(fromNodeId: string, type: ResourceType): Promise<ID>;
  listEmittedResources(): Promise<EmittedResource[]>;

  // Pattern Application
  applyPattern<P extends PatternType>(
    pattern: P,
    config: PatternConfig<P>
  ): void;
  readonly appliedPatterns: AppliedPattern[];

  // Subworkflow
  spawnSubworkflow(
    type: WorkflowType,
    input: Record<string, unknown>,
    options?: SubworkflowOptions
  ): Promise<SubworkflowHandle>;

  // Node Communication
  getNodeOutput(nodeId: string): Promise<Record<string, unknown> | null>;
  waitForNode(nodeId: string, timeoutMs?: number): Promise<Record<string, unknown>>;

  // Logging
  log(level: LogLevel, message: string, data?: Record<string, unknown>): void;
  debug(message: string, data?: Record<string, unknown>): void;
  info(message: string, data?: Record<string, unknown>): void;
  warn(message: string, data?: Record<string, unknown>): void;
  error(message: string, data?: Record<string, unknown>): void;

  // Utilities
  sleep(ms: number): Promise<void>;
  checkCancellation(): void;  // Throws if workflow cancelled
}

export type ResourceType = 'instance' | 'run' | 'allocation' | 'object';
export type LogLevel = 'debug' | 'info' | 'warn' | 'error';

export interface EmittedResource {
  type: ResourceType;
  id: ID;
  priority: number;
  emittedAt: TimestampMs;
}

export interface AppliedPattern {
  pattern: PatternType;
  appliedAt: TimestampMs;
  nodeId: string;
  config: Record<string, unknown>;
}

export interface SubworkflowOptions {
  timeout?: number;
  priority?: 'low' | 'normal' | 'high';
}

export interface SubworkflowHandle {
  workflowId: ID;
  manifestId: ID;
  wait(): Promise<SubworkflowResult>;
  cancel(reason: string): Promise<void>;
}

export interface SubworkflowResult {
  status: 'completed' | 'failed' | 'cancelled';
  output: Record<string, unknown> | null;
  error: WorkflowError | null;
}

// =============================================================================
// SINGLE STEP CONTEXT (Simplified Workflow Alternative)
// =============================================================================

/**

## 13.6 Guidance: Where to Put New Types

### Adding a New Resource Type

1. Define the materialized type in `types.ts`:
   ```typescript
   export interface NewResource {
     id: ID;
     // ... fields
   }
   ```

2. Define the record type in `db.types.ts`:
   ```typescript
   export interface NewResourceRecord {
     id: ID;
     // ... snake_case fields
   }
   ```

3. Add converters in `db.types.ts`:
   ```typescript
   export function recordToNewResource(record: NewResourceRecord): NewResource;
   export function newResourceToRecord(resource: NewResource): NewResourceRecord;
   ```

4. Add API types in `api/resources.types.ts`:
   ```typescript
   export interface CreateNewResourceRequest { ... }
   export type GetNewResourceResponse = ApiResponse<NewResource>;
   ```

### Adding a New Workflow Type

1. Create `workflows/workflow-{name}.types.ts`:
   ```typescript
   export interface {Name}WorkflowInput { ... }
   export interface {Name}WorkflowOutput { ... }
   ```

2. Add to `WorkflowType` union in `engine.types.ts`:
   ```typescript
   export type WorkflowType =
     | 'launch-run'
     | 'new-workflow'  // Add here
     | string;
   ```

3. Define blueprint in `workflows/workflow-{name}.ts`

### Adding Provider-Specific Types

Provider-specific types live in `providers/provider.types.ts`. Core provider
types like SpawnOptions are defined there and referenced by workflow types.

**SpawnOptions** (defined in providers/provider.types.ts):

```typescript
// Cross-reference: SpawnOptions is defined in providers/provider.types.ts
// Used by spawn-instance node executors and provider spawn() methods

interface SpawnOptions {
  spec: string;                      // Instance type (g5.xlarge, A100x1, etc)
  region?: string;                   // Target region/zone
  spot?: boolean;                    // Use spot/preemptible pricing
  maxSpotPrice?: number;             // Maximum spot price ($/hour)
  initChecksum?: string;             // Init script checksum for warm pool matching
  tags?: Record<string, string>;     // Tags to apply to instance
  bootstrap: BootstrapConfig;        // Bootstrap configuration
  snapshotId?: string;               // Spawn from snapshot
  volumeId?: string;                 // Attach existing volume
  networkConfig?: NetworkConfig;     // Subnet/network configuration
  clientRequestId?: string;          // Client-provided request ID for idempotency
}
```

**Adding custom provider types:**

1. Add to `providers/provider.types.ts`:
   ```typescript
   export interface AWSInstanceExtension {
     availabilityZone: string;
     spotRequestId: string | null;
   }
   ```

2. Use type intersection for provider-specific resources:
   ```typescript
   type AWSInstance = Instance & AWSInstanceExtension;
   ```

### Adding Pattern/Feature/Heartbeat Types

All live in `providers/provider.types.ts`:

- **New pattern**: Add to `PatternType` union, define config interface, add to `PatternConfig` discriminated union
- **New feature provider**: Define config/state types, implement `FeatureProvider<TConfig, TState>`, add to `FeatureProviderName` union
- **New heartbeat task**: Add to `HeartbeatTask.type` union, use `TaskReceipt` for completions

**Note:** Remote agent communication types (Ch11) live in `api/agent.types.ts`.
Don't confuse provider heartbeats with agent heartbeats.

## 13.9 Type Naming Conventions and Anti-Patterns

| Type Category     | Naming Pattern              | Example                    |
|-------------------|-----------------------------|----------------------------|
| Materialized      | `{Resource}`                | `Instance`, `Run`          |
| Database record   | `{Resource}Record`          | `InstanceRecord`           |
| API request       | `{Action}{Resource}Request` | `CreateRunRequest`         |
| API response      | `{Action}{Resource}Response`| `ListInstancesResponse`    |
| Workflow input    | `{Workflow}Input`           | `LaunchRunInput`           |
| Workflow output   | `{Workflow}Output`          | `LaunchRunOutput`          |
| Status enum       | `{Resource}Status`          | `AllocationStatus`         |
| Config type       | `{Feature}Config`           | `HoldDurationConfig`       |
| Options type      | `{Feature}Options`          | `MaterializationOptions`   |

### Anti-Patterns to Avoid

1. **Mixed concern types** - Don't put API shapes in db.types.ts
2. **any usage** - Use generics or unknown with type guards
3. **Inline complex types** - Extract to named interfaces
4. **Circular imports** - Use type-only imports: `import type { X } from`
5. **Overly generic types** - Prefer specific types with discriminated unions
6. **String enums for status** - Use union types with const assertions
