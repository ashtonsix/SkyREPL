================================================================================
15. OPERATIONS
================================================================================

## Purpose

Operational concerns for running SkyREPL in production: observability, CI/CD,
database migrations, auto-updates, licensing, and security. These support long-term
maintainability and enable sustainable commercial deployment.

================================================================================

## 15.1 Observability

### Telemetry Stack

OpenTelemetry for structured logging and traces, Loki for log aggregation, Grafana
for visualization. Workflows tagged with runtime topology signatures (provider,
region, instance type) enable grouping and comparison across similar executions.

### Error Reporting

Self-managed deployments have opt-in error reporting (Sentry or similar). Managed
instances have mandatory telemetry for operational support. Users control attached
metadata (stripped PII by default).

### Workflow Observability

Each workflow execution creates a trace showing step timing, decisions, retries,
and resource allocations. Topology signatures enable queries like "show all
Lambda Labs launch failures in us-west" or "compare P99 latency for warm vs cold
starts on AWS."

================================================================================

## 15.2 CI/CD and Testing

### OrbStack Generalization

OrbStack provider serves as local testing backend. All control plane logic is
testable without cloud credentials. CI runs full workflow test suite against
OrbStack in isolated environments.

**Contract-based testing philosophy:**

Intent contracts (Chapter 7) define provider-agnostic behavior: required inputs,
guaranteed outputs, allowed phases, cleanup priorities. The contract is the unit
of trust. Tests validate that the control plane handles contract-compliant behavior
correctly (retry logic, compensation, state transitions). Any provider implementing
the same contracts inherits the validated behavior - the shared 80% tests once
via OrbStack, not repeated per provider.

### Fault Injection

Chaos testing via controlled provider failures (timeouts, capacity errors, network
partitions). OrbStack implements fault injection hooks simulating real-world
failures while maintaining contract compliance. Example: simulated timeout during
provision_instance respects contract cleanup priorities and required outputs.

### Contract Compliance Testing

Each provider must pass a contract compliance test suite verifying correct
implementation. Separate from integration testing:

- Registry-based parametric testing (Chapter 8) runs the same suite against all providers
- Capability-gated tests skip when provider lacks a capability (e.g., snapshot creation)
- Explicitly verifies required fields, phase transitions, idempotency, error handling
- Passing compliance means OrbStack-validated control plane behavior applies

### Property-Based Testing

Core invariant: "Kill workflow at any step -> no orphaned resources." fast-check
generates random kill points and verifies cleanup succeeds. Additional properties
validate allocation state transitions and manifest lifecycle. Each intent's
required outputs and cleanup priorities (Chapter 7) provide test specifications.

## 15.3 Database Migrations

### Migration Files

Numbered SQL files in control/migrations/ directory: 001_initial.sql,
002_idempotency_keys.sql, etc. Applied sequentially on control plane startup.
Migration state tracked in schema_migrations table (number, applied_at, checksum).

### Rollback Support

Consider .up.sql / .down.sql pairs for reversible migrations. Critical for multi-user
deployments where downtime requires fast recovery. Not all migrations are reversible
(data transformations may be lossy).

### Migration Testing

CI validates migrations apply cleanly against empty database and realistic fixtures.
Property test: apply all migrations then rollback all → database state matches fresh init.

================================================================================

## 15.4 Auto-Updates

### Update Mechanism

Control plane checks for updates on startup (opt-out via config). Updates download
new binary and restart service. Preflight validates migration compatibility and
database backup before applying.

### Multi-User Authorization

Single-user deployments auto-update by default. Multi-user deployments (>1 active
user in last 30 days) require admin authorization. Preflight shows changelog and
estimated downtime before proceeding.

### Rollback Safety

Previous version binary retained for emergency rollback. Database migrations log
applied SQL for manual reversal. Critical failures during update trigger automatic
rollback to last known-good state.

## 15.5 License and Billing

### License Model

Source-available license (BSL 1.1 or similar). Free tier for deployments with ≤3
users. Commercial tier: $18/month base (includes 3 users), $6/month per additional
seat. License enforcement focuses on redistribution, not internal use or source
modification.

### Seat Tracking

Control plane tracks user activity per month. Users identified by API key fingerprint
or OAuth identity. Anonymous usage (no auth) counts as single shared seat. Billing
API provides monthly active user counts for self-service upgrades. Admins prompted
to deactivate users inactive >1 month (one-click deactivation frees seat but
preserves history).

### User-Month Billing

Bill user-months only when user was active (executed runs, viewed dashboard, or
invoked CLI that month). Dormant users don't count toward seat usage. Prevents
billing inflation for seasonal usage patterns.

### Enforcement Tiers

**Underprovision 1-5 seats:**
- Preflight warning on `repl run` + 5-second delay
- Warning includes upgrade link and current seat usage
- Doesn't block functionality (nag-ware, not enforcement)

**Missed 1 billing period:**
- Warning messages on CLI invocations
- Email reminders to admin contacts
- No functional restrictions yet

**>8 users + missed 2 periods:**
- Lockout with escalating warnings beforehand
- Read-only access for all users
- Admin must resolve billing to restore access

**Underprovision by >5 users:**
- New signups create "inactive" accounts with read-only access
- Admin must explicitly activate users or upgrade seat count

## 15.6 Deployment Models

### Self-Managed

User runs control plane on own VPS. Full control over data and configuration. User
responsible for updates and maintenance. Suitable for organizations with existing
infrastructure and DevOps capacity.

### Managed

Multi-tenant control plane pilots user-connected clouds. NEVER runs user code
directly (liability, AML/ATF/content concerns). Provisions infrastructure on user 
behalf, but users run code on their machines. Clear separation of control plane
orchestration from execution environment. Managed deployments handle updates,
monitoring, and availability automatically.

================================================================================

## 15.7 Security

### Authentication

Bearer tokens initially (single-user, VPS-deployed). OAuth integration encouraged
after >3 seats (GitHub, Google, corporate SSO). Tokens stored hashed in database,
never logged in plaintext.

### Authorization

Basic role system: Admin (all operations), User (run jobs, view own resources),
Viewer (read-only). Workflow permissions checked upfront before execution, not
lazily during execution. Prevents partial execution followed by auth failure.

### API Security

Rate limiting per user (100 req/min default). API keys scoped to specific operations
(run-only keys for CI, admin keys for control plane management). Token rotation
encouraged via expiry warnings (90-day default).

### Secrets Management

Run environment variables encrypted at rest. Agent decrypts on instance startup
using ephemeral key from control plane. Secrets never logged or stored in workflow
traces. Audit log tracks secret access: who, when, which run.

## 15.8 Extensibility & Future

### Runbooks

scripts/extra/ pattern for diagnostics and custom integrations. User-contributed
scripts for common tasks live outside core codebase, easily modified without
affecting system stability. Encourages community contributions without mainline code
quality concerns.

### AI Agent Piloting (Future)

Workflow observability creates datasets for agent-assisted debugging. Structured
logs with topology signatures enable ML-based anomaly detection. Example: "P99
latency spike on Lambda Labs us-west correlates with weekend spot market dynamics."

### Profiling

Covered by chaos testing + property-based tests. No separate profiling infrastructure
needed initially. Performance characteristics emerge from observability data during
normal operations. If bottlenecks appear, add targeted per-component profiling.

## 15.9 Timing Configuration

This section provides non-normative operational guidance for timing-related
configuration. All values are tunable via environment variables. Adjust based on
deployment characteristics (network latency, provider mix, workload patterns).

### Constraint Chains

Timing parameters must satisfy ordering constraints. Control plane validates
constraints at startup and fails fast on violations.

**Heartbeat Chain (liveness detection):**
```
heartbeat_interval < heartbeat_ack_timeout <= stale_detection_threshold
```
- Agent sends heartbeats faster than timeout threshold
- Allows tolerance for missed beats before declaring failure
- Control and agent agree on when to consider connection dead

**Cleanup Chain (orphan prevention):**
```
reconciliation_interval < manifest_cleanup_interval
```
- Reconciliation runs multiple times before manifest cleanup
- Gives multiple chances to detect orphans
- Manifest cleanup is fallback, not primary mechanism

**Hold Duration Chain (cost control):**
```
default_hold <= max_ssh_extension <= max_explicit_extension <= absolute_max_hold
```
- Default hold is shortest (normal case)
- SSH activity can extend, but bounded
- Explicit extension can go further, still bounded
- Absolute max prevents runaway costs

**Workflow Chain:**
```
node_timeout < workflow_timeout
compensation_timeout <= node_timeout (rollback should complete in time)
```

**Spawn/Snapshot Chain:**
```
spawn_poll_interval << spawn_timeout
snapshot_poll_interval << snapshot_timeout
orphan_min_age > spawn_timeout (avoid killing in-progress spawns)
```

### Suggested Default Values

Values below are production-reasonable starting points. All times in human-readable
format; internally converted to milliseconds.

**Heartbeat & Liveness:**

| Parameter | Env Var | Suggested | Notes |
|-----------|---------|-----------|-------|
| Heartbeat interval | REPL_HEARTBEAT_INTERVAL | 10s | Agent → control frequency |
| Degraded threshold | REPL_HEARTBEAT_DEGRADED_MS | 2m | Early warning for connection issues |
| Ack timeout | REPL_HEARTBEAT_ACK_TIMEOUT | 15m | Agent shuts down if no ack |
| Stale detection | REPL_STALE_DETECTION | 25m | Control marks allocation failed |

Trade-off: Lower values = faster failure detection, more network overhead. Longer
values (shown above) provide researcher-friendly tolerance for network hiccups and

### Provider-Specific Overrides

Providers inherit base timing values and may override specific parameters.

**OrbStack (local development):**
- spawn_timeout: 30s (instant provisioning)
- spawn_poll_interval: 1s
- terminate_timeout: 10s
- cache_instance_state_ttl: 10s
- reconciliation_interval: 10s

Fast iteration trumps network efficiency in local dev.

**Lambda Labs / RunPod:**
- spawn_timeout: 5m (faster than AWS typical)

**AWS:**
Uses base defaults. Consider longer spawn_timeout (15m) for capacity-constrained
regions or large instance types.

### Deployment Profiles

**Development (OrbStack):**
Aggressive timings for fast feedback. Lower hold durations to avoid cluttering
local resources. Example:

```bash
REPL_HEARTBEAT_INTERVAL=5s
REPL_STALE_DETECTION=15s
REPL_DEFAULT_HOLD=1m
REPL_RECONCILIATION_INTERVAL=10s
```

**Production (single-user):**
Suggested defaults work well. Consider:

```bash
REPL_ABSOLUTE_MAX_HOLD=12h  # Tighter cost control
```

**Production (multi-user/managed):**
More conservative timeouts, longer grace periods:

```bash
REPL_HEARTBEAT_ACK_TIMEOUT=45s
REPL_STALE_DETECTION=45s
REPL_ORPHAN_MIN_AGE=15m
REPL_MANIFEST_CLEANUP_INTERVAL=10m
```

**High-latency networks:**
Account for network delays (or use even longer values from suggested defaults):

```bash
REPL_HEARTBEAT_INTERVAL=20s
REPL_HEARTBEAT_ACK_TIMEOUT=20m
REPL_STALE_DETECTION=30m
REPL_CLI_API_TIMEOUT=60s
```

### Failure Modes

**Network partition (agent-control):**
Both sides detect failure independently. Agent via ack_timeout, control via
stale_detection. Both initiate cleanup. Termination is idempotent.

**Slow provider response:**
Spawn/snapshot timeouts prevent indefinite waiting. Compensation terminates
partial resources. Reconciliation catches anything missed.

**Presigned URL expiry during transfer:**
Large file transfers may exceed TTL. Client should request fresh URLs for
long transfers. Multipart upload mitigates this for large files.

**Race: hold expiry vs active SSH:**
Hold expiry is the contract. SSH activity extends hold only if heartbeat reports
activity before expiry check runs. Users can explicitly extend if needed.

### Observability

Track these metrics to detect timing-related issues:

- Heartbeat ack latency distribution (P50, P95, P99)
- Stale heartbeat detection rate
- Hold expiry count (normal vs forced)
- Orphan detection rate per reconciliation cycle
- Spawn timeout rate by provider
- Compensation execution time
- Retry rate by operation type

Alerts to consider:
- Heartbeat ack P99 approaching ack_timeout
- Orphan rate exceeding baseline
- Spawn timeout rate exceeding 1%
- Compensation failures
