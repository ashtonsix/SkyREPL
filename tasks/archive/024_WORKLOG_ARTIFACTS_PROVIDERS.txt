Design Notes

AGENT-03: SSE backpressure on slow consumers
  Spec: Ch11. Carries: EX3.
  The control plane sends commands to agents via SSE (GET /v1/agent/commands).
  If a slow consumer can't keep up, the write buffer grows unbounded.

  Implementation:
    - Track queue depth per SSE connection in sse-protocol.ts
    - Priority hierarchy: commands > status > heartbeat_ack
    - When queue exceeds HIGH_WATER_MARK (e.g., 64 messages):
      1. Drop heartbeat_ack messages first
      2. If still overloaded, drop older status messages
      3. Commands are NEVER dropped (they're idempotent + acked)
    - Emit warning log when dropping begins
    - Optionally: disconnect consumer after sustained overload (e.g., 5 min)
  Impl: impl/control/src/api/sse-protocol.ts (control side), impl/agent/sse.py (agent side).
  ~30-50 LOC, 3-5 tests.

AGENT-04: Degraded mode
  Spec: Ch11 §11.5. Carries: nothing (fresh task).
  Progressive degradation: healthy → degraded → panic.
  Currently agent either works or crashes — no middle ground.

  Implementation (agent-side, Python):
    State machine in agent heartbeat loop:
      - healthy: control plane reachable, heartbeat succeeding
      - degraded: control plane unreachable but run still active
        → buffer logs to disk (ring buffer, max 10MB)
        → continue executing current run
        → retry heartbeat with exponential backoff (5s → 10s → 30s → 60s cap)
        → if control plane returns: flush buffered logs, return to healthy
      - panic: degraded for > PANIC_TIMEOUT (default 5 min)
        → trigger panic diagnostics (#AGENT-01, already built)
        → self-terminate

  Control-plane awareness:
    The control plane already has heartbeat timeout enforcement (#LIFE-02).
    No control-plane changes needed — degraded mode is purely agent-side.

  Impl: impl/agent/heartbeat.py (agent side, already has heartbeat loop).
  ~80-120 LOC (Python), 5-8 tests.

Both Phase A tasks are independent and can be dispatched in parallel.

AGENT-07: Artifact collection
  Spec: Ch11 §11.11. Depends: #LIFE-01 (DONE).
  Pseudo: agent/artifacts.py.l1.txt + .l2.txt.

  Agent side (Python):
    - Run config includes artifact_patterns: list of glob patterns
      (e.g., ["output/**", "*.csv", "model.pt"])
    - After run completion (before finalize), agent:
      1. Scan workdir for matching files
      2. For each file: compute checksum, determine size
      3. Request upload URL from control plane
      4. Upload artifact bytes
      5. Report artifact manifest (list of uploaded artifact metadata)
    - Errors: best-effort. Log failures, don't fail the run for artifact issues.

  Control-plane side (TypeScript):
    - Wire POST /v1/agent/artifacts endpoint (currently stubbed in agent.ts)
      → receives artifact metadata from agent
      → stores artifact in blob storage (reuses existing storage.ts)
      → records in manifest_resources (resource_type='artifact')
    - Add artifact upload endpoint: POST /v1/artifacts/upload
      → accepts multipart file upload
      → stores in blob storage, returns artifact ID
    - OR simpler: reuse existing blob/check + blob upload flow
      → agent uses POST /v1/blobs/check then uploads missing blobs
      → then POST /v1/agent/artifacts to register blob → artifact mapping

  The simpler blob-reuse approach is preferred — less new surface area.

  ~200-300 LOC across agent + control plane, 15-20 tests.

SNAP-04: Artifact resource type + API + CLI
  Spec: Ch03. Depends: #AGENT-07.
  Pseudo: resource/artifact.ts.l1.txt.

  Scope (extended beyond original epic task to close the E2E loop):
    DB:
      - artifacts table: id, run_id, manifest_id, name, path, checksum,
        size_bytes, blob_id, content_type, created_at
      - DDL migration (append to 001_schema.sql or new migration)

    Resource module: resource/artifact.ts
      - createArtifact(runId, metadata) → id
      - getArtifact(id) → Artifact | null
      - listArtifactsByRun(runId) → Artifact[]
      - deleteArtifact(id) → void (for cleanup-manifest use)

    API endpoints:
      - GET /v1/runs/:id/artifacts → list artifacts for a run
      - GET /v1/artifacts/:id/download → download artifact content (via blob)

    CLI commands (minimal, ~50 LOC in commands/):
      - `repl artifact list <run-id>` → table of artifacts
      - `repl artifact download <artifact-id> [--output=dir]` → download to local

  The CLI wiring is small enough to fold into this task rather than creating
  a separate CLI-08 subtask. Full CLI-08 (snapshot + orphan subcommands)
  remains a future task.

  ~150-200 LOC, 8-12 tests.

PROV-01: Validate provider abstraction (dry-run AWS)
  Spec: Ch08 §8.1. Depends: #LIFE-07 (DONE in G1b).
  Pseudo: provider/type-boundaries.ts.l1.txt (STILL USEFUL).

  Goal: Implement an AWS EC2 provider skeleton that compiles against the
  ComputeProvider interface and makes real(ish) API calls. Identify and fix
  any gaps in the abstraction before building full providers in G3.

  Implementation:
    1. NEW: provider/compute/aws.ts — AWS EC2 provider implementation
       - spawn(): EC2 RunInstances call with security group, user-data bootstrap
       - terminate(): EC2 TerminateInstances (idempotent: ignore already-terminated)
       - list(): EC2 DescribeInstances with tag filter (repl-managed=true)
       - get(): EC2 DescribeInstances by instance-id
       - generateBootstrap(): cloud-init user-data script
       - createSnapshot(): EC2 CreateImage (AMI)
       - getSnapshotStatus(): EC2 DescribeImages
       - deleteSnapshot(): EC2 DeregisterImage + delete backing snapshot
       Capabilities: { snapshots: true, spot: true, gpu: true, multiRegion: true,
                       persistentVolumes: true, idempotentSpawn: true }

    2. AWS SDK dependency: @aws-sdk/client-ec2 (tree-shakeable, ESM)
       Only this one package needed for dry-run validation.

    3. Validation approach:
       - Unit tests with mocked AWS calls (no real AWS account needed)
       - Type-level: does the Provider<AWSInstance, AWSSnapshot> satisfy
         ComputeProvider? If not, fix the interface.
       - Behavioral: do error codes map cleanly to provider error taxonomy?
       - Spot: does requestSpotInstance/cancelSpotRequest interface work?
       - Bootstrap: does generateBootstrap produce valid cloud-init?

    4. Interface gaps to probe (from spec §8.1):
       - Security group management: does spawn() config handle this?
       - IAM role attachment: is this a spawn option or separate step?
       - Multi-region: does the provider interface support region selection?
       - VPC/subnet selection: where does this live in the interface?
       - Key pair management: how does SSH key injection work?

    5. Output: documented list of interface changes needed (if any),
       possibly a provider implementation guide in docs/.

  Pseudo: provider/compute/aws.ts.l1.txt + .l2.txt, type-boundaries.ts.l1.txt.
  ~300-400 LOC, 10-15 tests.

  Note: This is NOT a production-ready AWS provider. PROV-02 builds the full
  thing. This is interface validation with enough implementation to be confident
  the abstraction works.
