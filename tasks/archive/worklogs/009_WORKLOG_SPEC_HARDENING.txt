PHASE C: SPEC TRANSFORMATION
================================================================================

Principle: SUBSTANCE PRESERVATION. Pseudo-code duplicated verbatim in impl/ L2
files serves no unique purpose in the spec. Replace with normative requirements
+ cross-ref. That's not deletion — it's the spec evolving toward specification.

Treatment by content category:

  REPLACE with normative requirements + cross-ref:
    Function implementations already duplicated in impl/ L2 files.
    The bulk transformation target. Each function block becomes a behavioral
    contract: what the system MUST/SHOULD do, stated independently of the
    code. The cross-ref (already placed by Phase B) points readers to the
    reference implementation.

  CONDENSE to key fields + constraints + cross-ref:
    Type/interface definitions already duplicated in impl/ L2 files.
    Keep the interface name, key fields, and any field-level MUST/SHOULD
    constraints. Remove the full verbatim TypeScript interface body.

  KEEP AS-IS (already specification-grade):
    SQL DDL (data model), SQL example queries (illustrative),
    JSON request/response (API contract), CLI command definitions (UX spec),
    ASCII diagrams + blueprints (architectural), configuration tables,
    normative prose already using MUST/SHOULD.

  REWRITE TO SPEC PSEUDOCODE:
    Pseudo-code that serves a specification purpose beyond implementation
    reference (defines an algorithm, illustrates a constraint, documents a
    protocol). Rewrite from TypeScript/Python into expressive, language-
    agnostic pseudocode suitable for a specification. Preserve substance;
    change form.
