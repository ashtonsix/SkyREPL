USER DIRECTIVES
================================================================================

Original T5 requirement (from TODO.txt):

  Heartbeat Timeout: increasing to 15 minutes significantly lowers the "false
  positive" termination rate, with the accepted compromise of an increased
  "cost leak" window if an instance genuinely hangs. Given SkyREPL's core
  insight is lifecycle chaos, 15 minutes is a safe "researcher-friendly"
  default. Agent should commit to self-termination at minute 15, and actually
  terminate at minute ~16-20, using the same checkpoint path as spot
  interruption to handle (harden for resilience if control plane unavailable:
  second presigned S3 URL the agent can try if first fails, when control plane
  comes back online manifest created before workflow started will show where to
  find artifact [just update to include second failover location],
  reconcillation should be able to infer instance terminated and that a
  checkpoint was uploaded immediately before. this means users who configure
  checkpoints properly and launch multi-hour/day jobs that interrupt due to
  agent panic don't lose work done up until point of panic). Timing constraints
  need to be updated: normatively, control plane shouldn't terminate instance
  that fails heartbeat until unreachable for "agent commits to self termination
  after X duration without heartbeat ack" + "expected time for agent to
  checkpoint" + margin; advisory: 15 minutes self-termination, 25 minutes
  control plane forces termination.

Additional requirement (2026-02-04):

  Early Warning System: DEGRADED_CONTROL status update at 2 minutes to provide
  UX visibility before the 15-minute cut.

================================================================================
TIMING MODEL
================================================================================

Timeline (agent perspective):
```
t=0        Last successful heartbeat ack
t=2m       DEGRADED_CONTROL status (early warning, UX visibility)
t=15m      Agent commits to self-termination, starts checkpoint
t=16-20m   Agent checkpoint complete, self-terminates
t=25m      Control plane forces termination (if agent failed to self-term)
```

Timing constants (timing.ts.l2.txt updates):
```typescript
HEARTBEAT_INTERVAL_MS: 10_000          // 10s (unchanged)
HEARTBEAT_DEGRADED_MS: 120_000         // 2m  (NEW - early warning threshold)
HEARTBEAT_ACK_TIMEOUT_MS: 900_000      // 15m (was 30s - agent self-term trigger)
STALE_DETECTION_MS: 1_500_000          // 25m (was 30s - control plane force-term)
```

Constraint (normative):
  HEARTBEAT_INTERVAL_MS < HEARTBEAT_DEGRADED_MS < HEARTBEAT_ACK_TIMEOUT_MS
    < STALE_DETECTION_MS

Rationale for 25m control plane timeout:
  15m (agent self-term) + 5m (checkpoint budget) + 5m (margin) = 25m

EARLY WARNING DISCOVERY (Explore Results)
================================================================================

Five parallel explore agents examined the codebase for early warning patterns.

### Key Finding: No Intermediate Warning States Exist

Current system is BINARY:
- Healthy: last_heartbeat < 30s ago
- Failed: last_heartbeat >= 30s → immediate `suspect:heartbeat_timeout`

No gradual degradation. No user-visible warning before failure.

### Existing Patterns to Leverage

1. **workflow_state format**: `<workflow>:<phase>` (e.g., `spawn:pending`)
   - DEGRADED_CONTROL fits as: `degraded:control_plane_unreachable`
   - Agent already reports workflow_state in HeartbeatMessage

2. **CLI preflight warnings** (impl/cli/src/preflight.ts.l1.txt):
   - Existing tiers: info / warning / critical
   - Types: orphans, budget, credentials, stale_allocations, provider_issues
   - DEGRADED instances could appear here

3. **HeartbeatMessage fields** (impl/shared/src/api/agent.ts.l1.txt:70-77):
   - workflow_state: string (can carry degraded status)
   - dropped_logs_count: number (existing degradation signal)
   - Could add: `control_plane_status?: 'connected' | 'degraded' | 'unreachable'`

4. **Status icons** (impl/cli/src/ui/formatters.ts.l1.txt):
   - ⏳ pending, ⏵ running, ✓ done, ✗ failed
   - Could add: ⚠️ degraded

### Agent-Side Detection

Current (impl/agent/heartbeat.py.l1.txt):
- Tracks time since last ack
- At 30s: initiate shutdown
- No intermediate reporting

Proposed:
```python
time_since_ack = now - last_ack_time
if time_since_ack > HEARTBEAT_DEGRADED_MS:  # 2m
    workflow_state = 'degraded:control_plane_unreachable'
    # Continue heartbeating, but report degraded status
elif time_since_ack > HEARTBEAT_ACK_TIMEOUT_MS:  # 15m
    initiate_panic_checkpoint()
```

### Control Plane Visibility

Option A: Agent reports degraded status in HeartbeatMessage
  - Control plane stores in instance.workflow_state
  - CLI queries and displays
  - Pro: Simple, uses existing infrastructure
  - Con: Only visible when heartbeat succeeds (ironic)

Option B: Control plane derives degraded from last_heartbeat
  - No agent change needed
  - Reconciliation marks degraded at 2m before suspect at 25m
  - CLI queries last_heartbeat and derives status
  - Pro: Works even if heartbeats failing
  - Con: Requires reconciliation change + CLI derivation logic

Option C: Hybrid (RECOMMENDED)
  - Agent reports degraded status when it detects 2m gap
  - Control plane ALSO derives degraded from last_heartbeat for display
  - Double confirmation: agent knows it's degraded, control plane can verify

### CLI Integration Points

1. `repl status` / `repl list` - Show degraded instances with ⚠️ icon
2. `repl run` preflight - Warn if target instance is degraded
3. Real-time progress - Show degraded status during workflow execution
4. SSH connect - Notice if instance control plane connection is degraded

### 2. Checkpoint Failover Design

Current state:
  - Single presigned URL per blob (4h TTL from timing.ts)
  - Generated at workflow start via StartRunMessage

Required:
  - Dual URLs (primary + failover) for checkpoint blobs
  - Agent tries primary, falls back to failover if primary fails
  - Both URLs point to same logical blob (same S3 key, different presign)

Options explored:
  A) Include failover URLs in StartRunMessage ← RECOMMENDED
     - Cleanest: URLs generated at workflow start alongside primary
     - No schema changes needed
     - 4h TTL sufficient for 25m window
  B) Store in manifest metadata
     - More complex, requires manifest update
  C) Pre-generate at instance launch
     - Wasteful if checkpoint never needed

Decision: Option A (StartRunMessage)

### 4. Reconciliation Changes

Current (30s):
  - Instance stale after 30s without heartbeat
  - Immediate FAILED transition

