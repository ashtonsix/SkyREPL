Step 1:  Compensation scope fix (SD-G1-01) CRITICAL                [Md] DONE

--------------------------------------------------------------------------------
Design Notes
--------------------------------------------------------------------------------

Step 1: Fix compensation to single-node scope (SD-G1-01) — CRITICAL

  The spec is unambiguous across THREE chapters:
    Ch06 §6.5: "Scope: Single failed node only (no cascade through completed nodes)"
    Ch07: "Workflows do NOT cascade compensation through completed nodes"
    Ch09: "Previously completed nodes are NOT compensated"

  Current impl (compensation.ts:216-256, compensateWorkflow) iterates ALL
  completed nodes in reverse execution order and compensates each one. This is
  a full saga-style rollback that violates the core architecture.

  The architectural intent:
    - COMPENSATION = single-node error recovery (immediate, during workflow)
    - TEARDOWN = manifest-level resource destruction (delayed, via cleanup workflow)
    - Complex teardown belongs at manifest/policy level, NOT in workflow execution
    - Manifests track all resources, enabling proper cleanup regardless of failure point

  Why saga-style is WRONG:
    1. Conflates error recovery with resource teardown
    2. Could interfere with resource claiming (another workflow may have claimed)
    3. Compensates nodes that succeeded — nothing to recover
    4. Bypasses the manifest cleanup system that was designed for this purpose
    5. Compensation is for workflows with "a good chance of expeditious recovery"
       — completed nodes don't need recovery

  Fix:
    1. Delete compensateWorkflow() function entirely (compensation.ts:216-256)
    2. In handleWorkflowFailure (engine.ts:931-968):
       - Remove call to compensateWorkflow(workflowId)
       - Find the single failed node from the nodes array
       - Call compensateFailedNode(workflowId, failedNode.node_id) for ONLY that node
       - Seal manifest (this already happens — manifest cleanup handles the rest)
    3. In handleWorkflowTimeout (engine.ts):
       - Compensate ONLY currently-running nodes (not completed ones)
    4. In handleCancellation (engine.ts):
       - Compensate ONLY currently-running node if applicable
       - Do NOT compensate completed nodes
    5. Remove the 'rolling_back' workflow state if it only exists for saga rollback
    6. Update compensation.ts tests to verify single-node scope
    7. Verify existing tests still pass — the behavioral change is that completed
       nodes are NO LONGER undone on failure

  Files: compensation.ts, engine.ts, state-transitions.ts (if rolling_back state).
  Tests: update compensation tests, add test verifying completed nodes are NOT
         compensated on workflow failure.

Step 2: Engine bug fixes (SM-01, SM-03, SM-05)

  SM-01: Fallback retry path broken [engine.ts:832-833]
    handleRetry fallback branch calls resetNodeForRetry on a running node.
    resetNodeForRetry requires status='failed'. Transition silently returns
    WRONG_STATE and node stays stuck in 'running' forever.
    Fix: call failNode() before resetNodeForRetry, matching the same_params path.

  SM-03: Attempt init inconsistency [engine.ts:235 vs patterns.ts:106+]
    Blueprint nodes start at attempt=1, pattern-inserted nodes start at attempt=0.
    After startNode (which increments), blueprint nodes show attempt=2 on first
    execution, pattern nodes show attempt=1. Affects retry strategy comparison.
    Fix: normalize all node creation to attempt=0 (engine.ts submit + patterns.ts
    already at 0). startNode increments to 1 on first execution.

  SM-05: Stale node.attempt in determineRetryStrategy [engine.ts:459]
    The node object passed to executeNode is fetched before startNode increments
    the attempt counter. determineRetryStrategy receives pre-increment value.
    Fix: re-read node from DB after startNode, before determineRetryStrategy.

  Files: engine.ts, patterns.ts (already at 0, just verify).
  Tests: existing retry tests validate behavior. Add 1-2 tests for fallback path.

Step 3: Manifest safety (SD-G1-02, SD-G1-03, SD-G1-04, SD-G1-09)

  SD-G1-02: isManifestExpired omits owner_type='manifest' count check
    Spec §4.1 requires: "COUNT(manifest_resources WHERE owner_type='manifest') = 0"
    as condition 4 for expiry. Currently checks only conditions 1-3.
    Fix: add subquery to isManifestExpired in resource/manifest.ts.

  SD-G1-03: listExpiredManifests omits resource ownership check
    Same issue at the scanner level. Expired manifest list may include manifests
    with unclaimed resources, triggering premature cleanup.
    Fix: add NOT EXISTS subquery to listExpiredManifests in material/db/manifests.ts.

  SD-G1-04: sealManifestSafe uses raw retention instead of workflow-type policy
    Engine's sealManifestSafe reads manifest.retention_ms (usually NULL) instead
    of calling sealManifestWithPolicy which applies workflow-type-specific retention.
    Fix: replace sealManifestSafe body with call to sealManifestWithPolicy,
    passing workflow type from the workflow record.

  SD-G1-09: cleanup-manifest entry doesn't validate SEALED state
    cleanupManifest() validates existence but not status. DRAFT manifests can
    have cleanup workflows submitted (they fail at node execution).
    Fix: add SEALED check in cleanupManifest() entry point, return 400 for DRAFT.

  Files: resource/manifest.ts, material/db/manifests.ts, engine.ts,
         intent/cleanup-manifest.ts.
  Tests: add 3-4 tests (expiry with unclaimed resources, policy-based retention,
         DRAFT rejection).

Step 9: Manifest cleanup deadlock (HIGH)

  The SD-G1-02/03 fix (Step 3) added an owner_type='manifest' check to
  listExpiredManifests and isManifestExpired, per spec §4.1. But resources are
  inserted with owner_type='manifest' and NOTHING ever changes this after
  sealing. Result: manifests with resources never appear in the expired list.
  The entire cleanup pipeline is effectively disabled.

  Root cause: sealManifest (state-transitions.ts:358-399) clears
  current_manifest_id on resource tables but does NOT update owner_type in
  manifest_resources. After sealing, manifest_resources rows still show
  owner_type='manifest'.

  Fix: In sealManifest, after detaching resources (line 394), add:
    UPDATE manifest_resources SET owner_type = 'released'
    WHERE manifest_id = ? AND owner_type = 'manifest'
  This marks resources as released, allowing isManifestExpired and
  listExpiredManifests to correctly identify expired manifests.

  Files: state-transitions.ts.
  Tests: update resource-manifest tests, add test verifying sealed manifest
         with released resources appears in expired list.

Step 10: Finalize retention policy fix (MEDIUM)

  finalize.ts:91-118 seals the manifest BEFORE handleWorkflowComplete runs.
  It uses: manifest.retention_ms ?? 30 * 24 * 60 * 60 * 1000 (30 days default).
  But sealManifestWithPolicy for launch-run is 30 * 60_000 (30 minutes).
  Since sealManifestSafe is idempotent (returns early if already SEALED),
  finalize's 30-day retention wins. Effective retention is 1440x too long.

  Fix: Remove the manifest sealing block from finalize.ts (lines 91-118).
  Let handleWorkflowComplete call sealManifestSafe with the correct policy.
  The finalize node's job is allocation completion + replenishment, not sealing.

  Files: finalize.ts.
  Tests: existing finalize tests should still pass. Add test verifying
         launch-run manifest gets 30-minute retention (not 30-day).

Step 11: Crash-path manifest seal (MEDIUM)

  Two catch blocks in engine.ts call failWorkflowTransition but NOT
  sealManifestSafe:
    1. _executeLoopInner catch (lines 368-386)
    2. submit .catch on executeLoop (lines 248-259)

  If the execute loop crashes, the manifest stays DRAFT forever.
  recoverWorkflows handles stale-pending but not already-failed+DRAFT.

  Fix: Add sealManifestSafe(manifestId) calls to both catch blocks,
  matching the pattern in handleWorkflowFailure/handleCancellation/etc.
  Need to look up workflow.manifest_id from the workflowId.

  Files: engine.ts.
  Tests: add test that crashes execute loop and verifies manifest is sealed.

Step 12: cleanupInstance provider termination (MEDIUM)

  cleanup-resources.ts:187-211 cleanupInstance only marks DB state
  (workflow_state='terminate:complete'). It does NOT call the provider API
  to actually terminate the VM. Comment on line 204: "For now, just mark as
  terminated. In the future this would spawn a terminate-instance subworkflow."

  Fix: After marking DB state, call provider.terminate(provider_id) to
  actually destroy the VM. Look up instance record for provider + provider_id.
  Wrap in try/catch — best-effort, log warning on failure.

  Files: cleanup-resources.ts.
  Tests: add test verifying provider.terminate is called during instance cleanup.

Step 13: Warm-path instance manifest tracking (MEDIUM)

  On the warm path (resolve-instance finds a warm allocation),
  claim-allocation.ts emits the allocation to the manifest (line 56) but
  NOT the instance. The instance was emitted to the previous workflow's
  manifest by spawn-instance, but nothing transfers or re-emits it to the
  new manifest.

  If the workflow fails before finalize, the instance isn't tracked in the
  current manifest. Partially mitigated by finalize handling termination
  directly, but failure paths leave instances untracked.

  Fix: In claim-allocation.ts, after claiming the allocation, also emit
  the instance: ctx.emitResource("instance", result.data.instance_id, 50).
  This ensures the instance is tracked in the new manifest regardless of
  whether the workflow succeeds or fails.

  Files: claim-allocation.ts.
  Tests: add test verifying warm-path manifest contains both allocation
         and instance resources.
