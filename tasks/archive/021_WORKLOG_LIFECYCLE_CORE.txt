Design Notes
--------------------------------------------------------------------------------

LIFE-01: Resource layer stubs
  The resource layer sits between API/intent code and the db/ material layer.
  Currently impl has db/ queries but no resource-level operations with:
    - State machine enforcement (atomic transitions)
    - Business rule validation
    - Event emission for side effects

  Files to create/fill:
    - control/src/resource/allocation.ts  (claim, release, transition, atomic ops)
    - control/src/resource/instance.ts    (register, deregister, state mgmt)
    - control/src/resource/run.ts         (create, bind, complete, fail)

  Reference: impl-pseudo allocation.ts.l1, instance.ts.l1, run.ts.l1
  Spec: Ch03 §3.1-3.3, Ch05 §5.1-5.4

  #BL-11 folds in: updateAllocationStatus() with atomic transitions, claimAllocation() two-phase.

  Estimated: 400-600 LOC across 3 files, 30+ tests.

WF-01: Compensation framework
  Compensation = undo-on-failure for workflow steps. Currently the engine runs
  forward-only. Need:
    - Compensatable step type (action + compensate pair)
    - Compensation stack tracking per workflow
    - On failure: unwind stack in reverse order
    - Partial compensation (some steps may not be compensatable)

  Reference: impl-pseudo compensation.ts.l1 (L1) and compensation.ts.l2 (L2)
  Spec: Ch06 §6.5, §6.8

  Key design question: compensation as engine-level concern (step metadata)
  vs. separate compensation workflow. L2 pseudo suggests engine-level.

  Estimated: 300-500 LOC, 20+ tests.

--------------------------------------------------------------------------------
Design Notes
--------------------------------------------------------------------------------

All four tasks follow the same pattern: wire a scanner/enforcer into the
background loop. Each polls a table for stale/expired rows and takes action.

LIFE-02: Heartbeat timeout
  - Poll: allocations WHERE status='ACTIVE' AND last_heartbeat < threshold
  - Action: transition to FAILED, trigger compensation on parent workflow
  - Threshold: configurable, default 60s (spec §5.3)
  - Pseudo: main.ts.l1 (background loop section)

LIFE-03: Hold expiry
  - Poll: allocations WHERE status='AVAILABLE' AND hold_expires_at < now
  - Action: release hold (set hold columns to NULL or transition)
  - Pseudo: hold-expiry.ts.l1

LIFE-04: Manifest expiry
  - Poll: manifests WHERE expires_at < now AND status='SEALED'
  - Action: queue cleanup-manifest intent (or mark for cleanup)
  - Note: no cleanup intent yet — just queue/mark for #LIFE-09

LIFE-05: Blob GC
  - Poll: orphaned blobs (ref count = 0, grace period elapsed)
  - Action: delete from object store
  - Add cleanup_processed_at column (#BL-13)
  - Route through resource layer (#BL-08)
  - Pseudo: garbage-collection.ts.l1

Each: 80-150 LOC, 5-10 tests.
