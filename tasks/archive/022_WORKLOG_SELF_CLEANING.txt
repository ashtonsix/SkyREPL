Design Notes
--------------------------------------------------------------------------------

WF-06: runSingleStep helper
  Convenience wrapper: create 1-node workflow, execute, return result directly.
  Hides DAG ceremony for simple atomic operations. Useful for terminate-instance
  nodes that do one thing (e.g., "call provider.terminate()").
  Spec: Ch06 §6.1. ~30-50 LOC, 3-5 tests.

WF-05: pause() / resume() / retry()
  Currently throw "not implemented" in engine.ts. Wire to state machine:
    - pause(workflowId): running → paused (freeze executeLoop)
    - resume(workflowId): paused → running (re-enter executeLoop)
    - retry(workflowId): per spec §6.2.1, creates NEW workflow with same inputs
      (not a state machine transition — preserves audit trail)
  State machine additions: 'paused' state, running↔paused transitions.
  ~80-120 LOC, 5-8 tests.

WF-02: IAR (Idempotent-Atomic-Retriable) pattern
  Spec: Ch06 §6.3.1. Transform: A → B → C becomes A → A' → B → C with
  A' as inserted parallel node. Constraints: max 1 insertion depth, target
  node must be pending.
  Implementation in patterns.ts — currently stub throws "not implemented".
  Need: insertAndReconverge() that splices a new node into the DAG before
  a target, wiring edges correctly. Engine already does topological sort,
  so DAG mutation is the core work.
  Pseudo: workflow/patterns.ts.l1+l2 (§InsertAndReconverge section).
  ~120-200 LOC, 8-12 tests.

WF-03: Subworkflow contract
  Spec: Ch06 §6.4. Parent spawns child workflow, waits for completion.
  Child failure propagates to parent. MAX_SUBWORKFLOW_DEPTH = 3.
  Key semantics from spec:
    - Parent sees: status, output, resource summary (black box)
    - Manifests independent: child seals own manifest on completion
    - Parent claims resources by querying child's SEALED manifest
    - finalize node blocks until all children complete
  Engine currently has depth check but executeSubworkflow throws.
  Need: spawn child via engine, poll/await completion, propagate failure,
  cancel children on parent cancel.
  Pseudo: engine.ts.l2 (§Subworkflows section).
  ~200-300 LOC, 10-15 tests.

WF-04: PFO (Parallel Fan-Out) pattern
  Spec: Ch06 §6.3.3. Spawn N independent branches, join at convergence node.
  MAX_PARALLEL_BRANCHES = 16. Two join modes: all-succeed, first-failure-cancels.
  Transform: A → B becomes A → A1/A2/A3 → B (fan-out from A, join at B).
  Implementation in patterns.ts — needs parallelFanOut() that:
    1. Creates N branch nodes
    2. Wires each from source to join node
    3. Engine executes branches concurrently (Promise.all or similar)
    4. Handles join semantics (all-succeed vs fail-fast)
  Depends on WF-02 (IAR) because PFO branches should be retriable.
  Pseudo: workflow/patterns.ts.l1+l2 (§ParallelFanOut section).
  ~150-250 LOC, 10-15 tests.

Steps 3 and 4 (WF-02 and WF-03) can run in parallel — no dependency between them.
Step 5 (WF-04) blocks on step 3 (WF-02).

LIFE-12: Resource claiming protocol
  Spec: Ch04 §4.2. Manifest-resource ownership with claim/release semantics.
  Currently ctx.claimResource() returns false always.
  Core functions needed:
    - sealManifest(id): DRAFT → SEALED transition with retention policy

    - claimResourceAtomic(newManifestId, resourceType, resourceId): atomic transfer
    - claimWarmInstance(ctx, spec): query SEALED manifests, claim instance
    - parentClaimSubworkflowResource(parentCtx, subworkflowId, resourceType)

    - triggerCleanup(id): spawn cleanup-manifest workflow

  Pseudo: resource/manifest.ts.l1 (comprehensive L1+L2).
  ~300-450 LOC, 15-20 tests.

LIFE-11: Allocation reconciliation
  Spec: Ch05 §5.5. Background runner with 6 reconciliation task types:
    1. CLAIMED timeout (stuck claims > CLAIMED_TIMEOUT → FAILED)
    2. Warm pool health (stale AVAILABLE allocations)
    3. Orphaned claims (CLAIMED with no active workflow)
    4. Stalled transitions (stuck intermediate states)
    5. Provider state sync (verify instance alive with provider)
    6. Allocation aging (AVAILABLE too long → reclaim)
  Some overlap with existing heartbeat/hold checks — reconciliation is the
  catch-all safety net that runs at a slower cadence (60s per pseudo).
  Pseudo: background/reconciliation.ts.l1 (comprehensive L1+L2).
  ~200-300 LOC, 12-18 tests.

LIFE-07: terminate-instance intent
  Spec: Ch07 §7.2. 6-node DAG (spec blueprint):
    validate-instance → drain-allocations → drain-ssh-sessions →
    cleanup-features → terminate-provider → cleanup-records

  Node executors (new files in workflow/nodes/):
    1. validate-instance.ts: Check instance exists, set workflow_state
       to 'terminate:draining', acquire termination lock (idempotent)
    2. drain-allocations.ts: Query ACTIVE allocations for instance,
       transition each to COMPLETE (or FAILED if run in progress),
       notify agent via SSE CancelRunMessage
    3. drain-ssh-sessions.ts: Query debug_hold allocations, broadcast
       termination warning, wait drain_timeout_ms, force disconnect.
       SSH not yet implemented — stub with TODO for now.
    4. cleanup-features.ts: PFO over FeatureInstallations for instance.
       No features implemented yet — execute as no-op PFO (0 branches).
    5. terminate-provider.ts: Call provider.terminate(instanceId).
       Handle NOT_FOUND gracefully (already terminated).
    6. cleanup-records.ts: Set workflow_state='terminate:complete',
       close usage records, update instance status.

  New file: intent/terminate-instance.ts (blueprint definition + error handling map).
  New file: intent/terminate-instance.schema.ts (TypeBox schemas).
  Compensation: each reversible node gets compensate handler.
    - drain-allocations: restore allocation status
    - terminate-provider: NOT reversible (no compensation)
    - cleanup-records: NOT reversible
  Pseudo: intent/terminate.ts.l1+l2, terminate.types.ts.l1.
  ~400-600 LOC across 8+ files, 25-35 tests.

Design Notes
--------------------------------------------------------------------------------

LIFE-09: cleanup-manifest intent
  Spec: Ch07 §7.4, Ch04 §4.3. Priority-ordered resource cleanup.

  Blueprint (from spec):
    load-manifest-resources → sort-by-priority → cleanup-resources-loop
    (PFO by type) → delete-manifest

  Cleanup priority (highest first):
    90: allocations (release instance slots)
    80: runs (delete files, logs, records)
    70: artifacts (delete S3 objects)
    65: features (Tailscale, other feature provider resources)
    60: other objects (generic objects)
    50: instances (terminate via provider — reuses terminate-instance)
    10: snapshots (last — may be reused)

  Per-provider scoping: query DISTINCT provider, spawn per-provider subworkflow
  (uses #WF-03 subworkflow contract). Each subworkflow cleans only its provider's
  resources.

  Resource-type-specific cleanup handlers:
    - cleanup-allocations.ts: delete DB records
    - cleanup-runs.ts: delete files/logs, then DB records
    - cleanup-objects.ts: delete from S3 (if applicable), then DB records
    - cleanup-instances.ts: spawn terminate-instance subworkflow (reuses #LIFE-07)
    - cleanup-snapshots.ts: provider snapshot deletion, then DB records

  New file: intent/cleanup-manifest.ts (blueprint + orchestration).
  New file: intent/cleanup-manifest.schema.ts (TypeBox schemas).
  New files: workflow/nodes/cleanup-*.ts (5-6 resource-type handlers).
  Pseudo: intent/cleanup-manifest.ts.l1+l2, cleanup.types.ts.l1.
  Depends: #LIFE-06 (DDL — done), #WF-03 (subworkflow), #WF-04 (PFO),
           #LIFE-12 (claiming protocol for cross-manifest resource transfer).
  ~400-600 LOC across 8+ files, 20-30 tests.

  Error handling (from spec):
    NOT_FOUND → skip_and_continue (already deleted)
    PROVIDER_API_ERROR → retry_then_skip (best effort, max 3)
    S3_ERROR → log_and_continue (don't block for S3 blip)
    DATABASE_ERROR → fail_workflow (critical)

LIFE-10: cleanup-manifest background trigger
  Wire manifestCleanupCheck() (already detection-only from #LIFE-04) to
  actually spawn cleanup-manifest intents for expired manifests.
  Guard: check no existing cleanup workflow running for manifest.
  Limit: max 10 cleanups per scan cycle to prevent overload.
  ~40-80 LOC, 5-8 tests.
