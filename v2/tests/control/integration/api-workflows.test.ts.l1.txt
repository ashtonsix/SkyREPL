# integration/api-workflows.test.ts - Layer 2 API integration tests

## SETUP
- Start control plane with workflow engine enabled
- Clean database before each test
- Configure test provider (OrbStack mock)
- HTTP client for API calls
- SSE client for event streaming

## Launch-Run Tests
- POST /api/v2/launch-run creates workflow
  - Request body: {command, spec, files, init_checksum}
  - Returns 202 Accepted
  - Response includes workflow_id
  - Response includes stream URL
  - Workflow created in database
- SSE delivers events
  - Connect to stream URL
  - Receive workflow.started event
  - Receive node.complete events
  - Receive logs from agent
  - Events in correct order
- Completes with output
  - Receive workflow.complete event
  - Output includes exit_code
  - Output includes artifacts
  - Allocation state updated (HELD or COMPLETE)
  - Connection closed after complete

## Terminate Tests
- POST /api/v2/terminate-instance
  - Request body: {instance_id, force: false}
  - Returns 202 with workflow_id
  - Creates termination workflow
  - Workflow includes drain step
  - Stream URL in response
- Instance terminated after completion
  - Workflow completes
  - Instance status = terminated
  - Provider API called (delete instance)
  - Allocations cleaned up
  - Tailscale deregistered
- Force flag skips drain
  - Request with force: true
  - Drain step skipped
  - Immediate termination
  - SSH sessions killed
  - Faster completion

## Snapshot Tests
- POST /api/v2/create-snapshot
  - Request body: {instance_id, name, tags}
  - Returns 202 with workflow_id
  - Creates snapshot workflow
  - Stream URL in response
- Requires capable provider
  - Providers with snapshot support only
  - 400 error if provider doesn't support
  - Error message explains requirement
- Returns snapshot ID
  - Workflow completes
  - Output includes snapshot_id
  - Snapshot recorded in database
  - Provider metadata stored

## Extend Hold Tests
- POST /api/v2/extend-hold
  - Request body: {allocation_id, duration_minutes}
  - Returns 202 with workflow_id
  - Updates hold_until timestamp
  - Returns updated allocation in output
- Respects max hold duration
  - Max hold (e.g., 24 hours) configured
  - Request beyond max → capped at max
  - Warning in response
  - Total hold duration tracked

## Cancel Tests
- POST /api/v2/cancel
  - Request body: {workflow_id}
  - Returns 202 Accepted
  - Workflow status → CANCELLED
  - Running nodes interrupted
  - PENDING nodes marked SKIPPED
- Single-node compensation runs
  - Failed/cancelled node compensated
  - Compensation completes before cancel finishes
  - Compensation errors logged but don't block
  - Resources cleaned up

## Status/Stream Tests
- GET /api/v2/status/:workflow_id
  - Returns workflow status
  - Includes progress (completed/total nodes)
  - Includes current node
  - Includes elapsed time
  - 404 if workflow not found
- GET /api/v2/stream/:workflow_id
  - Returns SSE stream
  - Delivers historical events if workflow started
  - Delivers live events as they occur
  - Closes on workflow complete/cancel
  - Reconnect delivers events since last position

## Preflight Tests
- GET /api/v2/preflight
  - Query params: spec_checksum, estimated_duration
  - Returns preflight response
  - No side effects (read-only)
- Returns warnings
  - Budget warnings if cost high
  - Quota warnings if near limit
  - Orphan warnings if detected
  - Warnings array in response
- Budget warnings calculated
  - Estimates cost based on spec
  - Compares to configured budget
  - Warning severity: info/warning/error
  - Includes cost breakdown
- Orphan detection works
  - Checks for untracked resources
  - Returns list of orphan candidates
  - Suggests cleanup actions
  - Does not auto-cleanup
