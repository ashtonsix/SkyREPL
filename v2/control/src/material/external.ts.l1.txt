# external.ts - Provider State Caching

## Purpose
- In-memory TTL cache for ephemeral provider state
- NOT persisted to database (memory-only, cleared on restart)

## Core Function
- `getWithCache<T>(key, ttlMs, fetcher)` - TTL-based cache wrapper

## Cache Keys
- Format: `{provider}:{type}:{region}:{spec}` (e.g., `aws:spot_prices:us-east-1:g5.xlarge`)
- Instance: `instance:{id}:external`

## TTL Defaults
- Spot prices: 5 minutes (300,000 ms)
- Instance external state: 30 seconds
- Snapshot external state: 5 minutes

## Cache Invalidation

Hooks for explicit cache invalidation on known state changes:

- `invalidateInstanceCache(instanceId)` - Clear cached instance state
  - Called on: spawn complete, terminate complete, allocation state change
  - Ensures critical operations see fresh state

- `invalidateProviderCache(provider, pattern)` - Clear provider-specific keys
  - Example: `invalidateProviderCache('aws', 'instance:*')`

- `invalidateCacheByKey(key)` - Clear specific cache entry
  - Direct cache key targeting

## Cache Bypass for Critical Operations

`CACHE_BYPASS` flag for operations requiring guaranteed fresh state:

- `getWithCache<T>(key, ttlMs, fetcher, options?: { bypassCache?: boolean })`
  - If `options.bypassCache=true`, always fetch fresh (skip cache read)
  - Used by: orphan detection, allocation state transitions, budget checks
  - Still updates cache after fetch for subsequent non-critical reads

## Cache Staleness Risks

TTL-based caching creates staleness windows where decisions may use stale data:

**Staleness Window:** Up to 30s for instance state (TTL default)

**Risk Scenarios:**
1. Instance terminates during TTL window → allocation may claim terminated instance
2. Spot interruption during TTL window → workflow may not detect state change
3. Manual provider changes during TTL window → orphan detection misses resources

**Mitigations:**
1. Critical operations use `bypassCache=true` (orphan detection, allocation claims)
2. State transition hooks call `invalidateInstanceCache()` on known changes
3. Workflow nodes verify state before destructive actions (terminate checks if alive first)
4. Grace periods (10min orphan detection) exceed max staleness window

**Trade-offs:**
- Cache: Reduced provider API calls, lower latency, rate limit protection
- Bypass: Guaranteed fresh state, higher API cost, potential rate limits

## Error Handling
- Cache read failure → fetch fresh
- Cache write failure → non-critical (fetch again next time)
- Cache cleared on restart (memory-only, not persisted)
