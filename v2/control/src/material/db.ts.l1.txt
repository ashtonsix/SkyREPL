### claimAllocation Concurrency

CONCURRENCY: Optimistic locking strategy for allocation claims.
Multiple workflows may attempt to claim the same allocation simultaneously.
This function ensures exactly one succeeds using a compare-and-swap approach.

**Implementation:**
1. SELECT allocation WHERE id = ? AND status = 'AVAILABLE' AND run_id IS NULL
   - Read current state including updated_at timestamp
   - If not found or status/run_id mismatch, return null immediately (claim failed)

2. UPDATE allocations
   SET run_id = ?, status = 'CLAIMED', updated_at = <current_timestamp>
   WHERE id = ? AND updated_at = <original_updated_at>
   - The WHERE clause includes the original updated_at value from step 1
   - This ensures the UPDATE only succeeds if no other transaction modified the row

3. Check UPDATE result (rows affected)
   - If 0 rows affected: Another transaction won the race, return null (claim failed)
   - If 1 row affected: Claim succeeded, return updated allocation

**Race conditions handled:**
- Two workflows claim same allocation: First UPDATE wins, second gets 0 rows affected
- Allocation deleted between SELECT and UPDATE: 0 rows affected, claim fails gracefully
- Allocation status changed (e.g., FAILED): 0 rows affected, claim fails

**Performance notes:**
- SQLite's default isolation (SERIALIZABLE) ensures atomicity
- Index on (status, run_id) enables fast AVAILABLE allocation queries
- updated_at index (idx_allocations_updated) enables efficient race detection

## B21: Cleanup Processed Tracking

Schema addition to manifest_resources:
```sql
ALTER TABLE manifest_resources ADD COLUMN cleanup_processed_at INTEGER;
CREATE INDEX idx_manifest_resources_cleanup ON manifest_resources(manifest_id, cleanup_processed_at);
```

This timestamp prevents double-processing of blob reference decrements during cleanup.

## Error Handling

All database operations can throw these errors:

| Error | Cause | Handling |
|-------|-------|----------|
| DATABASE_CONNECTION_ERROR | SQLite file locked, disk full, permissions | Fail immediately, log details |
| DATABASE_QUERY_ERROR | Invalid SQL, type mismatch | Fail immediately (bug) |
| CONSTRAINT_VIOLATION | Foreign key, unique constraint | Fail immediately with constraint name |
| MIGRATION_ERROR | Schema version mismatch, invalid migration | Fail on startup, require manual fix |
| TRANSACTION_ERROR | Deadlock, timeout (rare in SQLite) | Retry once, then fail |

### Connection Error Handling

```
function getDatabase(): Database
  if (!connection) throw DATABASE_CONNECTION_ERROR('Database not initialized')
  if (!connection.open) throw DATABASE_CONNECTION_ERROR('Database connection closed')
  return connection
```

### Constraint Violation Handling

```
function createAllocation(data): Allocation
  try:
    return execute(INSERT...)
  catch (e):
    if (e.code === 'SQLITE_CONSTRAINT_FOREIGNKEY'):
      throw CONSTRAINT_VIOLATION('Instance or Run not found')
    if (e.code === 'SQLITE_CONSTRAINT_UNIQUE'):
      throw CONSTRAINT_VIOLATION('Allocation already exists for this run')
    throw e
```

### Migration Error Handling

```
function runMigrations(): void
  const current = getMigrationVersion()
  const target = getLatestMigrationVersion()

  if (current > target):
    throw MIGRATION_ERROR(`Database version ${current} is newer than code version ${target}`)

  for each migration from current+1 to target:
    try:
      transaction(() => {
        executeMigrationSQL(migration)
        setMigrationVersion(migration.version)
      })
    catch (e):
      throw MIGRATION_ERROR(`Migration ${migration.version} failed: ${e.message}`)
```

### Transaction Retry

```
# Note: Uses calculateBackoff from ../../../../shared/src/config/timing.ts.l2.txt
# calculateBackoff provides exponential backoff with ±10% jitter to prevent thundering herd

function transaction<T>(fn: () => T, retries = 1): T
  for attempt in 0..retries:
    try:
      return db.transaction(fn)
    catch (e):
      if (e.code === 'SQLITE_BUSY' && attempt < retries):
        sleep(calculateBackoff(attempt))  // Jittered exponential backoff
        continue
      throw TRANSACTION_ERROR(e.message)
```
