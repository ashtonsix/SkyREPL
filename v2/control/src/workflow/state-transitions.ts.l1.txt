# state-transitions.ts - State machine ownership and transition rules

## PURPOSE

Defines ownership boundaries for state transitions across allocations, manifests, and workflows.
State transitions have explicit owners (workflow nodes, control plane background tasks, or agent).

## ALLOCATION STATE MACHINE

### States

- AVAILABLE - Instance advertises empty run slot (warm pool)
- CLAIMED - Run assigned, workdir being prepared
- ACTIVE - Run executing
- COMPLETE - Allocation finalized (terminal state)
- FAILED - Lifecycle failure (not run failure, terminal state)

### Transition Ownership

```
AVAILABLE â†’ CLAIMED
  Owner: Workflow (claim-allocation node)
  Trigger: Warm pool allocation claimed during launch-run workflow
  Validation: run_id NULL â†’ run_id SET, status AVAILABLE â†’ CLAIMED atomically

CLAIMED â†’ ACTIVE
  Owner: Control plane (on sync_complete log)
  Trigger: Agent reports sync_complete (files downloaded successfully)
  Mechanism: Agent emits sync_complete to log stream, control plane watches logs
  Validation: Allocation must be in CLAIMED state, run_id matches

CLAIMED â†’ FAILED
  Owner: Control plane (on sync timeout/error)
  Trigger: Agent reports sync failure OR timeout waiting for sync_complete
  Timeout: 10 minutes default (configurable via SYNC_TIMEOUT_MS)
  Side effects: Instance marked suspect, typically triggers termination

ACTIVE â†’ COMPLETE
  Owner: Workflow (finalize node)
  Trigger: Run completes (success or failure)
  Parameters: debug_hold_until timestamp (optional, for debugging)
  Note: Run exit_code != 0 is NOT a failure state - still transitions to COMPLETE
  Side effects:
    - Sets completed_at timestamp
    - Sets debug_hold_until if configured (preserves allocation for debugging)
    - Creates new AVAILABLE allocation on same instance if warm pool eligible
  Race condition: If allocation already FAILED when transition attempted, treat as success
    - Scenario: Run completes while heartbeat timeout triggers FAILED transition
    - Resolution: Log warning, return success (run completed despite allocation failure)
    - Rationale: Run execution completed successfully; allocation state is infrastructure concern

ACTIVE â†’ FAILED
  Owner: Control plane (heartbeat timeout) OR agent (spot interruption)
  Control plane triggers:
    - Heartbeat timeout (>30 seconds without heartbeat)
    - Instance termination detected in provider
  Agent triggers:
    - Spot interruption warning received
    - Fatal agent error (agent crash, OOM)
  Side effects: Instance marked suspect, typically triggers termination
  Race condition: May race with workflow's ACTIVEâ†’COMPLETE transition
    - Winner: Whichever UPDATE executes first (optimistic locking)
    - Loser: If workflow attempts COMPLETE but finds FAILED, treat as success
```

### Warm Pool Replenishment (CRITICAL)

When allocation transitions ACTIVE â†’ COMPLETE, warm pool must be replenished:

```
ACTIVE â†’ COMPLETE (with warm pool replenishment)
  1. Transition existing allocation: status='ACTIVE' â†’ status='COMPLETE'
  2. Set completed_at timestamp
  3. Set debug_hold_until if configured (optional, for debugging access)
  4. Check if instance should remain in warm pool:
     - workflow_state ends with ':complete' (not ':error')
     - No pending termination request
     - Instance not marked for cleanup
  5. If warm pool eligible:
     - Create NEW allocation record: { instance_id, status='AVAILABLE', run_id=NULL, workdir=nextWorkdir() }
     - Log: "Created warm pool allocation for instance ${instanceId}"
  6. If NOT warm pool eligible:
     - Do not create new allocation
     - Instance will be cleaned up by manifest expiry
```

NOTE: Allocations are not "recycled" - each run gets a fresh allocation record.
The warm pool consists of AVAILABLE allocations with run_id=NULL.

### Invalid Transitions

All other transitions are invalid and return error:

- AVAILABLE â†’ ACTIVE (must go through CLAIMED)
- CLAIMED â†’ COMPLETE (must go through ACTIVE)
- COMPLETE â†’ any state (terminal state for that allocation)
- FAILED â†’ any state (terminal state)

### Concurrent Modification Handling

All transitions use optimistic locking:

```typescript
UPDATE allocations
SET status = ?, updated_at = ?
WHERE id = ? AND status = ? AND updated_at = ?
```

If update affects 0 rows, transaction rolls back and caller retries or fails.

## MANIFEST STATE MACHINE

### States

- DRAFT - Manifest under construction, accepting resources
- SEALED - Manifest finalized, no more modifications
- EXPIRED (derived) - Computed from expiresAt timestamp, not stored

### Transition Ownership

```
DRAFT â†’ SEALED
  Owner: Workflow (seal-manifest node OR workflow completion)
  Trigger: Workflow calls ctx.sealManifest() or workflow completes
  Side effects: Sets releasedAt timestamp, computes expiresAt
  Validation: Manifest must be in DRAFT, no pending resource additions

SEALED â†’ EXPIRED (derived, not a real transition)
  Owner: N/A (computed property)
  Trigger: Current time > expiresAt
  Note: Expired manifests are eligible for cleanup workflows
  Query: WHERE status = 'SEALED' AND expires_at < ?
```

### Invalid Transitions

- SEALED â†’ DRAFT (cannot unseal)
- DRAFT â†’ EXPIRED (must seal first)
- Manifests never transition back to earlier states

### Resource Addition Rules

- DRAFT: Resources can be added via ctx.emitResource()
- SEALED: No resource additions allowed
- EXPIRED: No modifications allowed (cleanup workflows claim resources)

## WORKFLOW STATE MACHINE

### States

- pending - Workflow submitted, not yet started
- running - Workflow execution in progress
- completed - Workflow finished successfully
- failed - Workflow failed (node error, validation, timeout)
- cancelled - User-requested cancellation
- rolling_back - Compensation in progress (after failure)

### Transition Ownership

```
pending â†’ running
  Owner: Workflow engine (executeLoop)
  Trigger: Engine picks up workflow from queue
  Side effects: Sets startedAt timestamp, begins node execution

running â†’ completed
  Owner: Workflow engine (completeWorkflow)
  Trigger: All nodes completed successfully, no pending/failed nodes
  Side effects: Sets finishedAt, seals manifest, emits output

running â†’ failed
  Owner: Workflow engine (handleWorkflowFailure)
  Trigger: Node failure with no retry/fallback path
  Side effects: Sets error, compensates failed node, seals manifest

running â†’ cancelled
  Owner: API handler (engine.cancel)
  Trigger: User calls POST /workflows/:id/cancel
  Side effects: Marks running nodes as skipped, seals manifest

running â†’ rolling_back
  Owner: Workflow engine (on node failure)
  Trigger: Failed node has compensation handler
  Side effects: Executes compensation, then transitions to failed

rolling_back â†’ failed
  Owner: Workflow engine (after compensation)
  Trigger: Compensation completed (success or failure)
  Side effects: Final state, workflow cannot be retried
```

### Invalid Transitions

- completed â†’ any state (terminal)
- failed â†’ running (B1: INVALID - use engine.retry() to create NEW workflow, not state transition)
- cancelled â†’ running (cannot resume cancelled workflows)
- rolling_back â†’ completed (compensation always leads to failed)

**B1: Retry After Compensation Clarification**
- retry() does NOT transition failedâ†’running (this is invalid)
- retry() creates a NEW workflow with same inputs (new workflow ID)
- Original workflow remains in 'failed' state for audit trail
- New workflow starts fresh with status='pending'
- This maintains state machine integrity while allowing retries

### Pause/Resume (Future)

Not implemented in initial version, but reserved:

```
running â†’ paused (future)
  Owner: API handler (engine.pause)
  Side effects: Stops new node execution, preserves state

paused â†’ running (future)
  Owner: API handler (engine.resume)
  Side effects: Resumes node execution from paused state
```

## NODE STATE MACHINE

### States

- pending - Node waiting for dependencies
- running - Node executing
- completed - Node finished successfully
- failed - Node execution error
- skipped - Node bypassed (conditional branch not taken)

### Transition Ownership

```
pending â†’ running
  Owner: Workflow engine (executeNode)
  Trigger: All dependencies completed, node ready to execute
  Side effects: Sets startedAt, increments attempt counter

running â†’ completed
  Owner: Node executor (handler returns success)
  Trigger: Handler returns without error
  Side effects: Sets finishedAt, stores output, marks dependents ready

running â†’ failed
  Owner: Node executor (handler throws error)
  Trigger: Handler throws error, retry exhausted
  Side effects: Sets error, triggers retry logic or workflow failure

pending â†’ skipped
  Owner: Workflow engine (pattern application)
  Trigger: Conditional branch not taken, or workflow cancelled
  Side effects: Node never executes, dependents reevaluated

failed â†’ pending (retry)
  Owner: Workflow engine (retry logic)
  Trigger: Error is retryable, attempt < maxAttempts
  Side effects: Increments attempt, sets retry_reason
```

### Crash Recovery Transitions

When control plane restarts:

```
running â†’ pending
  Owner: Recovery process (recoverWorkflows)
  Trigger: Control plane restart, node executor is idempotent
  Condition: isIdempotent(nodeType) === true
  Side effects: Increments attempt, sets retry_reason = 'crash_recovery'

running â†’ failed
  Owner: Recovery process (recoverWorkflows)
  Trigger: Control plane restart, node executor is NOT idempotent
  Condition: isIdempotent(nodeType) === false
  Side effects: Marks node failed, workflow handles failure path
```

## IMPLEMENTATION NOTES

### Background Tasks

Control plane runs these background tasks:

- Debug hold expiry: Every 60s, clears debug_hold_until WHERE status='COMPLETE' AND debug_hold_until < NOW
- Heartbeat timeout scanner: Every 30s, finds instances WHERE last_heartbeat < NOW - 30s
- Manifest cleanup: Every 60s, finds manifests WHERE status='SEALED' AND expires_at < NOW
- Orphan scan: Every 3600s (1hr), scans provider resources for orphans (dry-run mode)

### Background Task Intervals (Standardized)

| Task | Interval | Rationale |
|------|----------|-----------|
| Hold expiry | 60s | Balances responsiveness with DB load |
| Heartbeat timeout | 30s | Must catch timeouts within 2x heartbeat interval |
| Manifest cleanup | 60s | Synced with orphan scan for consistency |
| Orphan scan | 3600s (1hr) | Balances thoroughness with provider API rate limits |

### Atomic Transitions

All state transitions use database transactions with optimistic locking:

```typescript
async function transitionAllocationState(
  allocationId: string,
  fromStatus: AllocationStatus,
  toStatus: AllocationStatus,
  updates: Partial<Allocation> = {}
): Promise<{ success: boolean; error?: string }> {
  return db.transaction(async () => {
    const allocation = await db.getAllocationForUpdate(allocationId);

    if (!allocation) {
      return { success: false, error: 'Allocation not found' };
    }

    if (allocation.status !== fromStatus) {
      return {
        success: false,
        error: `Invalid transition: expected ${fromStatus}, got ${allocation.status}`
      };
    }

    await db.updateAllocation(allocationId, {
      status: toStatus,
      updated_at: Date.now(),
      ...updates
    });

    return { success: true };
  });
}
```

### Error Handling

Transition validation failures return structured errors:

```typescript
{
  code: 'INVALID_STATE_TRANSITION',
  message: 'Cannot transition allocation from COMPLETE to ACTIVE',
  category: 'conflict',
  details: {
    allocation_id: 'abc123',
    current_state: 'COMPLETE',
    requested_state: 'ACTIVE',
    valid_transitions: []
  }
}
```

These map to HTTP 409 Conflict via errors.ts error mapping.
