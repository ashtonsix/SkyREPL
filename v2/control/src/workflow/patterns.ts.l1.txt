## Pattern Descriptions

### single-step (Ch6.1)
Single node workflow - execute one operation with no branching.

**Use case:** Simple operations like extend-debug-hold, cancel-run
**Example:** extend-debug-hold workflow has one node that updates allocation.debug_hold_until

### insert-and-reconverge
Insert node A' before B: A → B becomes A → A' → B (both must complete)

**Use case:** Add setup step before existing operation
**Example:** Insert tailscale-install before start-run

### conditional-branch
Choose between two paths based on condition or error

**Use case:** Warm pool vs spawn decision, spot vs on-demand fallback
**Example:** Check warm pool → branch to claim-allocation or spawn-instance

### parallel-fan-out
Execute multiple branches concurrently, join when all complete

**Use case:** Multi-region deployment, parallel artifact uploads
**Example:** Deploy to us-east-1 and us-west-2 simultaneously

**A4: PARALLEL_FAILURE_POLICY**
- If ANY branch fails, ALL branches are affected
- Completed branches ARE compensated when a sibling branch fails
- Workflow fails if any parallel branch fails
- Rationale: Parallel branches are part of same logical operation
- Example: If us-east-1 deploy succeeds but us-west-2 fails:
  1. us-west-2 branch compensation runs (cleanup failed deploy)
  2. us-east-1 branch compensation runs (cleanup successful deploy)
  3. Workflow fails overall
- To allow partial success, use separate workflows instead of parallel branches

**B5: SKIPPED NODE HANDLING**
- Skipped nodes count as "satisfied" for dependency resolution
- Join node doesn't wait for skipped branches
- Example: If conditional branch skips parallel branch A:
  1. Branch A nodes marked as 'skipped'
  2. Join node treats skipped nodes as satisfied dependencies
  3. Join node executes when all non-skipped branches complete
- Prevents deadlock when parallel branches are conditionally skipped

### retry-with-alternative
On failure, compensate and try alternative approach

**Triggered by:**
- CAPACITY_UNAVAILABLE errors (e.g., spot capacity exhausted in region)
- Engine calls determineRetryStrategy() → returns action='retry' + pattern='retry-with-alternative'
- Alternative node config includes different parameters (region, instance type, etc.)

**Execution:**
1. Original node fails with retryable error
2. Engine calls node compensation handler
3. Create new node with alternative config via applyRetryWithAlternative()
4. Alternative node maintains same output contract
5. Downstream nodes continue with alternative's output

**Example Flow:**
```
spawn-instance-us-east-1 (CAPACITY_UNAVAILABLE)
  → compensate (cleanup partial resources)
  → spawn-instance-us-west-2 (alternative region)
  → success, workflow continues
```

**Constraints:**
- Max 3 retry attempts (configurable per error type)
- Alternative must satisfy same output contract
- Each retry increments attempt counter
- After max retries exhausted → fail workflow

## Implementation Notes

- Patterns modify workflow node graph in-place
- Node dependencies updated automatically
- Pattern application logged for debugging
- Invalid pattern application throws ValidationError
