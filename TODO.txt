## 10 Consideration Points for the L2 Transition 

DO NOT START AS-IS: CURATE/REFINE/RESEARCH/VALIDATE/ETC TO PLAN TENTATIVE FIRST STEPS INTO L2 TRANSITION

Moving from L1 (Outlines) to L2 (Detailed Pseudocode) is where you stop describing *what* happens and start defining the **logical edge cases** of *how* it happens. Here are the 10 critical points to steer your L2 drafting:

### 1. Atomic State Transitions (The `WHERE` clause)

Every status update in L2 pseudocode must include its "pre-condition" in the SQL logic.

* **Bad:** `UPDATE allocations SET status = 'CLAIMED'`
* **L2 Standard:** `UPDATE allocations SET status = 'CLAIMED' WHERE id = ? AND status = 'AVAILABLE'`
* This is the only way to prevent two parallel workflow nodes from "double-claiming" the same warm instance.

### 2. The "In-Flight" Buffer for Deduplication

In `storage.ts.l2`, consider the race condition where two users upload the same large file simultaneously. Since you don't have a `UNIQUE` constraint on the checksum (to allow for log appends), L2 needs to define how `checkBlobsExist` handles blobs that are currently being uploaded but aren't "finalized" yet.

### 3. SSE "Byte-by-Byte" Handling

The Python agent's `sse.py` needs a robust generator pattern. In L2, you must explicitly handle the `\n\n` SSE message separator and what happens if a `read(1)` call returns an empty byte (indicating the control plane closed the socket).

### 4. Workflow Timeout vs. Node Timeout

Ensure the `engine.ts` execution loop checks the **Workflow-level** timeout at the start of every iteration, while the individual **Node-level** timeouts are managed by the `Promise.race` or timer within the node's execution block.

### 5. Idempotency Key Cleanup Policy

L2 should define the "Garbage Collection" for the `idempotency_keys` table. If you keep keys forever, the table bloats; if you delete too fast, retries become dangerous. A sliding 24-hour window is usually the L2 standard here.

### 6. The "Partial Artifact" Spot Strategy

When a spot interruption occurs, the agent has a narrow window (e.g., 2 minutes). The L2 logic for `spot.py` should prioritize the **Checkpoint Script** first, and then attempt an **ordered upload** of artifacts (smallest or most recent first) to maximize data recovery before the VM vanishes.

### 7. Explicit "Materialization Depth"

In `materialization.ts`, define a `depth` or `include` parameter.

* Does `materializeInstance()` always fetch all its `Allocations`?
* L2 pseudocode should define "shallow" vs "deep" materialization to prevent accidental "N+1" query patterns where fetching one instance triggers 50 sub-queries.

### 8. Backpressure in Log Streaming

In the WebSocket handlers for logs, consider what happens if the CLI client is on a slow connection. Does the Control Plane buffer infinite logs in memory? L2 should define a **dropping policy** or a **high-water mark** for the in-memory WebSocket buffers.

### 9. Two-Phase Spawn Recovery (A13)

This is your "Ghost Instance" handler. If Phase 1 (DB record) exists but Phase 2 (Cloud ID) is missing, L2 must define how the `reconciler` queries the Cloud Provider by **Tag** (the `spawn_idempotency_key`) to see if the instance was actually created before giving up.

### 10. SSH Config "Host Collisions"

Your CLI generates `~/.repl/ssh_config`. L2 should define how it handles naming if two different instances have the same "Short ID" prefix. (e.g., adding a counter or using more characters of the slug).
