# unit/engine.test.ts - Workflow engine unit tests

## SETUP
- In-memory database with migrations
- Mock executor for node handlers
- Test workflow definitions (simple, complex, branching)
- Helper to advance time for timeout testing
- Clean database between tests

## Submission Tests
- Creates workflow with DRAFT manifest
  - submitWorkflow creates new workflow record
  - Manifest created in DRAFT status
  - Workflow linked to manifest
  - Nodes created from DAG definition
  - Returns workflow_id
- Idempotency key returns existing
  - Submit with same key twice
  - Second call returns same workflow_id
  - No duplicate workflow created
  - Manifest not duplicated
  - Safe for retries
- Different params same key conflicts
  - Same idempotency key, different input
  - Returns 409 conflict error
  - Error includes existing workflow_id
  - Prevents accidental duplicate work

## Execution Tests
- Nodes execute in dependency order
  - Node A depends on B → B runs first
  - Node completion updates status
  - Downstream nodes unblocked correctly
  - Topological sort respected
- Concurrent nodes run in parallel
  - Independent nodes start together
  - Executor receives concurrent calls
  - No artificial sequencing
  - Completion order non-deterministic
- Failed node stops workflow
  - Node fails → workflow status = FAILED
  - Downstream nodes marked SKIPPED
  - Compensation triggered
  - Error details captured
- Completed nodes not re-executed
  - Workflow resumed after restart
  - COMPLETE nodes stay COMPLETE
  - Only PENDING/RUNNING nodes re-evaluated
  - Idempotent handlers tolerate re-execution

## Pattern Tests
- applyPattern modifies DAG
  - Pattern application creates new nodes
  - Existing nodes updated (dependencies)
  - DAG remains valid (no cycles)
  - Changes persisted to database
- Constraints validated
  - Invalid pattern config → error
  - Constraint violations caught early
  - Error messages reference specific constraint
  - Validation happens before DAG modification
- Application persisted
  - Pattern metadata stored in workflow
  - Can reconstruct DAG from DB
  - Pattern ID tracked for debugging
  - Rollback possible if needed

## Compensation Tests
- Only failed node compensated
  - Node fails → its compensation runs
  - Successful nodes not compensated
  - Compensation is single node
  - Multiple failures → multiple compensations
- Transferred resources not compensated
  - Resource ownership transferred → skip compensation
  - Transfer recorded in workflow_state
  - Prevents double-free
  - Transfer check before compensation
- Handler called correctly
  - Compensation handler receives node context
  - Has access to node output (partial)
  - Can mark compensation as failed
  - Timeout applies to compensation too
- Crash recovery tests
  - Running workflows resumed on startup
  - Engine queries for status='RUNNING'
  - Resumes from last completed node
  - State reconstructed from database

## Crash Recovery Tests
- Running workflows resumed
  - On engine start, find RUNNING workflows
  - Resume execution from checkpoint
  - No duplicate work
  - Progress since last persist preserved if possible
- Running nodes reset/failed
  - RUNNING nodes at startup
  - Reset to PENDING if idempotent
  - Mark FAILED if not idempotent
  - Logged for operator review
- Idempotent nodes retried
  - Nodes marked idempotent=true
  - Safe to retry after crash
  - Output same regardless of attempts
  - Non-idempotent nodes require manual intervention
