# state-transitions.ts - State Machine Ownership & Transitions

Defines ownership boundaries for state transitions across allocations, manifests, and workflows.
See L2 for two-phase CAS pattern, TransitionResult type, atomic transition functions, crash recovery.

## TransitionResult Pattern

All transitions return: { success: true; data: T } | { success: false; reason: 'NOT_FOUND' | 'WRONG_STATE' | 'RACE_LOST'; current?: T }
Enables callers to distinguish failure modes

## Allocation State Machine

Status: AVAILABLE | CLAIMED | ACTIVE | COMPLETE | FAILED

Transitions:
- AVAILABLE -> CLAIMED (workflow claim-allocation node)
- CLAIMED -> ACTIVE (control plane on sync_complete log)
- CLAIMED -> FAILED (control plane on sync timeout/error)
- ACTIVE -> COMPLETE (workflow finalize node, warm pool replenishment)
- ACTIVE -> FAILED (control/agent on heartbeat timeout or spot interrupt)

Warm pool replenishment: ACTIVE->COMPLETE creates NEW AVAILABLE allocation on same instance

## ManifestStateMachine

Status: DRAFT | SEALED (EXPIRED is derived, not stored)

Transitions:
- DRAFT -> SEALED (workflow seal-manifest node or completion)

Resource addition rules: DRAFT accepts resources, SEALED rejects (except A13 recovery with allowRecovery=true)

## WorkflowStateMachine

Status: pending | running | completed | failed | cancelled | rolling_back

Transitions:
- pending -> running (engine executeLoop)
- running -> completed (all nodes completed)
- running -> failed (node failure, no retry/fallback)
- running -> cancelled (user request)
- running -> rolling_back (compensation in progress)
- rolling_back -> failed (compensation complete)

Retry: Creates NEW workflow with same inputs, NOT failed->running transition

## Atomic Transitions

All transitions use two-phase CAS: SELECT to verify state, UPDATE with WHERE updated_at = original
Optimistic locking detects concurrent modifications
